# Bid Your Offer - Laravel Real Estate Auction Platform

## Overview
Bid Your Offer is a Laravel-based real estate auction platform designed to streamline property sales through a transparent bidding system. It supports multiple auction types including property, buyer, seller, landlord, and tenant agent auctions. The platform aims to enhance transparency and efficiency in the real estate market by offering functionalities for listing properties, managing bids, and facilitating agent interactions.

## System Architecture
The platform utilizes Laravel 8.x, PHP 8.2.23, and PostgreSQL. For the frontend, it integrates Laravel Mix with TailwindCSS and AlpineJS for a responsive UI, complemented by Livewire for dynamic content. Bootstrap is used for tab navigation. Location autofill is powered by a U.S. Census-based local database.

The core architecture uses a shared `TenantAgentAuction.php` Livewire component to handle form submissions for various agent types, managing submission, draft saving, and metadata persistence. Drafts employ an append-only versioning system. Dynamic model resolution is used to select the correct auction model based on context. Numeric inputs are sanitized, and configuration files centralize display logic. The `ServicesFormatter` groups buyer agent services, and `ListingDisplayHelper` standardizes field formatting. Conditional rendering of fields is managed based on user selections and listing types. PDF listing packets are generated using `barryvdh/laravel-dompdf` with dedicated field map classes and a universal Blade template. Select2 and Livewire are integrated for dynamic and static multi-select fields, utilizing `wire:ignore` for robust initialization and repair. A JSON Bridge Pattern synchronizes data between Select2 and Livewire for certain buyer multi-select fields. Wizard navigation uses a delegation pattern for tab transitions and validation. Address fields dynamically show/hide based on data presence. File uploads (photos/videos) utilize `wire:key` for stabilization and event delegation for validation.

Edit forms implement a comprehensive validation strategy with "Save Draft" and "Save Edit/Submit" actions. `doSaveEditWithSync()` performs full validation across all tabs, while `doSaveDraftWithSync()` allows partial saves by skipping required-field validation. Immutability rules are enforced for critical fields like `Listing Type` and `Current Representation Status with Broker` through disabled inputs with visual lock indicators and server-side re-reading from the database. Server-side PHP validation acts as a safety net for non-draft submissions. Conditional visibility for seller sale terms is handled by `applySellerProvisionVisibility()` and `applySellerFinancingVisibility()` functions, ensuring immediate UI updates and server-side synchronization.

A bidding period timer is implemented for seller, buyer, and landlord listing views, deriving active status from `auction_type` and `auction_time`. The buyer agent search functionality filters records based on `is_approved` and `is_draft` statuses, ensuring only approved, non-draft listings are visible.

The system design emphasizes modularity, separation of concerns, and a database-first approach, optimized for production deployment. The existing database schema for fees is immutable, with display-only formatting updates.

**`TenantBidMatchScoreHelper`** (`app/Helpers/TenantBidMatchScoreHelper.php`) uses a **logical field group** approach for Terms Match scoring. Instead of counting individual UI inputs, it defines 16 logical groups (e.g., "Lease Fee" = type + sub-value as ONE group, "Purchase Fee" = type + amounts as ONE group). Two key rules: (1) **Cascade deactivation** — when a bid's parent field is 'No' (e.g., `interested_purchase_fee_type`), the child groups are entirely excluded from the denominator, so only the parent's Yes→No change is counted; (2) **Composite value** — each group's DB sub-fields are normalized and joined into a single comparable string via `buildCompositeValue()`. The `changed_terms` output includes both the group primary key AND individual sub-field keys for backward-compatible mismatch badge display in the view. The `missing_services` display in the Private Data Modal is visible to all viewers (not just listing owners).

**Match Score Helper maxes (confirmed)**: Tenant=16 groups, Seller=15, Buyer=14, Landlord=17. `additional_details_broker` is intentionally excluded from ALL four helpers. All four helpers return canonical keys `services_matched_count`/`services_baseline_total` (plus legacy aliases); all four views use canonical keys only. **`BuyerBidMatchScoreHelper::getCatalog()`** uses `str_contains()` for property type matching (same as Seller/Tenant/Landlord) — `'Income Property'`, `'Commercial Property'` etc. are all correctly routed.

**Buyer bid view** (`resources/views/buyerAgentAuctionDetail.blade.php`): All action form routes use the `buyer.` prefix (e.g., `buyer.hire.agent.auction.bid.accept`) matching the `Route::prefix('buyer')->name('buyer.')` group in `routes/web.php`. The private modal includes all three bid actions (Accept, Counter Bid, Reject) for the listing owner. Match Score UI is structurally identical to Tenant (same dual/single score layout, same badge styles).

**Seller bid Private Data Modal** section A is labeled "A) Broker Compensation" (not "A) Seller's Broker Compensation").

A **Default Profile system** allows agents to save and reuse bid profile data (bio/About Agent, why_hire_you, what_sets_you_apart, marketing_plan) across all four bid forms (Tenant, Buyer, Landlord, Seller). Profiles are stored in the `agent_default_profiles` table (jsonb `profile_data`, keyed by `user_id + role_type + property_type`). The `AgentDefaultProfile` model provides `findForAgent()` and `upsertForAgent()` helpers. The dashboard is at `/agent/default-profiles` (route `agent.default-profiles.index`). All four bid forms (Tenant, Buyer, Landlord, Seller) are now Livewire components that auto-load the matching default profile on mount and expose `saveAsDefaultProfile()` / `loadDefaultProfile()` methods. Required fields on the Agent Overview tab for all four bid forms: `bio`, `why_hire_you`, `what_sets_you_apart`, `marketing_plan`, `year_licensed`. Tab-0 validation is enforced before advancing to step 2 in all Livewire wizard components via `goToNextStep()`.

The **Seller Agent Bid form** (`app/Http/Livewire/Seller/SellerAgentAuctionBid.php`) is a 6-tab Livewire wizard at route `GET /agent/seller/bid/add/{auctionId}` (name: `add_seller_agent_bid`). Tab partials live in `resources/views/livewire/seller-agent-auction-bid-tabs/commission-based/`. Property type from DB is normalized to short form (Residential/Income/Commercial/Business/Vacant Land) in `normalizePropType()`. After submit, redirects to `seller.agent.auction.detail`. Uses `->extends('layouts.main')->section('content')` for Livewire 2.x layout rendering.

The **Seller Agent bid view page** (`resources/views/hire_seller_agent/view.blade.php`) implements the same bid display system as the live Tenant bid view, including: sortable bid accordion cards (class `bid-accordion-header`, body `id="bidCollapse-{id}"`), Traditional vs Bidding Period visibility rules, `isListingOwner`/`isAgentViewer`/`canSeeBidSummary` variables, a Private Data Modal per bid (with Agent Overview, Services, Agent Presentation, Broker Compensation, Agent Credentials sections), Accept/Reject action buttons for the listing owner, and `AcceptedBidSummary` links. The `SellerAgentAuctionController::viewDetail()` eager-loads `bids.user`, `bids.meta`, `user`, and `meta`, and passes `$auth_id` and `$lowest_bidder`. Routes: `acceptSABid` (POST) and `rejectSABid` (POST).

**Critical data patterns for Seller bid view**: The `accepted` field on `SellerAgentAuctionBid` is stored as boolean `false` (not null/'0'), so state computation must use `(!$rawState || $rawState === '0')` — never `in_array(..., [null, 0, '0'], true)`.

**Critical data patterns for Landlord bid view**: The `accepted` field on `landlord_agent_auction_bids` stores the **string `'no'`** for undecided bids (NOT boolean false, NOT null, NOT '0'). Never use `in_array($rawState, [null, 0, '0'], true)` — it silently fails for 'no', producing `$state = 'no'` which breaks all action-row conditions. Always compute state as: `$_isTerminal = in_array((string)$rawState, ['accepted','rejected'], true); $state = $_isTerminal ? (string)$rawState : '0';`. The accepted controller stores 'accepted'/'rejected' as strings. The `is_sold` field on `SellerAgentAuction` stores string `'false'`, so must check with `!in_array($val, [true,'true',1,'1'], true)`. The `purchase_fee_type` in Seller bids stores `'flat'`/`'percentage'`/`'combo'`/`'other'` (short codes) — NOT human-readable strings like 'Flat Fee'. Combo fee format: `{percentage_combo}% of Total Purchase Price + ${flat_combo}`. Fields `auction_type` and `expiration_date` are accessed via `$auction->get->field` (meta table), not as direct columns.

**AcceptedBidSummary links** are present in all four role bid detail views (Tenant, Seller, Landlord, Buyer). When a bid is accepted, the listing owner and agent both see links to view, e-sign, and download the signed PDF summary via the `accepted-bid-summary.*` routes (`AcceptedBidSummaryController`).

**The Landlord Agent bid system** (`resources/views/hire_landlord_agent/view.blade.php`) now fully mirrors the Tenant bid system including: (1) **`LandlordBidMatchScoreHelper`** (`app/Helpers/LandlordBidMatchScoreHelper.php`) — same 16-logical-group architecture as Tenant, using Landlord-specific fields (`purchase_fee_type`, `broker_fee_timing`, `renewal_fee_type`, `tenant_broker_commission_structure`, `agency_agreement_timeframe`, etc.) and separate Residential/Commercial service catalogs; (2) **`LandlordAcceptedBidSummaryService`** (`app/Services/LandlordAcceptedBidSummaryService.php`) — generates Landlord-branded HTML summary using `AcceptedBidSummary` table (stores landlord `user_id` in `tenant_user_id` column, Landlord-specific HTML template in Sections 1 and 8); (3) **`LandlordAgentHiredNotification`** (`app/Notifications/LandlordAgentHiredNotification.php`) — broadcasts on `user.{auction.user_id}` when a bid is accepted; (4) `LandlordAgentAuctionBidController::accept_bid()` and `accept_counter_bid()` — wrapped in DB transaction, generate summary post-commit, notify both landlord and agent, redirect to `accepted-bid-summary.view`; (5) Match score badge (colored %) in bid card headers, Services/Terms breakdown in expanded card bodies, rich "Offered Services: X/Y matched" display, and Landlord E-Sign link alongside Agent E-Sign link in the Private Data Modal; (6) **Broker Compensation section in Private Data Modal** restructured from flat `div/div` layout to Tenant-style **A–G grouped sections** with `<ul><li>` lists and per-field mismatch highlighting — A) Landlord's Broker Lease Fee, B) Tenant's Broker Compensation, C) Lease-Option Details, D) Purchase Fee Details, E) Legal Terms (protection period, early termination, agency timeframe, property management), F) Brokerage Relationship, G) Additional Terms / Additional Details. Each section renders conditionally; the outer "2. Broker Compensation" wrapper only renders if at least one field has data. Duplicate "3. Additional Terms & Details" block removed; additional terms are now exclusively in section G. (7) **Services Offered section** in both the Private Data Modal and the inline allMeta section replaced with Tenant-style categorized `ul/li` bullet-point format — color-coded green (matched), blue (extra/not-in-baseline), red (missing); legend row; category headings with emojis (📢 Rental Marketing, 📋 Listing Presentation, 📸 Photography, 🏡/🏢 Showings, 📝 Tenant Application, 📃 Lease Preparation, 🚚 Move-In, 📑 Property Management, 💡 Leasing Strategy); ✍️ Additional Services for `other_services`; red "Missing Services (Not Offered)" box at bottom. A `$normForCat` helper normalizes both actual Unicode curly quotes and literal `\u2019` escape text for robust category-membership matching.

## F. Architectural Foundation (Version 2.4 — Source-Neutral Core)

> **Status of this section.** This foundation was added in Version 2.4 to establish the platform's **source-neutral core and cross-cutting intelligence architecture** (F1–F9) — the load-bearing layer beneath the catalog. It sits *beneath* the existing roadmap: §0–§10 are unchanged in scope and substance and are re-based to operate on the **Canonical Data Model** defined here rather than on any one source's native fields. Nothing in the intelligence catalog (§1–§10) is redesigned — it is bound one layer higher. F1–F3 establish the data, persistence, and compliance core; F4–F8 establish the trust, matching, marketplace, and learning layers that every consumer-facing surface runs on; F9 establishes **Location Preference DNA** — user-defined geometries as first-class canonical objects on the shared Location Intelligence taxonomy, generated by the same engine as property-side Location DNA and matched against it.

### F1. Canonical Data Model Foundation

**Principle.** Bid Your Offer is a **source-neutral intelligence platform**, not a database of one MLS. Every listing, party, and search criterion — regardless of origin — is normalized into a single **Canonical Data Model (CDM)** before any DNA, matching, Ask AI, analytics, recommendation, marketing-intelligence, explainability, or learning-loop logic touches it. **No downstream system may read a source-specific field directly.**

**RESO alignment.** The CDM adopts the **RESO Data Dictionary** (Property, Member, Office, Media, and related resources) as its canonical vocabulary, extended with BYO-specific canonical fields for concepts RESO does not cover (bid terms, demand-side criteria, DNA scores/vectors). RESO alignment is what lets Stellar MLS and future RESO-compliant MLSs map in with minimal per-source work, and keeps BYO portable as providers change.

**Supported sources — all map INTO the CDM (none is privileged in logic):**

| Source | Type | Adapter notes |
|---|---|---|
| Bid Your Offer (native) | First-party | BYO's own columns are one adapter among many, not the platform's internal schema |
| Stellar MLS | RESO MLS | Direct RESO Data Dictionary → CDM |
| Future MLS providers | RESO MLS | Reuse the RESO adapter pattern |
| RentCast | API | Rental comps / AVM adapter |
| ATTOM | API | Assessment / valuation / ownership adapter |
| Public records | Bulk / API | County assessor, deed, permit, FEMA flood adapters |
| CSV imports | File | Mapped via a declared, versioned column-mapping profile |
| Future APIs | API | New adapter only — zero change to downstream logic |

**Canonical field convention (normative).** Throughout §0–§10, **every field, input, tag, and score name denotes a *canonical* field.** Where a familiar BYO-native column name appears (e.g. in an §8 "Inputs:" list such as `water_access`, `additional_rooms`, `association_fee_includes`, `flood_zone_code`), it is shown as an *illustrative source mapping* — shorthand for its canonical equivalent resolved by the source-adapter layer — **not** a direct dependency. No score, tag, DNA vector, or match reads a source column; it reads the **canonical field** the adapter populated. This convention re-points the entire catalog to the CDM without altering any catalog substance.

**Source adapters (contract).** Each source has an adapter that:
1. **Maps** source fields → canonical fields via a declarative, versioned mapping profile.
2. **Stamps provenance** on every canonical value: `source`, `source_field`, `ingested_at`.
3. **Assigns confidence** (0–100) from source reliability × field completeness.
4. **Records freshness**: `source_as_of`, `last_refreshed`.
5. **Never mutates the intelligence layer** — it writes canonical values + metadata only.

**Per-field metadata (carried on every canonical value):** `value`, `source`, `source_field`, `confidence` (0–100), `provenance` (chain when derived from multiple feeds), `ingested_at`, `source_as_of`, `last_refreshed`.

**Source-priority / conflict resolution (deterministic, configurable).** When two sources populate the same canonical field:
1. **Field-level source priority** — an ordered per-canonical-field priority list (e.g. for `flood_zone_code`: FEMA public record > ATTOM > MLS remark).
2. **Freshness tiebreak** — within equal priority, most recent `source_as_of` wins.
3. **Confidence tiebreak** — within equal freshness, higher confidence wins.
4. **First-party human override** — a human-confirmed value (`human_confirmed: true`) is pinned and never silently overwritten by a feed; conflicts are **surfaced, not clobbered**.
Losing candidates are retained for audit and explainability.

**What must operate ONLY on the CDM:** all DNA — the **core DNA systems** (Property, Buyer, Tenant, Location DNA) and the **extended DNA vectors defined in §9** (Lifestyle, Financial, Intent, Audience, Investment, Personality/Vibe, Community, Commute/Mobility, Timeline, Risk, Sustainability/Green, Accessibility, Deal, Relationship/Agent-Fit, Pet, Behavioral); the Universal Matching Engine (§9 unified score / §F6 Universal Relevance Score); Universal Ask AI; analytics; the Recommendation Engine; Marketing Intelligence; Explainability; and the Platform Learning Loop (§F8 / Wave 8). None may branch on source identity except inside an adapter.

**Role & domain lenses (composed, not independent engines).** *Seller DNA*, *Landlord DNA*, *Luxury DNA*, and *Commercial DNA* are **not** separate DNA engines to be built — they are **lenses composed from the core and §9 DNA systems** above. *Seller/Landlord DNA* are the supply-side role views (property scores §8 + Deal DNA + Audience DNA); *Luxury DNA* is composed from the Luxury Score (§8) + Personality/Vibe DNA + luxury lifestyle tags (§1); *Commercial DNA* is composed from the §6 Use-Fit engine + Investment DNA + the Commercial-Fit-Out Score (§8). Implementers build the underlying systems once; the lenses are read-time compositions over them.

### F2. DNA Persistence & Versioning

**Storage model.** The DNA layer is stored **separately from source ingestion and separately from the role/EAV listing tables**, keyed off the **canonical listing/party id** — never off a role-specific native table. This resolves the platform's native-column (Seller/Buyer) vs. EAV-meta (Landlord/Tenant) asymmetry: because DNA attaches to the canonical id, it is **role-symmetric and property-type-symmetric by construction**, independent of how any one role persists its raw fields.

- **Scalar scores & tags** (§8 0–100 scores; §10 tags) → dedicated canonical DNA tables (one row per listing/party per score/tag), not the four role auction tables.
- **DNA vectors** (§9 Lifestyle / Financial / Intent / Investment / Community / etc.) → versioned vectors with per-dimension confidence.
- **Vibe / semantic embeddings** (§5 personality embedding; "find me a home that *feels* like…") → a vector index alongside the canonical id.

**Symmetry guarantee.** Every DNA element persists in two parallel forms on the same 0–100 axis (§8): a **property/listing-side** value and a **demand-side preference weight** — both against the canonical id — so the matcher compares apples-to-apples across all four flows and all six property types.

**Versioning.** Every score / tag / vector carries a **version tag** = `{model_version, feature_set_version, canonical_schema_version}`, making intelligence reproducible, enabling bulk re-scoring when a model improves, and letting analytics compare cohorts scored under different versions.

**Regeneration triggers (automatic, idempotent, logged).** DNA regenerates when:
- a canonical field it depends on changes (new price, new commute anchor, fresh flood determination);
- new photo/media arrives (vision-derived §5 tags);
- a Location DNA enrichment refreshes (POI, FEMA, school, crime, traffic);
- a source adapter re-ingests or a higher-priority source supersedes a value;
- a model or feature-set version is published (bulk regeneration).
Each regeneration records what triggered it.

**Auditability.** Each DNA value stores its `explanation` string, `confidence`, `version tag`, an `inputs snapshot` (which canonical fields + values fed it, with provenance), and `generated_at`. Any question — "why is this a 94% match?", "why is this labeled luxury?" — is answerable from stored data, and any historical score is reconstructable.

### F3. Fair Housing & Compliance Guardrails

**Scope.** Platform-level controls binding on **inputs, matching/ranking, target audiences, marketing recommendations, matching reasons, and Ask AI outputs** — not optional per-field notes. They apply to every consumer-facing surface and are enforced at the canonical/intelligence boundary (F1), so a single control set governs every source.

- **C1 — Objective, explainable inputs only.** DNA, matching, and recommendations may consume only **objective property/location/transaction attributes and self-declared preferences.** Prohibited as inputs to any score or ranking: race, color, national origin, religion, sex, familial status, disability — and any **proxy** for them (e.g. inferring ethnicity from names/photos, religion from surname, familial status from vision analysis of occupants). Every input must trace to a canonical field with objective provenance.
- **C2 — No protected-class inference.** The platform must not infer, store, or act on any consumer's protected-class membership. Area demographic data (§4 Age Mix, Diversity Index) is **descriptive locale context only**: it may **never** feed consumer-facing match ranking or ad targeting, and is shown only as neutral, factual context the user explicitly requests.
- **C3 — Anti-steering.** The Recommendation Engine and Universal Matching Engine must not steer users toward or away from areas on any protected basis. Ranking may use user-declared lifestyle / commute / budget / school-*quality* signals; it may **not** use neighborhood demographic composition. School signals use objective ratings/proximity, never demographic makeup.
- **C4 — Compliance-safe target audiences (§2, §10.5).** Audience tags describe **property suitability and self-declared identity** and drive **marketing-copy tone and opt-in outreach only** — never protected-class ad delivery. Segments touching protected classes (familial status, national origin, religion, disability, age) may inform *content* ("home-office ready") but may **not** be used as ad-delivery or exclusion targeting parameters — consistent with HUD advertising rules and the HUD/Meta consent-decree standard.
- **C5 — Compliance-safe matching reasons & Ask AI.** Every match reason and Ask AI answer is generated **only** from the objective explanation strings (F2). The system must refuse steering-style or protected-class framing (e.g. "a great area for families like yours," "a community of people like you"). Safety, school, and demographic answers return **objective, sourced, confidence-tagged** data with neutral framing — never a protected-class recommendation.
- **C6 — Enforcement point.** Because all logic runs on the CDM, a single compliance filter over canonical inputs and generated outputs governs every source and surface: blocked inputs are excluded from feature sets by policy, and all generated copy/answers pass a compliance check before display.

### F4. Confidence & Completeness Scoring

The platform never presents intelligence as fact without qualifying how sure it is. Three distinct, composable confidence signals travel with every record, DNA element, and match:

- **Data Completeness Score (0–100) — per listing/party.** What fraction of the canonical fields relevant to this property/role/type are populated (importance-weighted), and how fresh/authoritative their sources are (from the §F1 per-field confidence + freshness). Answers *"how much do we actually know about this record?"* Drives completeness prompts ("add photos to unlock 6 more scores"), fair ranking (incomplete records are neither unduly penalized nor flattered), and completeness-gated score suppression.
- **DNA Confidence (0–100) — per score/tag/vector.** Defined in §8/§F2; formalized here as reflecting (a) completeness of the specific canonical inputs the score depends on, (b) source reliability of those inputs, and (c) model/feature-set maturity (version tag). A Boating Score from full water fields + geocode + photos is high-confidence; one inferred from sparse inputs is flagged low and de-weighted downstream.
- **Match Confidence (0–100) — per candidate pairing.** How trustworthy a given relevance/match score is, propagated from the DNA Confidence of every component that fed it plus the Data Completeness of both sides. A 92% relevance built on weak inputs is reported as "92%, low confidence," and the engine may prefer a slightly lower relevance with high confidence.

**Propagation rule (never inflates):** Data Completeness → DNA Confidence → Match Confidence. No downstream number may claim higher confidence than the weakest input it rests on. Every surface (search, Ask AI, Marketplace Intelligence, recommendations) shows the relevant confidence band; low-confidence intelligence is visibly labeled, never silently presented as certain.

### F5. Explainable AI (Explainability Guarantee)

**Guarantee.** Every score, tag, recommendation, and match is **explainable by construction** — no black-box output reaches a user. This is a hard architectural requirement, not a feature.

- **Every score/tag** carries the `explanation` string + `inputs snapshot` + `confidence` + `version tag` from §F2 (e.g. "Boating 92: saltwater canal, 60 ft frontage, private dock with lift, 1.2 mi to nearest inlet").
- **Every recommendation** ("deals for you," "homes you may like") carries a per-item reason traceable to the canonical inputs and DNA scores that produced it.
- **Every match** exposes a **per-category relevance breakdown** (§F6), so "why is this a 94% match?" returns a structured, sourced answer — which category contributed what, at what confidence.
- **Explanations are the only substrate for Ask AI and marketing copy** (per §F3/C5): generated language may only restate stored objective explanations — never invent rationale or introduce protected-class framing.
- **Reproducibility:** because explanations store their inputs snapshot + version tag, any historical explanation is reconstructable and auditable for compliance and dispute resolution.

### F6. Universal Relevance Scoring

**Purpose.** One engine ranks every pairing the platform cares about — buyer↔listing, tenant↔listing, listing↔demand (reverse), and client↔agent — on a single 0–100 **Relevance Score**, decomposed into transparent **category match scores**. This formalizes §9's "How the DNA systems compose" into the platform's canonical matching contract.

**Category match scores (each 0–100, each with its own confidence)** — the overall relevance is a weighted blend of independent category scores, so a user (and Ask AI) always sees *where* fit is strong or weak:

- **Property fit** — physical attributes vs. hard/soft criteria
- **Location fit** — Location DNA vs. Location Preference DNA (commute, schools, POIs) on the shared canonical taxonomy
- **Lifestyle fit** — Lifestyle DNA symmetry (§1/§8)
- **Financial fit** — affordability, financing compatibility, deal feasibility (Financial DNA)
- **Investment fit** — yield / risk / strategy alignment (Investment DNA)
- **Personality / Vibe fit** — aesthetic/character embedding similarity (§5)
- **Intent fit** — motivation alignment and intent-driven re-weighting (§3)
- **Risk fit** — exposure tolerance vs. property risk profile (Risk DNA)
- **Market Timing fit** — Timeline DNA + demand/absorption + days-on-market/urgency signals
- *(extensible: Pet, Accessibility, Sustainability, Deal, Community, Commute/Mobility, Relationship/Agent-Fit)*

**Composition rules (inherited from §9, formalized):**
- **Intent-driven weighting** — the searcher's Intent DNA and stated priorities dynamically re-weight the category mix (investor → Financial/Investment/Deal lead; relocating family → Location/Lifestyle/School/Commute lead).
- **Feasibility gates** — Financial and Market Timing fit can cap an otherwise-high relevance when a deal cannot close or timing conflicts.
- **Match Confidence** (§F4) accompanies the overall relevance and every category score.
- **Explainability** (§F5) — the category breakdown *is* the explanation of the match.
- **Symmetry** — the same engine, run in reverse, scores demand against a listing (feeds §F7).
- **Compliance** — category inputs obey §F3; no protected-class category or proxy may enter the blend.

**Cross-source, source-neutral matching (§F1).** Every Buyer and Tenant profile automatically matches against **all** supported property sources through the Canonical Data Model — **Bid Your Offer, Stellar MLS, future MLS providers, RentCast, ATTOM, Public Records, CSV imports, and future APIs** — because the engine reads only canonical fields. In one pass it evaluates the full canonical DNA stack — **Property, Buyer, Tenant, Location, Location Preference, Lifestyle, Investment, and Community DNA** plus the composed **role/domain lenses** (*Seller, Landlord, Luxury, Commercial* — per §F1, read-time lenses over the core systems, not separate engines) — and every future canonical intelligence layer. Matching stays completely source-neutral: **no source-specific field may exist outside the source-adapter layer.**

**Match tiers — never "no results" (Similar-Match intelligence).** The engine **never returns only exact matches.** When no exact match exists it automatically returns the closest-fitting properties, banded into transparent tiers — **Exact Match · Strong Match · Similar Match · Opportunity Match** — by overall Relevance and which categories cleared. Every surfaced result carries its per-category explanation (§F5) of **why** it was included and where it falls short, so "similar" and "opportunity" results are self-justifying rather than mysterious.

**Two-way matching (both directions).** The engine always runs both ways on the same canonical scores: **Buyer/Tenant criteria → properties** (across every supported source) **and properties → Buyer/Tenant criteria.** So every listing can identify the Buyer and Tenant demand profiles that match it, and every Buyer/Tenant profile can identify matching properties from every supported listing source. The reverse direction feeds the Seller/Landlord-facing **Marketplace Intelligence** view (§F7). **Continuous synchronization (triggers).** The Universal Matching Engine shall evaluate every active Buyer and Tenant profile whenever a new listing is created, imported, updated, or materially changed, and shall evaluate every listing whenever a Buyer or Tenant profile is created or updated, ensuring both supply and demand remain continuously synchronized across all supported data sources.

### F7. Marketplace Intelligence

**Purpose.** A demand-side read surface that turns the Universal Relevance engine (§F6, run in reverse) into actionable market context *for a listing* — answering the seller/landlord/agent question *"who wants this, and how does it sit in the market?"* This operationalizes the requirement that **listings identify matching buyer/tenant criteria**, not just the reverse.

**What it surfaces (per listing — aggregate and privacy-safe):**
- **Active demand** — count and strength of active buyer/tenant criteria whose reverse-relevance to this listing clears a threshold ("34 active buyers match ≥ 80%"). Formalizes Top-100 field #98 (Comparable-Buyer Demand Heat).
- **Tiered demand (privacy-safe).** The reverse match surfaces demand as aggregate **tiered counts** — *Exact / Strong / Similar Buyer Matches* and *Exact / Strong / Similar Tenant Matches* (the §F6 match tiers, in reverse) — so a seller/landlord/agent sees the depth and quality of demand for the listing without ever exposing an individual consumer's identity, contact, or protected attributes.
- **Audience segments** — which Target-Audience / Audience-DNA segments (§2/§9) the matching demand falls into, so marketing can be aimed precisely.
- **Market insights** — submarket context from Location DNA + analytics: comparable inventory, absorption / days-on-market, price positioning vs. comps, demand trajectory, seasonality.

**Rules.** Demand is shown **only in aggregate** (never exposing an individual consumer's identity or protected attributes); segment/insight generation obeys §F3 (no protected-class targeting); every figure carries its Match Confidence (§F4) and an explanation (§F5). Marketplace Intelligence is a *view* over canonical data + the matching engine — it introduces no new asked fields.

### F8. Continuous Learning, Feedback Loops & Behavioral DNA

**Purpose.** The platform improves from real outcomes rather than remaining static at launch weights. This formalizes Wave 8 into the definitive learning architecture and introduces **Behavioral DNA** as its consumer-side substrate.

**Lifecycle feedback events (captured, canonical, timestamped):** saved/hidden listing, viewed listing + dwell, search + refinement, Ask AI question, **showing** attended, **offer/bid** made or received, **lease** signed, **purchase/closing** completed, and post-transaction feedback. Each is an objective interaction signal — never a protected-class inference.

**Learning loops (each closes on a lifecycle stage):**
- **Engagement loop** — saves/views/searches refine a consumer's preference weights and Behavioral DNA in near-real-time.
- **Transaction loop** — showings → offers → leases → purchases → closings validate or correct the match model: which predicted-high matches actually converted, re-weighting F6 category weights and F4 calibration.
- **Outcome loop** — closed comparables feed analytics (pricing, absorption, audience accuracy) and Marketplace Intelligence (§F7).

**Governance.** Re-weighting operates on the **canonical, objective** feature set only (§F1); learned weights are versioned (§F2) so changes are auditable and reversible; the loop may **never** learn or encode a protected-class proxy (§F3) — a fairness monitor audits learned weights for disparate impact, and global updates roll out under a version tag with rollback.

**Behavioral DNA (future learning layer).** A consumer-side vector derived purely from *interaction* signals — saved/viewed/hidden listings, searches and refinements, offers, Ask AI questions, explicit feedback — capturing **revealed preference** (what a user *does*, which often differs from what they *say*). It **supplements, never overrides** self-declared criteria and Intent DNA, and is bound by the same explainability (§F5) and Fair Housing (§F3) controls. Catalogued as a DNA type in §9; its learning mechanism lives here in F8. (See also §10 field #50, Buyer Decision-Style Profile.)

### F9. Location Preference DNA (User-Defined Geometries & Constraint-Based Preferences as First-Class Canonical Objects)

**Principle — a searcher's *where* is DNA, not a filter.** Every user-defined geographic preference — a drawn **polygon**, a **circle**, an address-anchored **radius**, and any future supported geometry — is a **first-class Canonical Location Preference DNA object**, not merely a search filter that narrows a result set. Each geometry is a durable, enrichable preference object that the intelligence layer reasons over exactly as it reasons over a listing's location, generated on the **same canonical Location Intelligence taxonomy** the platform uses for property-side **Location DNA**. (Typed entries — preferred cities, ZIP codes, counties, state — and important-place anchors are the same kind of object, resolved to their canonical geographic entities.)

**One Location Intelligence Engine; two sides of one taxonomy.** Property **Location DNA** (what a listing's location *is* — POIs, flood zone, school districts / attendance, commute anchors, walk/transit context) and Buyer/Tenant **Location Preference DNA** (what a searcher's chosen area *offers and requires*) are generated from the **same Canonical Data Model (§F1)** by the **same shared Location Intelligence Engine**. A preferred polygon is enriched with the identical taxonomy of objective location facts as a property point, so the two are **directly comparable** rather than heuristically bridged.

**Per-geometry DNA generation — cached · versioned · explainable · auditable.** For each user geometry the engine computes a Location Preference DNA record on the shared taxonomy — the area's POI profile, transit / commute reachability, school context, environmental / flood context, and the objective location facts the user's anchors imply. Each record:
- is **cached** against the canonical criteria / party id (§F2) so it is computed once and reused, not recomputed on every search;
- is **versioned** with the `{model_version, feature_set_version, canonical_schema_version}` tag (§F2) — reproducible and bulk-regenerable when a model improves;
- carries its **explanation string, inputs snapshot, confidence, and `generated_at`** (§F2 / §F4 / §F5), so every preference-side fact is **auditable**;
- is **regenerated only when** the geometry changes, the underlying Location Intelligence changes (a POI / FEMA / school / transit refresh touching that area), or regeneration is **explicitly requested** — never recomputed needlessly.

**Matching — the Universal Matching Engine, on the shared taxonomy (§F6 Location fit).** The Universal Matching Engine compares a property's **Location DNA** against a buyer/tenant's **Location Preference DNA** on the **shared canonical taxonomy** — is the listing inside or near the preferred geometry, and how well do the area's canonical location facts satisfy the searcher's declared priorities — contributing that as the **Location fit** category score (§F6) with its Match Confidence (§F4). Because both sides speak one taxonomy, this is a direct DNA-to-DNA relevance computation, not a geometric containment test bolted onto scoring. The flexible-location flag and radius / important-place anchors are the levers §F6's **intent-driven weighting** uses to decide when geography **gates** an otherwise-strong match versus **softly re-weights** it.

**Two ways to express *where* — one canonical target.** Location Preference DNA supports both ways a searcher describes location, and both compile to the **same** structured canonical Location Preference DNA the Universal Matching Engine consumes:

- **A. User-defined geometries.** Explicit areas the user draws or names — **polygons, circles, radii, cities, ZIP codes, counties, states,** and future supported geometry types. As above, each is a **persistent Canonical Location Preference DNA object** on the same Location Intelligence taxonomy as Property Location DNA — **not merely a search filter.**
- **B. Constraint-based Location Preference DNA** (natural-language → canonical requirements) — described next.

**Constraint-based Location Preference DNA.** A user may describe their ideal location naturally — e.g. *"within 10 minutes of a beach, not in a flood zone, close to Madeira Beach Fundamental, within 5 minutes of a Publix, and within 5 minutes of a bus stop."* The platform translates that description into **structured canonical Location Preference DNA requirements** on the same taxonomy, so the matcher scores it identically to a drawn geometry. Each parsed requirement is canonical along four axes:

- **Target — a named place *or* a place category.** Either a specific named place (*Publix*, *Madeira Beach Fundamental*, a work address, a relative's home) **or** a category (*grocery store*, *A-rated elementary school*, *beach*, *bus stop*). Supported targets span, e.g.: beaches, waterfront, flood zones, schools, grocery stores (incl. named, e.g. Publix), hospitals, marinas, parks, airports, restaurants, entertainment, shopping, work locations, daycare, place of worship, gyms, family, friends, transit, bus stops, train stations, bike paths, boat ramps, commercial districts, and investment areas.
- **Proximity — travel time *or* distance.** Either a travel-time budget (*within 5 minutes*) **or** a straight distance (*within 1 mile*).
- **Travel mode.** driving, walking, biking, or transit — so "5-minute **walk** to a bus stop" and "5-minute **drive** to Publix" are distinct canonical constraints.
- **Preference strength.** **Must Have** (hard gate), **Strong Preference** (heavily weighted), or **Nice To Have** (lightly weighted) — e.g. *Must Have: outside flood zone*; *Strong Preference: within 5 min of Publix*; *Nice To Have: near restaurants*. Strength maps directly onto §F6's gate-vs-re-weight behavior.

**Structured UI and the natural-language layer coexist.** The existing structured Important Places / Radius / Commute UI remains fully supported and authoritative. A future natural-language layer simply lets a user *describe* their preferred location in plain words; the AI translates that description into the **same** structured canonical Location Preference DNA the Universal Matching Engine already consumes. The NL layer **complements** the structured UI — it does not replace it, and both produce identical canonical objects.

**Explainable location matching (per-constraint verdicts).** Because every geometry and constraint is canonical and independently scored, each property recommendation explains its location fit constraint-by-constraint — which **passed**, which **partially matched**, which **failed**, and which are **uncertain** because the underlying data is unavailable or low-confidence (§F4). For example:

- ✓ Beach within 8 minutes
- ✓ Publix within 4 minutes
- ✓ Bus stop within a 3-minute walk
- ✗ In Flood Zone X
- ⚠ School boundary unavailable (low confidence)

Verdicts use only objective, sourced, confidence-tagged Location Intelligence — never protected-class framing (§F3 / §F5).

**Explainability & Fair Housing (§F5 / §F3).** Explainable AI states **why** a property matches a user's preferred area in **objective, Fair Housing–compliant Location Intelligence** terms — "inside your drawn area; 8 min to your named commute anchor; within your preferred school attendance zone; low flood exposure" — each reason sourced, confidence-tagged, and reconstructable from the stored inputs snapshot. Preferred-area fit is **never** framed in neighborhood-demographic or protected-class terms; it uses objective geographic / commute / school-*quality* / environmental facts only (§F3 · C1 / C2 / C3 / C5).

## 0. Purpose & Framing

This is **not** an MLS audit and it is **not** a field-parity exercise. The Bid Your Offer ↔ Stellar MLS Gap Analysis (`docs/bid-your-offer-stellar-gap-analysis.md`) already established that BYO is at or ahead of MLS parity on physical property attributes. This document starts from that conclusion and asks a different question:

> **If we were building the most intelligent AI-powered real estate platform possible, what would we capture that neither Bid Your Offer nor the MLS captures today — and that would create a durable competitive advantage rather than just more data?**

The MLS describes *what a property is*. This roadmap describes *who a property is for, why someone wants it, what living there feels like, and what a machine can infer about all of the above*. That is the layer that powers matching, Ask AI, lifestyle search, target-audience marketing, and recommendations — and it is the layer no MLS collects.

### Design principles

1. **Don't collect data — create intelligence.** Every field must feed matching, Ask AI, search, marketing, or recommendations. If it only "documents," it belongs in the MLS, not here.
2. **Prefer AI-derived over human-entered.** Anything a model can infer from **canonical fields we already hold** (mapped from BYO and every connected source via the §F1 adapters; see §8) should be *computed*, not asked. We reserve human input for genuinely subjective, first-party knowledge (the seller's lived experience, the buyer's true intent).
3. **Symmetry across the four flows.** Property-side intelligence (Seller/Landlord) and demand-side intelligence (Buyer/Tenant) must be describable in the *same vocabulary* so the matching engine can compare them directly. A "boating lifestyle" property should match a "boating lifestyle" buyer.
4. **Intent and audience are first-class.** *Why* someone wants a property and *who* it is for are the two highest-leverage signals we are not capturing. They reshape ranking, marketing copy, and recommendation logic.
5. **Every subjective claim is auditable.** Human-entered lifestyle/personality tags become *claims*; AI-derived scores become *evidence*. Where they disagree, that disagreement is itself a signal (and a marketing-integrity guardrail).

## 3. Buyer / Tenant Intent

Yes — BYO should capture *why* someone wants a property, because intent (motivation) is the single most predictive ranking signal in real estate and it is entirely absent from the MLS, which only ever describes supply. Two buyers with identical budgets and criteria behave completely differently if one is a deadline-driven 1031 investor and the other a school-district-driven family, and intent tells the engine which listings to surface first, how urgently, and in what tone. This is fundamentally a **demand-side, self-declared** layer (Buyer-entered / Tenant-entered), though it is also strongly inferable from a consumer's declared audience, budget, financing, timing, and search behavior — so BYO should ask for it directly and back-fill/confirm it with AI. BYO already captures a coarse purpose (e.g. `current_status`, primary-vs-investment), so these should be modeled as a richer, structured, multi-select intent taxonomy that supersedes the coarse field rather than duplicating it.

### How intent should modulate match ranking and Ask AI tone

Intent is best modeled as a **ranking modifier and dimension re-weighter**, not just another filter. Declared intent should dynamically shift the weight the engine places on other signals: a `1031 Exchange` or `Relocation for Job` intent raises the weight of timing/availability and turnkey condition and injects urgency into notifications; an `Investment` / `Cash-Flow-Now` intent elevates yield/cap-rate/separately-metered signals while suppressing lifestyle tags; a `School-District-Driven` or `Medical-Proximity` intent promotes the corresponding Location DNA proximity to a near-dominant factor; an `Airbnb / STR` intent makes STR-permissive zoning/HOA a hard gate rather than a soft preference. Where a consumer declares multiple intents, they should be weighted (primary vs secondary) so the engine can blend them. Ask AI should read the same intent to set **tone and content**: numbers-and-returns framing for investor intents, empathetic and process-forward guidance for `First Home` / `Divorce / Life-Change`, lifestyle-and-community framing for `Relocation-for-Lifestyle` / `Retirement`, and diligence-warning framing (zoning, HOA, deadlines) for `STR` and `1031` intents. Because intent is also inferable, the platform should surface an AI-suggested intent for confirmation, improving capture rates without adding form burden.

Neighborhood *feel* is consistently the number-one thing buyers and renters actually research and the number-one thing the MLS never records — the MLS describes the four walls, not the block. BYO already geocodes every listing and runs Location DNA (commute times, POI / Important Places search areas, flood, schools), so this section deliberately goes **beyond raw geocoding and POI counts** into the **perceptual and qualitative** layer: personality, atmosphere, perception, and trajectory. The overwhelming majority of these fields should be **AI-derived** — a new Location DNA perception layer synthesized from POI density and category mix, review-sentiment mining (Google/Yelp), crime and demographic feeds, transit/traffic APIs, climate/insurance data, and satellite/street-view signals — with **optional human corroboration** (local resident, agent, seller/landlord) only where lived experience beats the model. Each item below is tagged pure-AI vs. human-corroborated.

Every field carries a machine `confidence` score, a `provenance` (which data sources fed it), and a `last_refreshed` timestamp so the platform can decay stale perceptions and re-run enrichment. All are keyed off the geocode, so they compute once per micro-neighborhood and attach to every listing inside it.

## 5. Property Personality

Property "personality" is AI-generated metadata describing what a home *feels* like — the qualitative character the MLS never records — and it is what powers natural-language search ("show me cozy historic homes with tons of light"), auto-generated marketing copy, and vibe-based buyer/seller matching. These descriptors should be produced primarily by a **vision model reading the listing photos** combined with structured fields (year built, style, sqft, features, lot), each emitted as a tag with a **confidence score (0-1)** and **evidence** (which photos/fields triggered it), then made **human-confirmable** by the seller or agent (confirmation raises confidence and locks the tag). Every descriptor maps to two consumer surfaces: **natural-language / semantic search** and **marketing tone**, and feeds vibe-matching in the Recommendation Engine.

Descriptors are non-exclusive (a home can be `coastal` + `light-filled` + `entertainers` + `turnkey`), stored as a scored tag array on Property DNA, with polarity noted where two tags conflict (e.g. `minimalist` vs `character-rich`) so search can rank rather than hard-filter.

## 6. Commercial Intelligence

For commercial, business, and land assets, the highest-value metadata is not another square-footage field — it is **use-fit intelligence**: an explicit, machine-readable answer to "what business could actually thrive in this space?" BYO already captures the physical envelope (zoning, ceiling height, building features, electrical service, space class) and the structured economics roadmap (rate unit, CAM, TI, escalations, dock config) is covered elsewhere; this section is deliberately orthogonal to both. Almost every item below is **AI-derived** from the intersection of zoning + existing build-out + location context + traffic/demographics, emitted with a confidence score and surfaced to the listing agent for one-tap confirmation or override. The result is a tag layer that powers cross-use search ("show me anything a dentist could open"), Ask AI ("could a brewery work here?"), buyer-side recommendation ("you asked for restaurant space — here are 3 second-gen spaces"), and marketing copy that speaks to the actual end user rather than the current owner.

### 6.1 Use-Fit Engine (meta / scoring fields)

These are the machinery that makes every use-fit tag below trustworthy and rankable.

#### Best-Use Ranking
- **What:** Ordered list (top 3–5) of the highest-probability viable uses for the space, each with a probability weight (e.g. `["quick-service restaurant": 0.82, "salon/spa": 0.74, "professional office": 0.61]`).
- **Why:** Lets search and recommendations lead with the strongest end-user narrative instead of forcing buyers to infer fit; drives "best use" marketing headlines and Ask AI's "what would work here?" answers.
- **Value source:** AI-Generated, Derived, Agent-entered (confirm/override)
- **Property types:** Commercial Sale, Commercial Lease, Business Opportunity, Vacant Land
- **Flows:** All
- **Becomes:** Property DNA, Target Audience Intelligence, Marketing Intelligence, Ask AI, Search Filter, Recommendation Engine
- **Priority:** Critical

#### Use-Fit Confidence Score
- **What:** Per-tag 0–100 confidence value plus a provenance flag (`ai_only` / `agent_confirmed` / `agent_overridden`) attached to every use-fit tag in this section.
- **Why:** Prevents AI-derived tags from being presented as fact; lets search rank agent-confirmed fits above speculative ones and lets Ask AI hedge appropriately ("likely, pending grease-trap verification").
- **Value source:** AI-Generated, Derived, Agent-entered
- **Property types:** Commercial Sale, Commercial Lease, Business Opportunity, Vacant Land
- **Flows:** All
- **Becomes:** Property DNA, Ask AI, Search Filter, Recommendation Engine
- **Priority:** Critical

#### Conversion Effort / Readiness Grade
- **What:** For each viable use, a grade of `turnkey` / `light-fit` / `moderate-buildout` / `heavy-conversion` estimating the capex distance between current condition and that use.
- **Why:** A "restaurant-ready" tag means nothing without knowing if it's a second-gen turnkey or a raw shell; drives capex-aware matching and Ask AI cost questions.
- **Value source:** AI-Generated, Derived, Agent-entered
- **Property types:** Commercial Sale, Commercial Lease, Business Opportunity
- **Flows:** All
- **Becomes:** Property DNA, Target Audience Intelligence, Ask AI, Recommendation Engine
- **Priority:** High

#### Prohibited / Ineligible Uses
- **What:** Explicit negative tags for uses blocked by zoning, deed restriction, covenant, or association rule (e.g. `no_drive_thru`, `no_food_service`, `no_cannabis`, `no_automotive`).
- **Why:** Prevents false-positive matches and lets Ask AI answer "no, cannabis is not permitted here" authoritatively; negative signal is as valuable as positive for filtering.
- **Value source:** Objective, AI-Derived, Agent-entered
- **Property types:** Commercial Sale, Commercial Lease, Business Opportunity, Vacant Land
- **Flows:** All
- **Becomes:** Property DNA, Search Filter, Ask AI, Recommendation Engine
- **Priority:** High

## 7. Investment Intelligence

Investors do not buy bedrooms — they buy **cash flow, risk profile, and upside**. This section captures the decision variables an acquisition analyst actually runs, so that BYO can match investor-buyers to deals, answer Ask AI ROI questions ("what's the cap rate at asking?", "is there loss-to-lease?"), and drive investor-facing recommendations. The defining principle: **most of these are DERIVED or AI-modeled** — computed from price, income, comps, location, and financing assumptions BYO already holds — not new questions posed to the seller. Items are explicitly marked DERIVED, AI-modeled, or entered so the platform knows which to compute silently and which to collect.

### 7.1 Composite investor signal

#### Investor-Fit Score
- **What:** Composite 0–100 score blending cap rate, cash flow stability, upside potential, and risk flags into a single investor-suitability signal, with a sub-breakdown.
- **Why:** Lets investor-buyers rank inventory instantly and lets the recommendation engine surface "deals for you"; the headline number for investor DNA.
- **Value source:** Derived, AI-Generated
- **Property types:** Income, Commercial Sale, Business Opportunity, Residential (investment)
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Target Audience Intelligence, Ask AI, Search Filter, Recommendation Engine
- **Priority:** High

### 7.2 Return metrics (DERIVED / computed)

#### Cap Rate
- **What:** Derived net operating income ÷ price, computed at asking and (where entered) at a bid price.
- **Why:** The universal commercial/income yardstick; enables cap-rate search bands and Ask AI "what's the cap rate?"
- **Value source:** Derived
- **Property types:** Income, Commercial Sale, Business Opportunity
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Ask AI, Search Filter, Recommendation Engine
- **Priority:** Critical

#### Cash-on-Cash Return
- **What:** Derived annual pre-tax cash flow ÷ cash invested, using configurable financing assumptions (down payment, rate, term).
- **Why:** The metric leveraged investors actually optimize; powers scenario-based Ask AI and investor matching.
- **Value source:** Derived
- **Property types:** Income, Commercial Sale, Residential (investment)
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Ask AI, Search Filter, Recommendation Engine
- **Priority:** High

#### Gross Rent Multiplier (GRM)
- **What:** Derived price ÷ gross scheduled income.
- **Why:** Fast comparative screen for income property; cheap to compute, useful in search and Ask AI.
- **Value source:** Derived
- **Property types:** Income, Commercial Sale
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Ask AI, Search Filter, Recommendation Engine
- **Priority:** Medium

#### Debt-Service Coverage Ratio (DSCR)
- **What:** Derived NOI ÷ annual debt service under standard financing assumptions.
- **Why:** The lender's gating ratio and a core financeability signal; investors screen on it heavily.
- **Value source:** Derived
- **Property types:** Income, Commercial Sale, Business Opportunity
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Ask AI, Search Filter, Recommendation Engine
- **Priority:** Medium

#### Break-Even Occupancy
- **What:** Derived occupancy percentage at which the asset covers operating expenses plus debt service.
- **Why:** A crisp risk indicator (how much vacancy can it absorb); strong Ask AI answer for cautious buyers.
- **Value source:** Derived
- **Property types:** Income, Commercial Sale
- **Flows:** Seller, Buyer
- **Becomes:** Property DNA, Ask AI, Recommendation Engine
- **Priority:** Medium

#### Land-to-Improvement Ratio
- **What:** Derived proportion of value attributable to land vs improvements (from assessment/comp data).
- **Why:** High land ratio flags redevelopment/teardown upside and depreciation considerations; investor and developer signal.
- **Value source:** Derived
- **Property types:** Commercial Sale, Income, Residential (investment), Vacant Land

## 8. AI-Derived Metadata (Compute, Don't Ask)

> **Canonical inputs (per §F1).** Every `Inputs:` list below names **canonical fields**. The BYO-native column names shown (e.g. `water_access`, `additional_rooms`, `association_fee_includes`) are illustrative source mappings resolved by the source-adapter layer — a score reads the canonical field, never a source column — so the same score computes identically for a BYO, Stellar, RentCast, ATTOM, public-record, or CSV-sourced listing.

BYO's competitive edge is that the intelligence below is **auto-calculated, never asked**. Every score is computed from **canonical fields** the platform already holds (mapped from every connected source per §F1) across the four flows, the six property types, and the existing DNA systems (Property DNA, Buyer DNA, Tenant DNA, Location DNA) — so we reduce form fatigue while producing richer, more consistent intelligence than any single MLS, which relies on agents typing free-text remarks. Each score is a **0–100 field** carrying a plain-language explanation string and a confidence value, and it **refreshes automatically whenever any input changes** (a new photo, an updated commute anchor, a fresh Location DNA flood result). Critically, every score is **symmetric**: it is computed both as a *property score* (how well this listing embodies the trait) and as a *buyer/tenant-preference weight* (how much this trait matters to this searcher), so the matcher can rank listings against people on the same 0–100 axis. Human-entered lifestyle tags from other sections are used as **training/corroboration signals, never as overrides** — if a seller tags "cozy" but the fireplace, small-lot, and interior-features inputs disagree, the model reconciles rather than blindly trusts.

- **Why:** Matches tenants/buyers to move-in-ready vs shell commercial space.
- **Value source:** Derived. Inputs: space classification (A–D), space type (vanilla/gray shell, re-let), restrooms/offices/conference-room counts, ceiling height, electrical service (3-phase), HVAC, loading/dock config, clear span/high bays, TI allowance, `condition_prop`.

- **Flows:** Seller, Buyer, Landlord, Tenant.
- **Becomes:** Property DNA, Buyer DNA, Tenant DNA, Search Filter.
- **Priority:** Medium

### Scoring architecture

Every score above is a **0–100 numeric field** stored alongside three companions: an **explanation string** (human-readable rationale surfaced verbatim by Ask AI — e.g. "Boating 92: saltwater canal, 60 ft frontage, private dock with lift, 1.2 mi to the nearest inlet"), a **confidence value** (0–100, reflecting input completeness and source reliability — a score computed from photos + geocode + full field set is high-confidence; one inferred from sparse inputs is flagged low), and a **version tag** (model + feature-set version, so scores are reproducible and re-computable when the model improves). Scores are **symmetric**: for every property-side score there is a parallel **buyer/tenant-preference weight vector** on the same 0–100 axis, derived from the searcher's criteria, commute prefs, Important Places, and stated intent — the matcher computes fit as a weighted distance/dot-product between the property score vector and the preference weight vector, so ranking is apples-to-apples. Scores **refresh automatically** whenever any input changes (new photo, updated commute anchor, fresh flood determination, price change), and the version tag lets us re-score historical inventory in bulk. Finally, **human-entered lifestyle tags from other sections are training/corroboration signals, not overrides**: they label examples for model calibration and can raise confidence when they agree with computed inputs, but a computed score is never silently replaced by a self-reported tag — disagreements are reconciled and, where material, surfaced for review.

### How the DNA systems compose

The DNA systems are not siloed profiles — they are weighted vectors that feed a single **unified match score**. For any candidate pairing (buyer↔listing, tenant↔listing, or client↔agent), the engine computes per-DNA similarity (Lifestyle, Financial, Intent, Community, Commute/Mobility, Timeline, Risk, Deal, Pet, Accessibility, Sustainability, Personality/Vibe, Investment, plus Relationship/Agent-Fit for agent matching) and blends them into one 0–100 result. The blend weights are **intent-driven**: Intent DNA and the searcher's stated priorities dynamically re-weight the mix (an investor pairing leans on Investment + Financial + Deal DNA; a relocating family leans on Lifestyle + Community + School-Fit + Commute/Mobility), while Financial and Timeline DNA act as **feasibility gates** that can cap an otherwise-high match when a deal simply cannot close or the timing conflicts. Each contributing DNA carries its own confidence, so the unified score reports both a rank and a confidence band, and every component exposes its explanation string to Ask AI — meaning a user can always ask "why is this a 94% match?" and get a per-DNA breakdown. Because all vectors share the 0–100 symmetric convention from §8, the same math ranks properties for a person, people for a property (Audience DNA), and agents for a client — one composable intelligence layer across the entire platform. This composition is formalized as the platform's **Universal Relevance Scoring** contract in §F6 (category-level match scores + Match Confidence + explainable breakdown), fed in reverse into Marketplace Intelligence (§F7) and continuously recalibrated by the learning loops (§F8).

### 10.11 Recommended Implementation Order

Sequenced so each wave unlocks the next: derive from what exists before asking users for more, and build the unified score last once every tag layer is populated.

1. **Wave 1 — Derived DNA & scores from existing structured fields (zero new user input).** Compute Property/Investment tags purely from **canonical fields already populated** (cap rate, NOI, pool, waterfront, financing — mapped from BYO and any connected source per §F1), because it delivers matching value immediately with no form changes or migrations. (Standing up the §F1 canonical model + BYO adapter is the prerequisite for this wave.)
2. **Wave 2 — Demand-side intent capture (Buyer/Tenant motivations).** Add lightweight motivation/purpose prompts to the criteria flows, since the ranking-effect modulation in 10.6 needs an explicit "why" that only the buyer/tenant can supply.
3. **Wave 3 — Lifestyle & target-audience tagging.** Derive the 10.4/10.5 tags from the launch-blocker and Wave-1 fields (schools, community features, ADU, accessibility), turning structured data into searchable lifestyle facets.
4. **Wave 4 — Location DNA neighborhood tags.** Run the geospatial pipeline (POIs, FEMA, schools, commute, crime/traffic) to emit the 10.9 tags, layering place intelligence on top of property intelligence.
5. **Wave 5 — Investment & commercial tag layer.** Finalize the 10.7/10.8 tags once commercial economics fields (rate unit, CAM, dock config, tenancy) land, so investor and use-fit matching becomes precise.
6. **Wave 6 — Marketing-tag generation + channel routing.** Auto-generate the 10.10 phrases from the now-rich tag set and route each to headline/social/email/Ask AI, closing the loop from data to demand generation.
7. **Wave 7 — Symmetric match-score integration.** Fold every tag layer into a single weighted property-side↔demand-side score (respecting `config/match_scoring.php` summing to 100), so all the intelligence converges into one ranked result.
8. **Wave 8 — New DNA systems + feedback loop.** Stand up the net-new AI DNA generators and a behavioral feedback loop (clicks, saves, accepted bids) that re-weights tags and scores over time, making the platform self-improving.

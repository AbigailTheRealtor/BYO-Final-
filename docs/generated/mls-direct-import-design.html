Headline

The building blocks already exist: a live Bridge/Stellar OData integration, a local bridge_properties cache holding full raw_json, a URL/text listing-import pre-fill pipeline, and a complete property-vs-criteria match engine. The two new features are mostly wiring existing parts together behind a normalized data seam, not building new engines.

  • Use Case A Seller/Landlord listing prefill — populates the create/edit form (facts only).
  • Use Case B Buyer/Tenant match check — analyzes a property against saved criteria and shows a match report; never populates a form.

They share only a lookup/cache/normalization core shared. Their user-facing routes and workflows are fully separate.

1 Phase 3 = facts only Seller/Landlord prefill imports objective fields only — address, price/rent, beds, baths, sqft, lot size, year built, property type, status, HOA, taxes, pool/garage/waterfront flags, coordinates, factual attributes. Users add photos/description manually. Fuller media only after written Stellar MLS confirmation.

Sources: Bridge/Stellar (preferred), existing URL/text parser, manual entry (FSBO/off-market), future RESO/Bridge MLSs. Neither the match engine nor the prefill service knows the origin. Scalable as sources are added.
3 B rollout gate New dedicated feature flag, default OFF. Controls rollout independently.
4 B criteria selection Auto-select the preferred saved profile via CriteriaListingResolver, with a dropdown to switch profiles.
5 Lookup UX MLS # primary (resolved to ListingKey internally). Address search is the fallback and shows a chooser when multiple units/properties match.

Compliance boundary (A vs B)

  • The facts-only rule constrains what A writes into a publishable form — a single allow-list to audit, aligned with the safe/restricted classification our AuditBridgeFields code already encodes.
  • B reads restricted fields (PublicRemarks, CDD status, etc.) internally to compute a score and never republishes them — a different, lower-risk act. B keeps full internal field access.

Phased plan
Abstraction designed-in now, implemented incrementally.
Phase 1designDONE

Investigation & design

This document. Confirmed Bridge lookup capability, local cache, field support, the match engine, criteria loaders, and Location DNA. Locked decisions above. No code.

Phase 2shared · backend only

Shared MLS lookup core + normalization seam

  • NEW BridgeListingLookupService: findByMlsNumber(), findByListingKey(), searchByAddress(). Local bridge_properties first → Bridge API on miss → upsert (also dispatches ComputeLocationDna).
  • NEW PropertyCandidate DTO + BridgePropertyCandidateAdapter (first adapter).
  • MIGRATION index listing_id and address/city (only listing_key is indexed today — else MLS#/address lookups table-scan).
  • REUSE BridgeApiService, BridgePropertyNormalizer, BridgeProperty. Unit-tested on fixtures. No UI.
Phase 3Use Case ASeller / Landlord only

Seller/Landlord listing prefill — facts only

  • NEW MlsListingPrefillService: PropertyCandidate → facts-only canonical-key array, same ['success','data','error'] shape as MlsListingImportService::import().
  • EDIT HasMlsImport — add importListingFromMls() / importListingFromAddress() beside the URL entry.
  • EDIT shared MLS import modal — add "By MLS #" / "By Address" tabs; keep URL/text parser as fallback tab (unchanged) and as a second candidate source.
  • REUSE MlsFieldMap + applyImportedFields() preview/apply. Scoped to Seller + Landlord create/edit only. initializeLimitedService() untouched (frozen).
Phase 4Use Case Bnew route · flag OFF

Buyer/Tenant match check

  • NEW standalone /match-check route + controller + Livewire + view. Behind the new feature flag (default OFF). Never writes a form.
  • NEW MlsPropertyMatchAnalysisService orchestrating the pipeline. Auto-select preferred criteria via CriteriaListingResolver + switch dropdown.
  • EDIT wire the scorer input to PropertyCandidate (see refactor note in risks).
  • REUSE BuyerMatchScorer, BuyerMatchResultBuilder, BuyerResultViewMapper, all 4 criteria loaders, BuyerCriteriaPayload. MLS#-primary lookup; address chooser for multiple units.
Phase 5Use Case Breport enrichment

Location DNA + commute enrichment

  • Load PropertyLocationDna + property_location_pois keyed ('bridge', bridge_property.id) → nearest grocery/beach/park/restaurant/school (miles) + five lifestyle scores.
  • Bridge rows get DNA from the existing dispatch; non-Bridge candidates geocode-then-compute.
  • Commute is a stub today; flood/schools are live bbox lookups. Surface all as "if available"; first view of a fresh import may show "enriching…".
  • Additive to B's report only — Use Case A untouched.
Phase 6Use Case Bnear-zero engine work

Better matches

When the entered property scores low, call the existing BuyerMatchService::match($payload, $cap, $role) and show entries scoring higher than the entered property. Mostly a UI section.

Laterpost-launch

Additional sources + fuller media

Manual-entry and other-MLS adapters against the same PropertyCandidate contract (zero change to A/B). Preview-only or fuller media/remarks for Use Case A pending written Stellar MLS confirmation.

Known dependencies & risks
  • Scorer↔BridgeProperty coupling — the one real refactor. Recommended: refactor the scorer's input to PropertyCandidate (Bridge path gets a BridgeProperty → PropertyCandidate mapper). Shortcut alternative: hydrate a transient non-persisted BridgeProperty — faster but leaks the Bridge shape; treat as debt.
  • Commute provider is a stub (travel_time_minutes always null) — real commute needs a live provider (separate decision).
  • Flood zone & school district are live bbox lookups, not stored per property — call on demand from the property's geocoded point.
  • Async Location DNA — a freshly-imported MLS row's DNA computes on a queue; design the report to show "enriching…" gracefully.
  • Indexes — MLS#/address lookups need the Phase 2 indexes to avoid table scans.

**Phase 2 · Batch 1b · Spatial Intelligence Platform**
Branch: `phase-2-batch-1b-postgis-schema-migrations`
Status: **cluster-independent portion authored; live EXPLAIN acceptance gate blocked on the verification cluster.**

This batch stands up the spatial core (SSOT §7.1–7.2) as **guarded, additive, consumer-free** migrations that run **only** against a dedicated managed-PostGIS connection. Nothing in the application reads these tables yet.

---

## How to run the migrations

The spatial migrations live in `database/migrations/spatial/` and are **not** part of the default `php artisan migrate`. They run only via an explicit path + connection:

```bash
php artisan migrate --path=database/migrations/spatial --database=pgsql_spatial
```

Status / dry-run / rollback (all path- and connection-scoped):

```bash
php artisan migrate:status   --path=database/migrations/spatial --database=pgsql_spatial
php artisan migrate --pretend --path=database/migrations/spatial --database=pgsql_spatial
php artisan migrate:rollback  --path=database/migrations/spatial --database=pgsql_spatial
```

### Why the separate path is safe by construction

Laravel's migrator globs `$path.'/*_*.php'` **non-recursively** (`Migrator::getMigrationFiles`). `database/migrations/*_*.php` cannot match `database/migrations/spatial/*.php`, so the default `migrate` and the SQLite test suite **never load these files**. Pinned by `tests/Feature/Spatial/SpatialMigrationIsolationTest`.

## The `pgsql_spatial` connection

Added in `config/database.php`. It reads **dedicated `SPATIAL_*` env vars only** — no fallback to `DB_HOST` / `DB_DATABASE` / `DATABASE_URL` — so it cannot accidentally resolve to the application database, and its absence is detectable.

| Env var | Purpose |
|---|---|
| `SPATIAL_DATABASE_URL` | Full DSN (Crunchy Bridge). Preferred. |
| `SPATIAL_PGHOST` / `SPATIAL_PGPORT` | Host / port (alternative to URL). |
| `SPATIAL_PGDATABASE` / `SPATIAL_PGUSER` / `SPATIAL_PGPASSWORD` | Database / credentials. |
| `SPATIAL_PGSSLMODE` | Defaults to `require` (Crunchy Bridge mandates TLS). |

With none set, the connection is inert and the migrations **fail closed** (see below). **No production cluster is provisioned yet** (SIA-D38 deferral).

## Fail-closed guard

Every migration inlines an identical guard (self-contained, matching this repo's migration convention) that runs before any DDL and refuses to proceed unless the resolved connection is a **configured PostgreSQL** target:

- connection **absent** (no url and no host) → `RuntimeException("… not configured …")`
- connection **present but not `pgsql`** → `RuntimeException("… not 'pgsql' …")`

Pinned for all 11 migrations, `up()` and `down()`, by `tests/Feature/Spatial/SpatialMigrationGuardTest`.

## Pinned extension versions (SIA-D39 / E-49)

Migration 01 asserts the installed versions **exactly** and throws on any mismatch:

| Extension | Pin |
|---|---|
| postgis | **3.6.3** |
| btree_gist | **1.7** |
| pg_trgm | **1.6** |

These are the measured Stage 0b baseline on Crunchy Bridge (PostgreSQL 16.14). `btree_gist` is required before migration 04 can build `gist(category_key, geom)` (SSOT §7.3).

## Rollback

- Each `down()` drops only what its `up()` created, with `DROP TABLE IF EXISTS … CASCADE`, in reverse FK order.
- **Extensions are never dropped on rollback** — migration 01's `down()` is a deliberate no-op (dropping `postgis` cascades to every geography column).

## Acceptance gate — blocked on the verification cluster

The **mixed-geometry KNN EXPLAIN** check (E-50 caveat closure) is the one item that requires the live cluster. Production `places.geom` is `geography(Geometry,4326)` (mixed); the Stage 0b spike proved the exact-set/index-only plan on `geography(Point)` only. Procedure:

1. Apply the 11 migrations on `pgsql_spatial`.
2. Load the Tier-2 mixed-geometry fixture (`spikes/phase-2-batch-1b-postgis-schema/fixtures/generate_tier2_fixture.sql`) → 5,000,200 rows, Point + Polygon + LineString, `ANALYZE`.
3. Run `spikes/phase-2-batch-1b-postgis-schema/validate/run_validation.php` (or `knn_explain.sql`) and confirm, per sparse category + park:
   - `Index Scan using places_cat_geom` · `Index Cond: category_key = …` · in-scan `<->` `Order By` · **0** rows removed · **0** Sort · **0** Seq Scan.
   - over-fetch(20) + spheroidal re-rank returns the exact nearest **set**, order-corrected.
4. **Fallback if the mixed type degrades the plan** (SSOT-sanctioned): run KNN against `centroid` (`geography(Point)`, already spike-validated).
5. Tear down the fixture partition; capture a `RESULT.md` in the Stage 0b format.

The plan-shape logic (`Tests\Support\Spatial\ExplainPlanShape`) and the fixture distribution (`Tests\Support\Spatial\FixtureCorpusPlan`) are already unit-tested — only the live `EXPLAIN` awaits the cluster.

# Ask AI — Suggested Question (Chip) vs. Typed Input Parity Audit
**Date:** June 2026

---

## Overview

The Ask AI feature exposes questions to users via two UI surfaces:

1. **Suggestion chips** — pre-generated question buttons rendered by the listing view Blade
   template. Each chip fires a `POST /api/ask-ai` request via JavaScript with the chip text
   as the `question` parameter.

2. **Typed input** — a free-text form also on the listing view page. It submits to the same
   `POST /api/ask-ai` route (or the equivalent Blade `listing/{type}/{id}/ask` route).

---

## Controller Parity

| Surface | Controller | Route | Runner call |
|---|---|---|---|
| Chips | `AskAiApiController::ask()` | `POST /api/ask-ai` | `AskAiRunnerV2Service::run()` |
| Typed input | `AskAiListingQuestionController::ask()` | `POST /listing/{type}/{id}/ask` | `AskAiRunnerV2Service::run()` |

Both controllers call `AskAiRunnerV2Service::run($role, $listingId, $question)` with the
same interface. The pipeline (classifier → context builder → contract → prompt → adapter →
final builder) is identical for both. **No divergence exists at the controller level.**

---

## Chip Question Coverage

Suggested chips are generated by the Blade template based on the listing type. The chip text
maps to phrase patterns registered in:

- `AskAiRunnerV2Service::LISTING_KEY_KEYWORD_MAP` — for `listing.*` field questions
- `AskAiRunnerV2Service::FAQ_KEY_KEYWORD_MAP` — for `faq_answers.*` pre-answered questions
- `AskAiQuestionClassifierService::KEYWORD_RULES` — for general question classification

## Typed Input — NLP Routing Correctness

For typed free-text questions the classifier uses `KEYWORD_RULES` for coarse routing and
`LISTING_KEY_KEYWORD_MAP` / `FAQ_KEY_KEYWORD_MAP` for field-level routing. The classifier
is order-dependent: `listing_facts` rules are evaluated before `property_standout` and
other types so that listing-data questions always win.

Classifier regression coverage for known phrases is provided by:
- `AskAiCoverageHarnessTest` (structural + phrase routing tests)
- `AskAiTaxRoofBedroomsNlpTest` (targeted phrase regression)
- `AskAiListingFieldPipelineE2ETest` (full pipeline per field)
- `AskAiLiveUiRegressionTest` (live-UI mismatch regression)

## Recommendations

1. **Keep chip text anchored to LISTING_KEY_KEYWORD_MAP keywords.** Any chip whose text
   does not contain a phrase from the map will fall through to generic classification and
   may not resolve a specific `listing.*` field.

2. **Run `AskAiCoverageHarnessTest` after any chip text change** to catch regressions in
   the keyword-to-field resolution path before they reach the live UI.

3. **Add a chip-text integration test** when new chips are introduced: assert that the
   chip text classifies to `listing_facts` and resolves the expected `listing.*` key.

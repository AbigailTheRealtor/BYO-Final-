# FIX 1 — Residential Edit: Personal Photo does NOT auto-populate

When opening an existing Residential Landlord listing for edit (or reopening a draft):

* the previously uploaded Personal Photo must **display as a preview**
* user must be able to **remove** or **replace** it
* saving edits must preserve it unless removed/replaced

## Implementation requirements

### A) Persist the stored photo path in DB (already likely exists)

* Ensure the record stores a durable path like `personal_photo_path` (or your existing column/key).

### B) Hydrate the photo path into Livewire state on Edit mount/loadDraft

In the Landlord **Edit** component `mount()` / `loadDraft()`:

* set Livewire property to the saved path (string), not boolean false, not null if present

Example expectation:

* `$this->personal_photo_path = $listing->personal_photo_path ?? null;`

✅ **Safeguard:** do NOT default to `false` (prevents “false” from rendering issues you already saw elsewhere).

### C) Show preview in Edit blade if a stored path exists

* If `personal_photo_path` exists, render an `<img>` preview (same styling as Create if Create shows preview).
* Provide “Remove Photo” action:

  * sets path to null in state
  * removes file (or flags for cleanup)
  * persists on save

### D) Persist replacement correctly

If user uploads a new photo on edit:

* overwrite the stored path
* (optional) delete old file to prevent orphan storage

### Acceptance tests (must pass)

1. Upload photo on Residential create → Save Draft → reopen draft → photo preview shows
2. Save final listing → Edit → photo preview shows
3. Replace photo → Save → refresh → new photo remains

3. `digital_photo_enhancement_other_text` (string, only if Other selected)

✅ **Safeguard:** enhancement types must be stored as an array/JSON, not a single string that loses multiple selections.

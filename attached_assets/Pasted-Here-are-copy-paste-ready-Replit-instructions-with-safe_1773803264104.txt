# REPLIT INSTRUCTIONS WITH SAFEGUARDS — FIX EDIT REQUIRED-FIELD VALIDATION FLICKER / SCREEN JUMPING FOR SELLER / BUYER / LANDLORD / TENANT

## ISSUE

After the recent fix for required fields on **Edit Listing**, the screen now **flickers / jumps up and down** when it tries to take the user to a required field.

That makes it hard to type into the field.

This needs to be fixed for:

* Seller
* Buyer
* Landlord
* Tenant

and for these **Edit Listing** actions:

* **Save Draft**
* **Submit / Save Edit**

This is **not** for the separate new-listing draft button.

---

# GOAL

Keep the required-field enforcement on **Edit Listing**, but stop the screen from:

* jumping up and down repeatedly
* re-scrolling over and over
* re-focusing the field continuously
* making it hard to type into the required field

The behavior should still:

* block save when required fields are missing
* identify the missing field(s)
* take the user to the missing field/tab like create does

But it should do that **once in a stable way**, not repeatedly.

---

# REQUIRED RESULT

On **Edit Listing** for Seller / Buyer / Landlord / Tenant:

* required fields should still be enforced on **Save Draft** and **Submit / Save Edit**
* user should still be taken to the missing required field/tab
* but the screen should **not flicker or bounce repeatedly**
* once the user is taken to the required field, the page should stay stable so they can type

# MOST LIKELY ROOT CAUSE TO CHECK

The new edit validation likely caused one of these problems:

## A) Scroll/focus logic is firing repeatedly

Example:

* validation finds missing field
* scrolls to field
* rerender happens
* scroll runs again
* focus runs again
* page keeps bouncing

## B) Livewire rerender is retriggering the required-field navigation loop

Example:

* after every state update, the “go to required field” logic runs again

## C) Event listeners were attached multiple times

Example:

* same submit or validation listener is being bound again after each render
* one click triggers several scroll/focus actions

## D) Auto-focus and auto-scroll are fighting each other

Example:

* one script scrolls to field
* another tries to focus another element
* then the page jumps back again

## E) Validation loop keeps trying to move to the “next” missing field before user can type

Example:

* instead of stopping on first missing field and waiting,
* it keeps cycling through missing fields continuously

# REQUIRED IMPLEMENTATION RULE

The required-field guidance should happen **once per user action**, not repeatedly.

## Correct behavior

When user clicks **Save Draft** or **Submit / Save Edit** on edit:

1. validate fields
2. identify first missing required field
3. open correct tab/section if needed
4. scroll to that field once
5. focus it once if appropriate
6. stop and let the user type

## Incorrect behavior

* repeated scrolling
* repeated focusing
* repeated jumping between missing fields
* refiring after every keystroke or rerender

---

# PREFERRED FIX APPROACH

Use the existing create-style validation UX, but make the edit path stable.

Possible safe approaches if applicable:

* add a guard so required-field scroll/focus runs only **once per submit attempt**
* clear/reset that guard only when the user submits again
* prevent duplicate event binding after rerender
* debounce or single-fire the scroll/focus routine
* stop the loop after the **first missing required field** instead of continuously cycling

Do **not** remove the tab-jump / required-field behavior entirely unless absolutely necessary. Just stop the repeated triggering.

---

# IMPORTANT UX REQUIREMENT

Once the user is taken to the missing field, they should be able to type without the screen jumping away from them.

So if create-style UX currently keeps trying to move to each missing field automatically in edit mode, adjust it so:

* it takes user to the first missing field
* waits for user interaction
* does not keep auto-jumping while they are typing

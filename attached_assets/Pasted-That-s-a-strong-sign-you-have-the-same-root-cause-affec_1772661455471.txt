# Step 1 — Ensure both selects are wrapped with wire:ignore (prevents Livewire DOM diff)

For BOTH fields, wrap the `<select>` in a parent element with `wire:ignore`:

* `#number_of_unit_type`
* `#condition_prop_buyer`

**Reason:** Livewire re-renders the select markup, which causes Select2 to detach/reset → flash.

# Step 2 — Guard Select2 init so it never re-initializes repeatedly

Create a small init function for each field, and **return early** if already initialized:

### Required guard:

If the select already has class `select2-hidden-accessible`, do nothing.

Apply to both fields.

**Reason:** repeated init/destroy causes the “disappear/reappear” behavior.

---

# Step 3 — Namespace + de-duplicate change handlers

Before binding change handlers, always:

* `.off('change.<namespace>')`
* then `.on('change.<namespace>', ...)`

Use distinct namespaces:

* `change.unitType`
* `change.propCondition`

**Reason:** When init runs more than once, you stack handlers, and the field gets set/cleared/set quickly → flash.

---

# Step 4 — STOP triggering Select2 change events inside message.processed

Search in these blades for any of the following **inside** `Livewire.hook('message.processed')`:

* `syncSelect2MultiSelects()`
* `.trigger('change')`
* `.trigger('change.select2')`
* `.val(x).trigger(...)`

✅ Remove those from `message.processed`

Allowed in `message.processed`:

* calling a guarded init function (Step 2)

Not allowed:

* triggering change events or syncing values that fires change handlers

**Reason:** This is the classic infinite loop chain:
`message.processed` → trigger change → Livewire update → `message.processed` → repeat
Even if it’s not infinite, it causes repeated resets/flash.

**Reason:** hidden init causes Select2 to render incorrectly, then reflow later → “flash”.

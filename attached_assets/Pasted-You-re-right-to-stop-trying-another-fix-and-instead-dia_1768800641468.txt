# ✅ Phase 2 — Apply the correct fix based on what Phase 1 shows

## Fix Path A: Click handler never fires

* Ensure `[data-wizard-next]` is on the actual clickable element (the `<button>`), not a wrapper that disappears.
* Confirm the button is not being replaced with another element without the data attribute after render.
* Add the data attribute in the shared partial that renders the button so it persists across rerenders.

## Fix Path B: Click fires but function is undefined

* The script defining `window.buyerEditWizardNextStep` is not executing.
* Move the function definition into a script that is guaranteed to load (same file, below the DOM, not inside a conditional).
* Ensure it is truly `window.buyerEditWizardNextStep = function(){...}` in global scope.

## Fix Path C: Validation blocks “Next” incorrectly

* Make `validateServicesTabGlobal()` only validate fields that are visible/selected.
* If “service type” isn’t set on Edit, set a default from saved meta OR default to full service.
* Log exactly which missing field triggers the block and adjust only that rule.

## Fix Path D (Most likely): JS tries to switch tabs but Livewire controls the active tab

This is the *cleanest permanent fix*:

### ✅ Replace JS navigation with Livewire navigation

1. Add Livewire methods in the Edit component:

* `nextTab()`
* `prevTab()`

2. In those methods, update the Livewire step/tab property (whatever Create uses) and do NOT rely on DOM class toggles.

3. Update the Edit buttons:

```html
<button type="button" wire:click="prevTab">Back</button>
<button type="button" wire:click="nextTab">Next</button>
```

4. Keep JS validation, but call it from Livewire by emitting a browser event OR by preventing NextTab if JS says invalid:

* simplest: before calling `nextTab`, run JS validation and only then `@this.call('nextTab')`.

This makes it immune to DOM cloning and rerenders.

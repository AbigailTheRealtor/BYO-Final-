Fix Direct Hire acknowledge submit redirect + prevent listing creation

Issue:
When a user accepts “Confirm Your Hire Request” and submits their info on the acknowledge page, example:
/hire/agent/direct/142/buyer/residential/acknowledge

The system currently redirects to a listing page and/or creates a listing. This is wrong.

Required behavior:
After Submit Hire Request, the user should stay within the Direct Hire flow and see a submitted confirmation/summary page showing:
- request submitted successfully
- selected agent / preset
- role: Seller, Buyer, Landlord, or Tenant
- property type
- submitted client details
- accepted hire terms / services / compensation terms

No listing should be created.

Applies to all Direct Hire acknowledge flows:
- Seller
- Buyer
- Landlord
- Tenant

Do NOT change:
- marketplace Hire Agent listing creation flows
- Offer Listing flows
- Offer Listing Preset counter terms
- bid/counter logic
- Match Score logic
- database schema unless audit proves unavoidable
- Limited Service code
- routes unrelated to Direct Hire acknowledge submit

Step 1 — Read-only audit first

Audit the full Direct Hire acknowledge flow:
- route for /hire/agent/direct/{agent}/{role}/{propertyType}/acknowledge
- controller method that displays acknowledge page
- controller method that handles Submit Hire Request
- blade used for acknowledge page
- any service/helper that creates listings after acknowledge submit
- redirect destination after successful submit

Before editing, report:
1. exact route names/URLs involved
2. exact controller method handling the submit
3. whether the current code creates a Seller/Buyer/Landlord/Tenant listing record
4. exact file list that will be edited
5. whether those files are shared with marketplace Hire Agent flows

Step 2 — Fix submit behavior

In the Direct Hire acknowledge submit handler only:

- validate the required client details as currently intended
- save the accepted hire request / accepted terms record if such a model already exists
- do not create a new public listing
- do not redirect to any listing view route
- redirect to a Direct Hire submitted summary/confirmation page

If no dedicated summary route/view exists, add one surgically for Direct Hire only.

Suggested route behavior:
GET /hire/agent/direct/{agent}/{role}/{propertyType}/submitted

This page should render the submitted/accepted terms from the Direct Hire request, not from a public listing.

Step 3 — Confirmation/summary page requirements

The submitted confirmation page should show:

Header:
“Hire Request Submitted”

Message:
“Your hire request has been submitted successfully.”

Summary details:
- Agent name
- Role: Seller / Buyer / Landlord / Tenant
- Property type
- Full Name
- Phone Number
- Email Address

Role-specific client details:
Seller/Landlord:
- Property Address
- Street Address
- City
- State
- ZIP Code

Buyer/Tenant:
- Areas of Interest

Accepted terms:
- Services selected from the agent preset
- Broker compensation / agency agreement terms
- Additional terms, if any

Important:
- This is a summary of submitted Direct Hire terms.
- It is not a public listing detail page.
- It should not include bid placement controls.
- It should not include “Only agents can place bids.”
- It should not create or display a new listing ID.

Step 4 — Preserve other flows

Make sure this fix only applies when the request came from Direct Hire acknowledge submit.

Do not break:
- normal Hire Agent listing creation
- Seller/Buyer/Landlord/Tenant marketplace listings
- agent bidding on marketplace listings
- Offer Listing Presets
- Request Changes / Counter Terms flows
- existing listing detail pages

Step 5 — Verification

Test all four Direct Hire acknowledge URLs:

Buyer:
Submit from:
/hire/agent/direct/{agentId}/buyer/{propertyType}/acknowledge

Expected:
- no BuyerAgentAuction/listing record is created
- redirects to Direct Hire submitted summary page
- submitted client details and accepted terms display

Seller:
Expected:
- no SellerAgentAuction/listing record is created
- redirects to summary page
- submitted client details and accepted terms display

Landlord:
Expected:
- no LandlordAgentAuction/listing record is created
- redirects to summary page
- submitted client details and accepted terms display

Tenant:
Expected:
- no TenantAgentAuction/listing record is created
- redirects to summary page
- submitted client details and accepted terms display

Regression checks:
- Creating a normal Hire Agent listing still creates a listing
- Marketplace listing detail pages still work
- Agent bid/counter flows still work
- Offer Listing flows still work
- Limited Service untouched

Stop before editing if the Direct Hire acknowledge submit handler is shared with listing creation in a way that cannot be safely separated without a dedicated Direct Hire branch.

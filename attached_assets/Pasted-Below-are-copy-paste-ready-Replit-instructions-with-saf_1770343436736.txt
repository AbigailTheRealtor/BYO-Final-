## 2) Fix the “Saved Drafts” page query to include Landlord drafts

There are usually two patterns:

1. A unified drafts page that merges multiple listing tables (tenant/buyer/seller/landlord)
2. Separate drafts sections per listing type

### B. Add/repair the Landlord draft dataset

Wherever drafts are loaded, ensure **Landlord drafts** are loaded using the correct draft conditions.

**Common reasons Landlord drafts don’t appear**

* The drafts page only queries tenant/buyer/seller tables and never added landlord
* It filters `status='Draft'` but Landlord saves `status='draft'` (case mismatch)
* It filters `submitted_at IS NULL` but Landlord uses `is_submitted` or `status`
* It filters by `user_id`, but Landlord uses `created_by` or `owner_id`
* Soft deletes (`deleted_at`) or visibility flags hide them unintentionally

### C. Standardize the Landlord “draft detection” logic on the drafts page

Implement Landlord draft detection as:

* Must match *any* of the platform’s draft markers used in that flow:

  * `status = 'draft'` (preferred)
  * OR `is_draft = 1`
  * OR `submitted_at IS NULL` **only if** that’s how Landlord works

**Safeguard:** do **not** change draft logic globally. Only add/adjust the Landlord query.

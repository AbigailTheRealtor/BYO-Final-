2. Apply referral attribution ONLY when a brand-new listing record is created
For each newly created listing:
- if the listing already has referring_agent_id populated, do nothing
- else determine attribution source in this priority order:
  A. session referral
     - referral.agent_id
     - referral.code
     - referral.captured_at
  B. authenticated user referral fallback
     - users.referred_by_agent_id
     - users.referral_source_code
     - users.referral_captured_at
  C. none

3. If attribution source A or B exists, write to the new listing:
- referring_agent_id
- referral_source_code
- referral_captured_at
- referral_locked = true

4. Update referral_visits if session('referral.visit_id') exists:
- set listing_id = newly created listing id
- set converted_to_listing = true
Fail gracefully if the row is missing or stale.
Do not break listing creation.

5. Increment listing_count on agent_referral_links only when:
- a brand-new listing was actually attributed
- and the agent/code pair is valid enough to identify the referral link row
Use atomic increment.

6. Preserve safety rules
- never overwrite listing referral once set
- do not change referral attribution on edits/updates
- do not clear referral session yet
- if session referral is absent but user fallback exists, still attribute
- if both are absent, do nothing
- if session visit id is stale/missing, still attribute the listing without crashing

## Goal

> **(555) 555-5555**

Users should **not** need to type parentheses `(` `)` or dashes `-`.

1. **Do NOT rename** the field, `wire:model`, database column, or validation key.
2. **Do NOT store formatting characters** in the database.

   * Store digits only: `5555555555`
3. **Do NOT change** required/optional status.
4. **Do NOT affect existing saved listings** (edit mode must re-format correctly).
5. Reuse the **same formatting approach everywhere** (no mixed methods).

## Step 1 — Use a Single Phone Mask for ALL Roles

Identify how phone formatting is currently implemented in the version that already works (if one exists), or add a **single shared phone-mask solution**.

The phone input must:

* Accept digits only
* Auto-insert formatting while typing
* Limit input to **10 digits**
* Gracefully handle paste (`5555555555`, `555-555-5555`, etc.)

---

## Step 2 — Apply the Mask to the Phone Input (UI Only)

Attach the phone mask to the **Phone Number input itself** (not the icon wrapper).

**Required display behavior:**

* User types: `5555555555`

## Step 3 — Re-Initialize Mask After Livewire Updates (Critical)

Because these forms use Livewire, the phone input may be re-rendered when:

* switching tabs
* saving drafts
* editing listings
* toggling conditional fields

### Requirement

The phone mask **must be re-initialized after every Livewire DOM update**, not just on page load.

**Safeguard:**
Ensure the initializer is idempotent (does not double-bind).

---

## Step 4 — Sanitize Before Save (Digits Only)

Before validation/save:

* Strip all non-numeric characters
* Persist **digits only** to the database

**Example:**

* UI shows: `(555) 555-5555`
* Stored value: `5555555555`

This must apply to **Tenant / Buyer / Seller / Landlord** consistently.

## Step 5 — Edit Mode Behavior

When editing an existing listing:

* If the stored value is `5555555555`
* The input should automatically render as `(555) 555-5555`

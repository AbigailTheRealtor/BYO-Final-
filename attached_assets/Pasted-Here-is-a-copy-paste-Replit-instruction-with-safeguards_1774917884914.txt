## Replit Instructions With Safeguards

### Goal

For **Hire a Tenant’s Agent** listings, change the bidding behavior so that each agent has only **one bid thread per listing**.

Right now, when the same agent edits or rebids, the system is creating multiple visible bid entries for the same agent. That is confusing.

Instead, make it work like this:

* **1 agent = 1 bid thread per listing**
* If that same agent edits, counters, resubmits, or updates terms, it should update the **same bid thread**
* Keep a **version/history log** inside that bid thread
* On the Tenant side, only show **one visible bid card per agent**
* The comparison/dashboard should use only the **latest active version** of that agent’s bid

---

## Scope

Fix this only for the **Hire a Tenant’s Agent** bid flow first.

Do not touch:

* Buyer
* Seller
* Landlord

unless the same logic is already shared in a clearly safe, reusable way and can be updated without breaking those flows.

---

## Safeguards

### 1. No destructive database changes

* Do **not** delete any data
* Do **not** rename or remove any existing tables or columns
* Do **not** drop old bid records
* If a schema update is absolutely needed, it must be **add-only** and backward-safe

### 2. Scope to active Tenant bid flow only

Only edit the files/components/routes actually used by:

* Tenant agent bid create
* Tenant agent bid edit
* Tenant agent counter flow
* Tenant bid display/history/comparison

Do not edit legacy or unrelated files just because the names look similar.

### 3. Preserve all existing bid history

* Do not wipe old bids
* Do not destroy previous revisions
* Existing duplicate bids should remain in the database unless there is a safe non-destructive way to hide/archive/merge display behavior
* Priority is fixing **future behavior** and **current display logic**

### 4. Do not change labels/tooltips/order unnecessarily

Keep the current UI wording unless absolutely needed for clarity.

### 5. Do not break accepted/countered/rejected logic

The Accept / Counter / Reject actions must stay tied to the correct agent bid thread.

---

## Desired behavior

### Current problem

The same agent is able to generate multiple visible bids on the same Tenant listing, so it can appear like:

* Agent 1 Bid
* Agent 1 Bid again
* Agent 1 Bid again

or the system may treat edits like new bids instead of updates.

### Correct behavior

For each **listing + agent** combination, there should only be **one active bid thread**.

If the same agent interacts again on that same listing:

* edit bid = update same thread
* counter response = update same thread
* resubmit = update same thread
* save revised terms = update same thread

Internally, keep:

* version history
* timestamps
* status changes
* prior values if already supported

But visually, the Tenant should see only **one current bid card per agent**.

---

## Recommended logic

### Unique business rule

For Tenant Agent bidding, enforce this rule:

**One active bid thread per agent per listing**

That means when an agent tries to submit a bid on a listing where they already have a bid thread:

* do not create a second visible sibling bid
* instead load/update the existing bid thread

### Safe implementation approach

If your current schema already has a bid table and separate meta/history records, reuse that structure.

Preferred pattern:

#### Master bid thread

One primary bid record per:

* `listing_id`
* `agent_user_id`

#### Revisions/history

Use either:

* an existing history/meta/version table, or
* an add-only JSON/meta history field, or
* a new add-only bid_versions table if truly necessary

But do **not** replace old records destructively.

## Required changes

### A. Submission logic

When an agent submits a bid on a Tenant listing, first check:

* does this agent already have an existing bid thread for this listing?

If **yes**:

* update that existing thread instead of inserting a new visible bid row

If **no**:

* create the initial bid thread

---

### B. Edit logic

When an agent edits a bid:

* do not create a new bid record
* update the same bid thread
* record a revision/history entry if history exists

---

### C. Counter logic

When a Tenant counters and the agent responds:

* keep everything inside the same bid thread
* do not create a second sibling bid row for the same agent

---

### D. Display logic

On the Tenant listing/bid comparison page:

* show only one bid card per agent per listing
* use the latest active version/status for display

If duplicate historical records already exist from old behavior, do not delete them. Instead:

* either group them by agent + listing and show the newest valid/current one
* or mark older duplicates as historical/non-primary in display logic only

---

### E. Dashboard logic

On the Agent dashboard:

* the bid should appear as one thread entry
* edits/counters should update that same entry
* recent notices can still reflect actions, but they should point back to the same bid thread

---

## If old duplicates already exist

Do **not** mass delete or hard-merge records.

Instead, implement a safe rule for Tenant display:

For any Tenant listing, if multiple records exist for the same `agent_user_id + listing_id`, display only the current primary/latest active one, and treat the others as historical unless they are needed for audit view.

If needed, add a safe field like:

* `parent_bid_id`
* `is_primary`
* `superseded_at`
* `version_number`

Only if absolutely necessary, and only as add-only schema changes.

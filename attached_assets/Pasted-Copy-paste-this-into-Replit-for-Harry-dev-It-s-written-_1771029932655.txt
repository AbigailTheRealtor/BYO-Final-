# PART A — FIX “POOL” NOT SAVING + NOT SHOWING + NOT REPOPULATING

## A1) Confirm DB column exists (or add it)

**Goal:** Persist pool for Income listings.

* Check the listings table used by Seller Agent listings (whatever table stores these listings).
* Confirm a column exists for pool, preferably:

  * `has_pool` (BOOLEAN, nullable)

### If missing: create migration

* Add column: `has_pool` boolean nullable

✅ Safeguard:

* Use a nullable boolean so “not answered” ≠ “No”.
* Do **not** store as “Yes/No” string unless the rest of the system does. Pick one and keep consistent.

## A2) Add model protection

In the **Seller listing model** (the Eloquent model used for these listings):

* Add `has_pool` to `$fillable`
* Add `has_pool` to `$casts` as boolean

```php
protected $casts = [
  'has_pool' => 'boolean',
];
```

✅ Safeguard:

* If `$guarded = []` is NOT used, missing fillable will silently fail saving.

## A3) Fix Livewire Create save mapping

In the **Create Livewire component** for Hire a Seller’s Agent:

* Ensure the input binds to a Livewire property that is used in the save method:

  * `wire:model="has_pool"`

In the save/submit method:

* Explicitly assign:

```php
$listing->has_pool = $this->has_pool;
```

✅ Safeguard:

* Do not rely on `$listing->fill($this->all())` unless you’re 100% sure the key names match.
* Add an assertion log (temporary) to confirm the Livewire value is not null at save.

## A4) Fix Livewire Edit repopulation

In the **Edit Livewire component** mount/load method:

* Explicitly load from DB:

```php
$this->has_pool = (bool) $listing->has_pool;
```

✅ Safeguard:

* If you store as string (“Yes/No”), map it consistently both directions:

  * Save: `Yes/No`
  * Load: `Yes => true`, `No => false`

# PART C — SERVICES: MUST DISPLAY ONLY SELECTED SERVICES, IN THE EXACT ORDER OF CREATION FORM

## C1) Root cause

Service order breaks when services are stored as:

* associative arrays (keys reorder)
* database joins (ORDER BY missing)
* booleans (rendered from a config list not tied to user selection order)
* grouped loops that sort by label

## C2) Required storage approach (ordered)

Store selected services in the listing as an **ordered JSON array** exactly as selected in the form.

Example field:

* `selected_services` (JSON)

Store as:

```json
[
  "List the property on the local Multiple Listing Service (MLS)",
  "Syndicate the listing to third-party platforms (e.g., Zillow.com...)",
  ...
]
```

✅ Safeguard:

* Do NOT re-sort this array on save.
* Do NOT “group then sort” before rendering.

## C3) Save logic (Create + Edit)

When saving:

* Build the ordered list from the creation form master list, filtering by checked items, preserving that master order.

Pseudo:

* `$master = config('services.income'); // in the exact UI order`
* `$selected = array_values(array_filter($master, fn($s)=>in_array($s['id'],$this->selectedIds)));`
* Save `$selected` in same order.

✅ Safeguard:

* Rendering should read from the stored ordered array, not from the master config.

## C4) Live listing rendering logic

Render category headers + bullet list using the stored ordered array.
If you store only IDs, render by ID but **keep stored order**.

✅ Safeguard:

* Ensure the following currently missing services are included in the Income master list AND can be selected AND render:

**MUST NOT BE MISSING on listing if selected:**

* “Assist with in-person or electronic contract signing, including e-signature setup and secure delivery of executed purchase agreements, addenda, and disclosures to all parties”
* “Coordinate with the Buyer’s Agent, Buyer’s Lender, Title, Escrow, and/or Attorney to prepare for Closing”
* “Assist in estimating Gross Rent Multiplier (GRM), Capitalization Rate (Cap Rate), or Price per Unit based on listing details and income property comparables”

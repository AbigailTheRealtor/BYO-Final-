# 2) ACCEPTABLE PROPERTY STYLES — MUST NOT FLASH WHEN OTHER PROPERTY PREFERENCES FIELDS CHANGE

### IMPORTANT CLARIFICATION

The problem is **not** mainly that Acceptable Property Styles itself flashes only when directly selecting it.

The real issue is:

* **Acceptable Property Styles may already be selected or sitting on the screen**
* then the user fills in **other fields under Property Preferences**
* and that causes the options inside **Acceptable Property Styles** to flash/re-render

So this bug is a **cross-field re-render / instability issue** inside the Property Preferences tab.

---

## Current problem

Under **Property Preferences**, when the user interacts with other fields in that tab, the **Acceptable Property Styles** options flash on screen.

That means **Acceptable Property Styles is being unnecessarily re-rendered, re-initialized, or visually reset** when unrelated Property Preferences fields change.

### Required fix

Make **Acceptable Property Styles** remain stable when other Property Preferences fields are updated.

Expected behavior:

* if the user already selected values in **Acceptable Property Styles**, those values should stay visually stable
* if the user clicks or types in other Property Preferences fields, **Acceptable Property Styles should not flash**
* unrelated Property Preferences field updates should not destroy/rebuild the Acceptable Property Styles widget
* no placeholder flicker
* no temporary UI reset
* no visible repaint of Acceptable Property Styles caused by other fields in the same tab

### Safeguard

Do not rewrite the entire Property Preferences tab.
Do not change working behavior of **Non-Negotiable Amenities and Property Features**.
Only stop the unnecessary re-render/re-init affecting **Acceptable Property Styles** when other Property Preferences fields update.

---

## Implementation direction

Check specifically whether **Acceptable Property Styles** is being affected by:

* a parent container Livewire re-render when unrelated fields change
* Select2 destroy/re-init triggered by generic tab-level hooks
* duplicated initialization hooks on `message.processed`
* hidden bridge syncing that resets the DOM even when the value has not changed
* keyed/blade rendering that causes the component subtree for Acceptable Property Styles to be rebuilt
* placeholder/default logic re-running when other Property Preferences fields update

The goal is to make **Acceptable Property Styles isolated and stable** so unrelated field changes in the same tab do not visually disturb it.

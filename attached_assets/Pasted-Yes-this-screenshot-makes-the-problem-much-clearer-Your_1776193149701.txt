## Step 3 — Normalize Conversation Resolution

If the current system does not already do this cleanly, add a single reusable resolver service.

Create only if needed:

`app/Services/ConversationResolver.php`

Use this logic:

* given:

  * authenticated sender ID
  * recipient ID
  * optional listing ID
  * optional bid ID
* find existing conversation between those exact two users for that context
* if none exists, create one
* do not create duplicate conversations for the same pair/context

Important:
If your platform wants one general thread per pair of users, reuse that.
If your platform wants listing-specific threads, then require `listing_id` in the resolver.
If your current design is already listing-context based, preserve that design.

Pseudo logic:

```php
find conversation where participants are these two users
and, if messaging is listing-contextual, listing_id matches
and, if bid-contextual, bid_id matches when applicable

if not found:
    create conversation with correct participants and context
```

Do not allow:

* self-conversations
* null recipient
* orphan conversations with only one participant

## Step 4 — Fix Conversation Participant Display in Dashboard

Your screenshot strongly suggests the dashboard cannot properly resolve the “other user.”

In the conversation model, add or fix a helper that returns the other participant relative to the logged-in user.

Example logic:

```php
public function otherParticipant($currentUserId)
{
    if ($this->user_one_id == $currentUserId) {
        return User::find($this->user_two_id);
    }

    if ($this->user_two_id == $currentUserId) {
        return User::find($this->user_one_id);
    }

    return null;
}
```

If you use a participants relation instead, adapt it accordingly.

Then in the Messages dashboard left sidebar:

* show the other user’s real name
* never show blank avatar/name if participant exists
* if missing, show a temporary fallback like `Unknown User`
* do not let a null participant break the thread rendering

## Step 8 — Ensure Agent-to-User Messages Route to the Correct User

This is the biggest business logic requirement.

When an **agent messages from a bid/listing context**, the recipient must be the actual listing owner or user attached to that listing flow.

Examples:

* Tenant listing → recipient is the tenant who created the listing
* Buyer listing → recipient is the buyer
* Seller listing → recipient is the seller
* Landlord listing → recipient is the landlord

Do **not** derive recipient from role label text.
Do **not** guess based on display strings.
Use the actual listing owner user ID or correct conversation participant mapping.

If messaging starts from bid context, confirm the bid’s listing owner or intended counterparty is used.

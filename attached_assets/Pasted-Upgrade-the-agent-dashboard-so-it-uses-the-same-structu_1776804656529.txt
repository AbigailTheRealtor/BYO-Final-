Upgrade the agent dashboard so it uses the same structural framework, layout patterns, terminology, and role-based experience as the current user dashboard, while preserving agent-side functionality.

Objective

The agent dashboard should match the user dashboard’s modern Hire Agent framework and UX pattern.

I want the agent dashboard to feel like the agent-side version of the same product, not an older separate system.

That means:

same role-based structure
same terminology
same general page behavior/layout patterns
same quality of dashboard experience
but agent-specific content/actions instead of user-specific ones
Important distinction

Do not make the agent dashboard identical to the user dashboard in data or actions.

Instead:

preserve agent-side transaction behavior
preserve agent-specific listings/bids/actions
preserve agent permissions
preserve Offer Playoff gating already implemented

But modernize the agent dashboard so it mirrors the user dashboard’s framework and consistency.

What needs to match the user dashboard
1. Role-based Hire Agent structure

The agent dashboard should use the same role naming and framing as the user dashboard:

Hire Tenant’s Agent
Hire Landlord’s Agent
Hire Buyer’s Agent
Hire Seller’s Agent

This structure should be reflected consistently across:

sidebar labels
page headings
dashboard sections
entry points
role filters/tabs
listing/bid views

Do not leave old role naming like:

Tenant Agent
Landlord Agent
Buyer Agent
Seller Agent
Auctions Listing / Agent Auctions Listing

unless needed internally in code only.

Visible UI should use the user-dashboard standard naming.

2. Dashboard home structure

Audit the current agent dashboard home and compare it to the user dashboard home.

I want the agent dashboard home to follow the same improved UX pattern as the user dashboard where appropriate, including:

clearer role-based organization
better summary sections
better visual hierarchy
cleaner quick access
cleaner notices/activity placement

But adapted for agent needs.

Agent dashboard should still show agent-relevant content such as:

My Hire Agent Listings
bids by role
recent notices
messages
QR & Hire Me
profile settings
any agent-side accepted/countered workflow notices
3. Role-specific pages should match the newer user-side layout pattern

The role-specific agent pages currently look older and inconsistent.

Examples from screenshots:

“Tenant’s Agent Auctions Listing”
“Landlord’s Agent Auctions Listing”
“Buyer’s Agent Auctions Listing”
older table/list layouts

These should be modernized to follow the same user-dashboard visual and structural pattern where appropriate:

consistent page titles
consistent filters/dropdowns
consistent listing cards/tables
consistent role naming
consistent empty states
consistent action placement
consistent notice/match/status presentation

Keep agent-side actions intact, but modernize the shell and structure.

4. Match feature parity where appropriate

Audit which dashboard features exist on the user side that should also exist on the agent side in an agent-appropriate form.

Examples to audit:

summary/home widgets
role grouping
listing counts
notices placement
recent activity structure
visual grouping of accepted / active / countered / awaiting response states
cleaner role switching/navigation behavior

Do not blindly copy user-side features.
Instead, bring over the same framework and usefulness, adapted to agent workflows.

Audit and compare these areas between user and agent dashboard:

dashboard home

Visible UI naming cleanup required

Standardize visible agent-side UI to the Hire Agent naming style used by the user dashboard.

Examples of desired visible naming:

Hire Tenant’s Agent Listings
Hire Landlord’s Agent Listings
Hire Buyer’s Agent Listings
Hire Seller’s Agent Listings

Avoid older visible titles like:

Tenant’s Agent Auctions Listing
Landlord’s Agent Auctions Listing
Buyer’s Agent Auctions Listing
Seller Agent Bids without apostrophe where not intended
generic “Auction” naming if this is really a Hire Agent listing view

Use the user dashboard’s naming logic as the visible UI standard.

Execution order
Step 1 — Audit first

Before changing code, audit:

which Blade/Livewire/controller files power the user dashboard structure
which files power the agent dashboard structure
which shared partials exist
what can be safely reused from the user dashboard
what must remain agent-specific
whether agent pages are using older templates that should be aligned or partially refactored

Do not change code yet.
First report the exact files and the safest implementation path.

Step 2 — Minimal shared modernization plan

Choose the smallest safe implementation path:

reuse shared dashboard partials where possible
align headings/labels/layouts first
then align page shells and section organization
then align notices/widgets where safe

Do not do a risky full rewrite if smaller targeted alignment gets us there.

Step 3 — Implement with parity mindset

Modernize the agent dashboard to the same framework as the user dashboard while preserving agent-specific logic and permissions.

Safeguards
No DB renames
No DB deletes
No destructive route deletions
No breaking Hire Agent listing creation
No breaking bidding/counter/accept/reject flows
No breaking accepted bid summary generation
No breaking Offer Playoff gating
No cross-role regression for Tenant / Landlord / Buyer / Seller flows
Do not rebuild from scratch if targeted alignment is sufficient
Stop and report if user and agent dashboards are too structurally divergent for a safe partial refactor
Deliverable

After the audit and implementation, report:

files audited
files changed
what user-dashboard structures/partials were reused
what visible titles/labels were standardized
what agent dashboard sections were aligned to the user framework
what role-specific pages were updated
confirmation that all four agent roles still work
confirmation that agent-specific actions still work
confirmation that no Hire Agent flows were broken
Final design goal

The final result should feel like:

same product, same dashboard family, same Hire Agent framework
but with:
user-side controls for clients
and
agent-side controls for agents

Not two totally different dashboard systems.

## Likely root cause

One of these is almost certainly happening:

* generic accepted summary route loads by `id` only
* accepted summary model/table is shared and not role-scoped
* seller listing is passing the wrong accepted summary id
* seller accepted summary creation saved the wrong record id
* shared accepted-summary page resolves first matching row and lands on Buyer because IDs overlap

---

## Required fix

Make Seller accepted-summary access **strictly seller-scoped**.

### Rule

A Seller listing must only ever open:

* Seller accepted summary record
* Seller bid
* Seller listing
* Seller acknowledgement state

It must never open a Buyer summary just because summary IDs overlap.

---

## Preferred implementation

Use whichever is safest with the least blast radius:

### Option A — safest if route source is wrong

Fix the Seller listing page so it passes the correct **Seller accepted summary identifier** and/or seller-specific role context when linking to the accepted summary.

### Option B — safest if loader is too generic

Harden the accepted-summary loader so it requires role-aware resolution, for example:

* seller route resolves seller summary only
* buyer route resolves buyer summary only
* landlord route resolves landlord summary only
* tenant route resolves tenant summary only

### Option C — if using shared route

If `/accepted-bid-summary/{id}` must stay generic, then the loader must also receive or derive a trusted role context from the source flow and only query the matching role/service.

* `find($id)` on a shared summary model without role scoping
* lookup by bid id only if bid ids overlap across roles
* fallback from seller to buyer if seller not found
* deriving role from rendered summary HTML
* keeping seller on a generic accepted summary route that can open any role’s record

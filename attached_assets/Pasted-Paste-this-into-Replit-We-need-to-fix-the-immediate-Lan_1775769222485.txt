# PART 2 — CENTRALIZE ACTION-ROW LOGIC

## Problem to solve

We do not want separate versions of this logic living in multiple files with slightly different variables.

## Refactor target

Create **one small shared Blade partial** only for the bid action-state area.

Example purpose:

* render active owner buttons
* render accepted state
* render rejected state
* render countered state
* render pending/non-owner states as already intended

This partial should only control the **action row / state buttons area**, not the entire modal or entire bid card.

## Suggested shared partial name

Use something like:

* `resources/views/.../partials/bid_action_row.blade.php`

Use the project’s existing folder structure and naming conventions.
Do not introduce a whole new architecture.
Just create one shared partial in the most appropriate existing bid-view partials folder.

## Inputs the partial should receive

Pass all required context explicitly, for example:

* bid
* auction
* isListingOwner if already computed
* isBidOwner if already computed
* route/action URLs already generated by parent
* role identifier if needed
* any flag needed for modal vs page styling only

Do **not** let the partial reach all over the page for unrelated variables unless necessary.

## Inside the shared partial

Normalize state in one place.

Example pattern:

```php
@php
    $_rawAccepted = data_get($bid, 'is_accepted');
    $_state = (string) $_rawAccepted;
    $_listingOwnerId = (string) data_get($auction, 'user_id');
    $_authId = (string) auth()->id();
    $_isOwner = $_listingOwnerId !== '' && $_listingOwnerId == $_authId;
    $_isSold = in_array(data_get($auction, 'is_sold'), [true, 'true', 1, '1'], true);
@endphp
```

Then use one canonical branch structure for:

* active + owner
* accepted
* rejected
* countered
* pending / info-only
* non-owner states

## Important

Do **not** change the intended business rules.
Only centralize them so all rendering locations use the same logic.

## FINAL SAFEGUARDS BEFORE IMPLEMENTATION

This is a **dashboard UI + read-only data layer task only**.

### HARD STOP RULES

If any step requires modifying:

* listing creation
* listing edit
* bid creation/edit
* counter logic
* accept/reject logic
* accepted summary services
* acknowledgement/signature logic

➡️ **STOP and report instead of implementing**

---

### ERROR-HANDLING RULE

If you encounter any **dashboard-only** error during implementation or testing, you may fix it **only if** the fix stays within:

* `DashboardController.php`
* `resources/views/dashboard.blade.php`
* small safe dashboard header/partial tweaks directly used by the dashboard

If the error requires touching listing pages, bid pages, counter pages, summary services, or core workflow logic:

➡️ **STOP and report instead of implementing**

---

### DATA RULE

All dashboard data must be:

* read-only queries
* derived from existing tables
* using existing fields exactly as they are

❌ Do NOT normalize or refactor bid status fields
❌ Do NOT create new status systems
❌ Do NOT modify database structure

---

### ROUTING RULE

* Only use existing routes already confirmed in Phase 2
* Do NOT create new routes unless absolutely necessary
* Do NOT change existing route behavior

---

### VIEW SCOPE RULE

You are ONLY allowed to modify:

* `resources/views/dashboard.blade.php`
* `app/Http/Controllers/DashboardController.php`

Optional:

* small safe tweaks to `resources/views/layouts/partials/dashboard_user_section.blade.php` if needed for UI

---

### DO NOT TOUCH

* any `hire_*` views
* any `agent_biding_listing/*`
* any `AcceptedBidSummaryService`
* any counter/bid partials
* any Livewire components tied to bidding
* any listing forms

---

### OUTPUT REQUIREMENT

After implementation, you must provide:

1. Exact files changed
2. Exact sections replaced
3. New queries added (read-only only)
4. Confirmation no business logic was modified
5. Screens tested (list all pages clicked and verified)

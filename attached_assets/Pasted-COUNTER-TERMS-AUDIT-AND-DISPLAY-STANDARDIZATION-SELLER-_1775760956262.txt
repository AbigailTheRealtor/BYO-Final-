COUNTER TERMS AUDIT AND DISPLAY STANDARDIZATION — SELLER, BUYER, LANDLORD, TENANT
READ FULLY BEFORE MAKING ANY CHANGES

We need a focused audit and cleanup of the counter terms display across all 4 roles.

TARGET GOAL
Make counter terms easier to read, more accurate, and negotiation-focused.

The current issue is that the counter terms pages appear to:
1. show lowercase yes/no values in some places
2. show compensation structure labels instead of the actual entered values
3. show requested services with checkmarks, which looks like a form instead of a negotiation comparison
4. appear to carry highlight/diff behavior forward from the original bid instead of only comparing against the immediately previous version

We need a full audit and safe standardization plan for Seller, Buyer, Landlord, and Tenant.

IN SCOPE
Audit and standardize only the counter terms display/rendering for these 4 role flows:
1. Seller
2. Buyer
3. Landlord
4. Tenant

This includes the role-specific View Counter Terms pages and any directly related Blade rendering/formatting logic used only for showing counter terms.

OUT OF SCOPE — DO NOT TOUCH
Do not modify:
- listing creation
- listing edit
- listing detail core behavior
- add-bid forms unless absolutely required to inspect source values
- bid submission logic
- counter submission logic
- accepted bid summary
- notifications
- match score formulas
- backend business logic unrelated to counter display
- route definitions
- controllers
- database schema
- migrations

If any requested change would require changing core business logic or schema, stop and report it instead.

PRIMARY AUDIT QUESTIONS

1. YES / NO FORMATTING
Audit all 4 counter terms displays and confirm whether yes/no style values are rendered consistently.
Target standard:
- Yes
- No
Both must always be capitalized.

2. COMPENSATION DISPLAY FORMAT
Audit all compensation sections in counter terms for all 4 roles.

Current problem example:
A lease fee may display only:
- Flat Fee + Percentage of Gross Lease Value

That is not sufficient for counter terms display.

Target format:
Show the actual entered values, not just the structure label.

Examples:
- $4,000 + 4% of Gross Lease Value
- 5% of Total Purchase Price
- $2,500 flat fee
- $1,000 Lease-Option Fee
- $2,000 + 3% if Purchase Option is Exercised

If helpful, the structure type may appear as a small secondary label, but the primary display must show the actual entered values.

Audit whether Seller, Buyer, Landlord, and Tenant all do this correctly.
If not, standardize the display.

3. REQUESTED SERVICES DISPLAY FORMAT
Audit how requested services are shown on counter terms pages.

Current concern:
Services appear with checkmarks like a form or checklist.

That is not the best format for negotiation comparison.

Target standard:
Do not use checkmarks for services in counter comparison display.

Instead, use clear text-based comparison sections such as:
- Added Services
- Removed / Not Included
- Changed Services if applicable

Unchanged services should not be visually overemphasized.

4. DIFF / HIGHLIGHT LOGIC
This is the most important audit.

Determine exactly what each counter terms page is currently comparing against:
- the original bid every time
- or the immediately previous counter / previous version

Target standard:
Each counter should compare against the immediately previous version, not always the original bid.

Correct behavior:
- First counter compares to original bid
- Second counter compares to first counter
- Third counter compares to second counter
- and so on

We do NOT want old differences from the original bid to keep showing as if they changed again in later rounds.

Audit all 4 roles and confirm whether current diff behavior is:
- correct
- partially correct
- wrong

If current rendering is carrying forward original-bid highlights into later counters, fix the display logic so only fields changed from the immediately previous version are highlighted.

5. ONLY HIGHLIGHT WHAT ACTUALLY CHANGED
For all 4 roles, the counter terms display should only visually emphasize:
- Changed
- Added
- Removed / Not Included

Do NOT over-highlight unchanged fields.

Unchanged fields should either:
- display normally with no highlight
or
- be omitted from the change-summary sections if that is already supported safely

STANDARD DISPLAY RULES TO APPLY IF SAFE

Apply these standards consistently across Seller, Buyer, Landlord, and Tenant:

A. Capitalization
- Yes
- No

B. Compensation
Primary line should show the real numbers and percentages entered by the user.
Do not show only the compensation structure label.

C. Services
Remove checkmark-style presentation for counter comparison.
Use negotiation-style text groupings instead.

D. Diff basis
Compare current counter to immediately previous version.

E. Highlighting
Only highlight:
- changed
- added
- removed / not included

Do not make unchanged fields look like active changes.

PREFERRED COUNTER DISPLAY APPROACH
If the current code supports it safely, the best display structure is:

1. Counter Summary
- Submitted by
- Date/time
- Compared against: Original Bid or Previous Counter
- Current status

2. Broker Compensation and Agency Terms
- Changed
- Added
- Removed / Not Included

3. Additional Details
- Show changed fields clearly

4. Requested Services
- Added Services
- Removed / Not Included
- No form-style checkmarks

Do NOT introduce major UI refactors if that risks breaking the current pages.
Accuracy and consistency are more important than redesign.

I hear you. The “dual sync” explanation sounds nice, but if it still doesn’t persist across create, draft, edit, and display, then one of two things is happening:

1. Livewire is never actually receiving the correct array state, so nothing real gets saved
2. It is saved, but your hydrate, mount, or display builder is reading from a different field, so it looks like it “didn’t save”

The fastest way to stop bleeding money is to force a truth test and then apply a bulletproof bridge that does not depend on hidden inputs timing.

Below is the approach I’d use on a Livewire + Select2 + wire:ignore setup when “setJsonModel” and “dual sync” still fail.

A. The UI selection never makes it into the Livewire public property
B. The Livewire property updates, but save is not writing it
C. Save writes it, but edit mount is not restoring it
D. Edit restores it, but listing display is reading from the wrong place

## Step 2, Stop using the JSON hidden input bridge as the primary path

Hidden inputs are the #1 reason this fails across drafts and edits because wire:model.defer plus DOM timing plus wire:ignore is inconsistent.

Instead, do this:

1. Livewire property is the source of truth
2. JS sets Livewire property directly using @this.set
3. JS also rehydrates Select2 from Livewire after any render

This removes dependence on JSON decode, hidden input discovery, and “processed” timing weirdness.

---

## Step 3, Use a stable Select2 Livewire bridge that always rehydrates

Use this exact pattern once, then apply it to every Select2 multi field.

### Blade markup

Wrap your select in a unique wrapper and keep the select wire ignored:

```html
<div id="acceptable_property_conditions_wrap" wire:ignore>
    <select id="acceptable_property_conditions"
            class="form-control"
            multiple>
        <!-- options -->
    </select>
</div>
```

Important, make sure the select has `multiple` for any multi select field.

---

### JS bridge, one function per field

Put this in the page JS for that form, create and edit both.

```js
function initSelect2LivewireMulti({
  selectId,
  livewireProp,
  placeholder = 'Select',
}) {
  const el = document.getElementById(selectId);
  if (!el) return;

  const $el = window.jQuery(el);

  if ($el.data('select2')) {
    $el.off('change.select2Livewire');
    $el.select2('destroy');
  }

  $el.select2({
    width: '100%',
    placeholder,
    closeOnSelect: false,
  });

  function normalize(val) {
    if (val === null || val === undefined) return [];
    if (Array.isArray(val)) return val;
    if (typeof val === 'string' && val.trim() === '') return [];
    try {
      const parsed = JSON.parse(val);
      return Array.isArray(parsed) ? parsed : [];
    } catch (e) {
      return [];
    }
  }

  function setFromLivewire() {
    const current = normalize(@this.get(livewireProp));
    $el.val(current).trigger('change.select2');
  }

  $el.on('change.select2Livewire', function () {
    const selected = $el.val() || [];
    @this.set(livewireProp, selected, true);
  });

  setFromLivewire();
}
```

That last `true` forces an immediate sync.

---

### Call it on load and after every Livewire update

This is the piece most people miss. If you don’t rehydrate after every update, the select will drift.

```js
document.addEventListener('livewire:load', () => {
  initSelect2LivewireMulti({
    selectId: 'acceptable_property_conditions',
    livewireProp: 'acceptable_property_conditions',
    placeholder: 'Select acceptable conditions',
  });

  Livewire.hook('message.processed', (message, component) => {
    initSelect2LivewireMulti({
      selectId: 'acceptable_property_conditions',
      livewireProp: 'acceptable_property_conditions',
      placeholder: 'Select acceptable conditions',
    });
  });
});
```

Repeat this exact call for each field that fails.

Example:

```js
initSelect2LivewireMulti({
  selectId: 'acceptable_unit_type',
  livewireProp: 'acceptable_unit_type',
  placeholder: 'Select acceptable unit types',
});
```

If your property names differ per component, match them exactly.

## Step 4, Fix the backend so Livewire never overwrites arrays with strings

Even if JS is correct, Livewire can overwrite arrays during mount or hydration if you are doing json_decode or pulling from metadata inconsistently.

In the Livewire component, enforce arrays at three points:

### A. Property definitions

```php
public array $acceptable_property_conditions = [];
public array $acceptable_unit_type = [];
public array $acceptable_property_style = [];
```

### B. Mount normalization

```php
$this->acceptable_property_conditions = is_array($this->acceptable_property_conditions)
    ? $this->acceptable_property_conditions
    : (array) ($this->acceptable_property_conditions ? json_decode($this->acceptable_property_conditions, true) : []);

$this->acceptable_unit_type = is_array($this->acceptable_unit_type)
    ? $this->acceptable_unit_type
    : (array) ($this->acceptable_unit_type ? json_decode($this->acceptable_unit_type, true) : []);

$this->acceptable_property_style = is_array($this->acceptable_property_style)
    ? $this->acceptable_property_style
    : (array) ($this->acceptable_property_style ? json_decode($this->acceptable_property_style, true) : []);
```

### C. Before save, normalize again

Right before your save writes, do:

```php
$this->acceptable_property_conditions = array_values(array_filter($this->acceptable_property_conditions ?? []));
$this->acceptable_unit_type = array_values(array_filter($this->acceptable_unit_type ?? []));
$this->acceptable_property_style = array_values(array_filter($this->acceptable_property_style ?? []));
```

Now your save path can store JSON or store directly, but the in memory Livewire values are always real arrays.

## Step 5, Fix “No Preferences (open to any condition)” not saving

This usually fails because the option value is `""` or `null` and gets filtered out.

Make the value a real string key, not empty.

Example option value:
`no_preferences`

Then in your normalization filter, do not remove it.

If you are doing `array_filter()` without a callback, PHP will remove values like `"0"` and empty strings. Use a callback instead:

```php
$this->acceptable_property_conditions = array_values(array_filter(
  $this->acceptable_property_conditions ?? [],
  fn($v) => $v !== null && $v !== ''
));
```

That keeps `no_preferences`.

---

## Step 6, Fix “can only select one”

This is always one of these:

1. missing `multiple` attribute on the `<select>`
2. select2 initialized without multi context because markup is wrong
3. select2 has `maximumSelectionLength: 1`

So confirm the select element has `multiple`, and confirm your init does not set max selection.

---

## Step 7, If it “saves” but doesn’t show on the listing

That’s a separate bug. Your listing display is likely reading from the JSON bridge fields, not the Livewire properties, or it reads different keys than you save.

Once Step 1 debug proves the arrays are correct in Livewire and Step 4 confirms save writes them, then the listing display must be updated to read the same stored field.

This is why the debug block matters. It tells you which layer is lying.

# A) “Other” input must save + display on Drafts and Edits

### Problem pattern

**Safeguard rule:** do not rename existing fields. Add only what’s missing.

You need **two separate fields**:

* `included_assets_selection` (string or array depending on your UI)
* `included_assets_other_text` (string)

### If you store this inside a JSON “details” column

Store it like:

```json

  "selection": "other",
  "other_text": "Convey golf cart + security system subscription"

```

Add column only if missing:

* `included_assets_other_text` (nullable text)

**Safeguard:** keep the existing `Included Property or Business Assets` value exactly as-is; do not overwrite it with the custom string.

# B) Draft Versioning: Saving changes creates a NEW Draft

### What you want (for testing)

* Drafts should be **append-only versions**
* Saving a draft never overwrites the previous one
* You can optionally keep “latest” pointing to the newest draft

---

## B1) Recommended draft structure

Create a `drafts` table if you don’t already have it (or adjust yours):

**drafts**

* `id`
* `user_id`
* `listing_type` (e.g., `hire_seller_agent`)
* `property_type` (income/residential/etc.)
* `status` (active/archived)
* `version` (int)
* `parent_draft_id` (nullable) (for version chain)
* `source_listing_id` (nullable) (if draft came from an existing listing)
* `payload` (json)
* `payload_hash` (string) (for “did anything change?” detection)
* timestamps

**Safeguard:** if you already store drafts inside listings, don’t rip it out—add versioning behavior around it.

---

## B2) “Save Draft” must *clone* instead of update

### Current bug

You are likely doing:

* `Draft::updateOrCreate([...])`

### Change to

When user clicks “Save Draft”:

1. Build the normalized payload
2. Compute a hash of payload (see below)
3. Compare against the last saved draft hash (if exists)
4. If different → create a NEW draft record:

   * `version = last_version + 1`
   * `parent_draft_id = last_draft_id`
   * `payload_hash = new_hash`
5. If identical → do not create a new draft (or still create if you want, but that gets noisy)

**Safeguard:** This prevents infinite “new drafts” when user clicks save without changes.

---

## B3) Payload hashing (change detection)

Create a deterministic JSON string:

* sort keys
* remove UI-only fields
* normalize numeric strings (commas removed, etc.)
* trim strings

Then hash:

* sha256(payload_string)

Store:

* `payload_hash`

---

## B4) Which draft opens when user clicks “Drafts”

Your UI should load:

* the **latest active draft** for that listing context (listing type + property type + maybe listing_id if editing an existing listing)
* OR show a list of versions

For testing, do:

* “Drafts” shows newest first (Version 5, 4, 3…)
* Clicking one opens that version’s payload

---

## B5) Edge cases (Safeguards)

* If user submits a listing from a draft:

  * mark that draft chain as `converted` or keep it; but don’t delete
* If user edits an existing listing and clicks “Save Draft”:

  * create a draft with `source_listing_id = listing.id`
  * version chain still applies

Replit Instructions — Fix “Hire a Seller’s Agent” Listings Not Appearing Under the Correct Tab (Tenant Profile)
Problem

When logged in under a Tenant profile and creating a Hire a Seller’s Agent listing:

the listing is successfully created in the database (or partially created),

but it does not appear anywhere in the UI,

and it does not show under the “Hiring Seller’s Agent” tab.

Expected Behavior

When a user (even under Tenant profile) creates a Seller role listing:

it must be saved with a clear role/type identifier (e.g., role = seller or listing_type = SELLER)

it must appear in the Tenant dashboard → Hiring Seller’s Agent tab

it must behave consistently with the Hire a Tenant listing behavior (filtering and rendering).

SAFEGUARDS (MUST FOLLOW)

DO NOT:

delete or rename any database columns, models, routes, migrations, enums, or tables

change any existing listing IDs, primary keys, or relationships

change UI styling, layout, icons, tooltips, or placeholder text

break existing working flows: Hire a Tenant, Hire a Buyer, Hire a Landlord

change “Agent bid” logic in this task

ONLY ADD / FIX:

missing role/type assignment on save

missing dashboard query/filter logic

missing tab mapping / route mapping

missing component wiring that causes “created but not displayed”

Step 2 — Fix the Listing Creation Save Payload for Seller Flow

In the Hire a Seller’s Agent listing creation submit handler (controller/action/API route), ensure the payload always sets the role/type field.

Required

When user submits Seller listing:

listing_type = "seller" (or "SELLER" depending on your system)

created_by_user_id = authUser.id

created_by_profile_type = "tenant" (optional, only if you store it)

status = "active" when published

status = "draft" when Save Draft

Implementation Notes

If your current Seller creation form does not set listing_type/role, the record may be saved with NULL or default value, causing dashboard filters to skip it.

Add a server-side fallback so even if the frontend forgets, the backend enforces it:

Backend rule (must enforce):

If route = “create seller listing” then force listing_type = SELLER.

Do not rely only on frontend.

Step 3 — Fix Dashboard Query Used by “Hiring Seller’s Agent” Tab

Find the query used to populate listings in the Tenant dashboard.

Common bug

The “Hiring Seller’s Agent” tab may:

point to the wrong dataset

filter incorrectly (e.g., filtering only profile_role = seller instead of listing_type = seller)

not exist / not wired to any data fetch at all

Required Query Logic

For Tenant dashboard → “Hiring Seller’s Agent” tab, fetch listings where:

created_by_user_id = currentUser.id

listing_type = SELLER

exclude drafts if the tab is only for active (or include drafts if you have a Drafts section)

Example filter logic (conceptual):

Active tab: status IN ("active","pending_bids","countered")

Drafts tab: status = "draft"

Key safeguard

Do not change the working Hire a Tenant query—duplicate the pattern and swap the type.

Step 4 — Fix the Tab-to-Filter Mapping (UI Wiring)

In the dashboard tab component where tabs are defined, confirm:

there is a tab entry for Seller

the tab calls the correct filter key

the component is rendering results for that tab

Common wiring issues

Tab label exists but the filter key is wrong (typo like "sellers" vs "seller")

Tab clicks but does not trigger fetch / state update

Tab loads but renders a different list component that expects different fields

Required Behavior

When clicking “Hiring Seller’s Agent”:

it should call the same listing card component used in other tabs

it should pass the same data shape

it should show “No listings yet” only if query returns none

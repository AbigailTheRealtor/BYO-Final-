# Core Rules To Implement

# A. Offered Services Match Rule

For every bid:

* **Matched** = listing requested services also offered by the agent
* **Missing** = listing requested services not offered by the agent
* **Extra** = services agent offered that were not requested
* **Denominator** = total number of normalized listing requested services

### Required invariant

For the same listing:
`matched + missing = denominator`

The denominator must be identical across all bids on that listing.

---

# B. Broker Compensation & Agency Agreement Terms Match Rule

For every bid:

* **Matched** = listing requested terms that exactly match the agent’s offered terms under the platform’s existing match rules
* **Changed / Mismatched** = listing requested terms present on both sides but with different values
* **Missing** = listing requested terms not offered or left blank by the agent when the listing has a value
* **Added** = agent supplied extra terms that were not present on the listing baseline, if your UI currently tracks this
* **Denominator** = total number of normalized listing requested terms that should be matchable

### Required invariant

For the same listing:
The Terms denominator must be identical across all bids on that listing.

Do not let the denominator vary because one agent filled more optional fields.

---

# Important Match Logic Rule

Both Services Match and Terms Match must be built from the **listing baseline** only.

### Never do this:

* count agent services as total possible
* count agent entered terms as total possible
* let optional agent-only fields increase denominator
* let hidden empty fields disappear in one bid and count in another

### Always do this:

* build the listing baseline first
* normalize it
* compare each bid against the same baseline

# Required Implementation

## 1. Use one normalization path for Services comparison

Normalize listing requested services and bid offered services before matching.

Normalization must:

* remove null / empty / false / whitespace only values
* trim strings
* flatten nested arrays
* deduplicate repeated entries
* normalize equivalent labels where needed
* handle “Other” custom values consistently
* ignore display only wrappers

Then calculate:

* matched
* missing
* extra

And set:

* denominator = `count(normalized listing requested services)`

---

## 2. Use one normalization path for Terms comparison

Normalize listing requested terms and bid offered terms before matching.

Normalization must:

* include only terms that are truly matchable for that listing type and branch
* remove null / empty / false / whitespace only values unless platform logic intentionally treats a false value as an explicit selected value
* trim strings
* normalize booleans consistently
* normalize numeric formatting consistently
* normalize percent / dollar / text formatting consistently
* handle custom “Other” values consistently
* exclude presentation only fields
* exclude agent only extra fields from denominator
* preserve property type specific term sets

Then calculate:

* matched
* changed / mismatched
* missing
* added if the UI currently supports it

And set:

* denominator = `count(normalized listing requested terms)`

# Special Term Safeguards

## Additional Terms

Audit how `Additional Terms` is handled.

If the platform already treats textual Additional Terms as part of matchable terms, preserve that, but make it consistent across all views and all roles.

If listing has Additional Terms and agent leaves it blank or changes it, it should not disappear from denominator on one bid and count on another.

---

## Zero vs blank

Be careful not to treat:

* `0`
* `0%`
* `$0`
* `No`
* `False`
  as empty if those are valid explicit selections.

---

## Hidden conditional fields

Only count fields in the denominator if they belong to the listing’s active branch and have a normalized listing baseline value that should be matchable.

Example:
Do not count Commercial only fields on Residential listings.
Do not count Lease Option fields if Lease Option was not selected in the listing baseline.

---

## Summary fields vs atomic fields

If the UI shows summary phrases like:

* “Broker Compensation Summary”
  but the actual comparison is built from multiple atomic term fields underneath, count the atomic matchable fields consistently.

Do not count both the summary wrapper and the atomic child fields.

---

# “Other” Handling

## Services

If “Other” is selected and custom text is present:

* treat as one normalized service item
* same format on listing and bid side
* do not let it count in one place and disappear in another

## Terms

If a term has an “Other” option and custom text:

* treat it as one normalized matchable term value
* use the same canonical format across listing and bid
* do not let one role count raw option plus custom text while another counts only one

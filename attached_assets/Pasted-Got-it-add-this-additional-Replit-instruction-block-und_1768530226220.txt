## 8) Fix Photo Upload Not Persisting After Save Draft (Hire a Buyer’s Agent)

**Issue (new):** Photo upload doesn’t show / didn’t save when you return to the listing from a saved draft.

### ✅ Expected behavior (Definition of Done)

Using `tenant@exp.com`:

1. Upload 1+ photos on a new **Hire a Buyer’s Agent** listing
2. Click **Save Draft**
3. Leave the page → reopen the draft
4. Previously uploaded photos **still display** (thumbnails or filenames) and are still attached to the draft
5. Saving again does **not** duplicate photos

---

## ⚠️ CRITICAL SAFEGUARDS (Photos)

* **Do NOT delete uploaded files** when the listing is saved as a draft.
* **Do NOT require re-upload** when revisiting a draft.
* **Do NOT store only Livewire temp paths** (those expire). You must persist a real stored file path + DB reference.
* **Do NOT overwrite** the entire media array on save; **merge** so previously uploaded photos remain unless user removes them.
* Apply only to **Hire a Buyer’s Agent** flow unless this same bug exists elsewhere.

## Implementation Requirements (works for most Laravel + Livewire setups)

### A) Store photo references in the database (NOT just in Livewire state)

Add/confirm you have one of these patterns:

**Pattern 1 (recommended): separate media table**

* `buyer_listing_photos` table:

  * `id`
  * `buyer_listing_id`
  * `path` (stored file path)
  * `sort_order` (optional)
  * `created_at`

**Pattern 2: JSON column on listing**

* `photos` JSON array on `buyer_agent_auctions` (or the Buyer listing table):

  * `["storage/buyer-listings/123/photo1.jpg", "..."]`

Either pattern is fine — the key is: **Save Draft must persist the stored paths**.

---

### B) On upload: immediately store to disk and create DB record

In the Livewire component handling Buyer photos:

1. Use `WithFileUploads`
2. When a user selects a file:

   * validate
   * store it to a permanent location (`storage/app/public/...`)
   * write the stored path to DB (or append to JSON column)

**Important:** Do not rely on Livewire’s `temporaryUrl()` for persistence.

---

### C) Save Draft must not wipe photos

In `saveDraft()` (Buyer flow):

* If using a media table: **do nothing special** other than ensuring listing ID exists before attachments
* If using JSON: merge existing + new photo paths

  * ensure `array_unique` to prevent duplicates

### D) Load Draft must rehydrate photos into the component state

In `mount()` / `loadDraft()`:

* Pull the stored photo paths from DB
* Set the Livewire property used for rendering thumbnails/list

So when you reopen the draft, the UI is reading from **DB-stored paths**, not temp upload state.

### E) Add a Remove Photo action (prevents “stuck” attachments)

Add a “Remove” button next to each photo that:

* removes the DB record (or removes the path from JSON)
* optionally deletes the physical file **only if you want**, but:

  * if you delete physical files, ensure it only happens on explicit remove (not on draft save)

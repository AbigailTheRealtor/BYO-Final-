But your bug is more likely one of these:

### 1. Wrong Livewire property is being synced

The Select2 field may visually show values, but Replit may be syncing to the wrong property name.

Example:

* UI field uses `property_items`
* listing display reads from something else
* save logic writes to another field
* edit logic loads from another field

So the field can look right in the edit UI but still not persist correctly.

### 2. Save logic is overwriting the value later

Their own note is a red flag:

> `syncAllSelect2BeforeSave()` reads `.val()` right before save and calls `@this.set()`

That means even if they synced correctly once on page load, something later may still be wiping or overwriting it before the actual save completes.

### 3. The listing display is reading the wrong column/property

If the edit form now holds the right value, but the actual listing page still shows blank, then the issue may be:

* save works, but listing display is reading wrong field
* save writes JSON, display expects plain array/string
* property is stored, but rendering condition treats it as empty

### 4. Re-render / duplicate init is still breaking Select2

If Select2 gets re-initialized after that one-time sync, the visible chips may remain but the actual bound value may still be unstable.

### 5. Property Styles and Lease Term may each have different pipelines

Even if both look similar in UI, they may use different:

* model names
* save mappings
* JSON encode/decode paths
* display helpers

So “both display in edit” does **not** prove both are saving correctly.

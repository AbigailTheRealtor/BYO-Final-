Re-audit and standardize all four View Counter Terms pages so they use the same visual shell, same state handling pattern, same diff treatment, and same action layout, while still rendering the correct role-specific and property-type-specific fields.

Target pages:
- /landlord/hire/agent/auction/bid/21/view-counter
- /buyer/hire/agent/auction/bid/18/view-counter
- /hire/seller/agent/auction/bid/13/view-counter
- /tenant/hire/agent/auction/bid/44/view-counter

GOAL
All four pages must look and behave the same at the framework level, while still showing the correct fields for:
- Tenant
- Buyer
- Seller
- Landlord
and their correct property/transaction types.

IMPORTANT
This is a parity and presentation audit, not a negotiation-logic rewrite.

DO NOT
- do not change database schema
- do not rename routes
- do not alter permissions
- do not rewrite counter save logic
- do not change match score formulas
- do not change baseline logic
- do not change who can view the page
- do not remove role-specific fields that are legitimately different

PRESERVE
- existing fallback logic for owner-submitted counter while awaiting response
- existing suppression of misleading dual-score when awaiting response
- existing diff data and score helpers
- existing role/property-specific term groups

AUDIT REQUIREMENTS

1. Use one consistent page shell across all four
Every page should render in this order:
- page title
- listing information
- bid information
- state notice
- match score summary
- Broker Compensation & Agency Agreement Terms
- Requested/Offered Services
- action area if applicable
- back button

2. Standardize titles
Use one consistent title system only:
- "Agent’s Counter Terms"
- "Your Submitted Counter Offer"
Do not leave old variants like:
- "Counter Terms For Agent's Bid"

3. Standardize state notices
All four pages must use the same notice style and placement for:
- no counter exists
- your counter was submitted and is awaiting response
- a counter was received
The wording and placement should be consistent.

4. Standardize Match Score block
All four pages must use the same visual score card shell.
If score data exists, render:
- overall score
- services score
- terms score
- compared-to label
If score data truly does not exist, use the same empty-state shell in the same location.

5. Standardize diff treatment
For both Terms and Services, all four roles must use the same visual treatment for:
- Changed
- Added
- Missing / Removed
Do not allow one role to show rich badges while another shows only raw text.

6. Standardize section styling
Unify:
- panel headings
- spacing
- dividers
- border radius
- typography
- badge style
- padding
- last updated placement

7. Keep role/property differences only in content
Examples:
- Buyer may have buyer compensation fields
- Seller may have seller compensation fields
- Landlord/Tenant may have lease-specific fields
That is fine.
What should NOT differ is the overall structure or visual behavior.

8. Use the strongest implementation as the benchmark
Seller currently appears to be the richest version.
Use the best existing implementation as the visual benchmark and align Tenant, Buyer, and Landlord to it.

9. Verify correct behavior per state
For each role, verify these states:
- true no-counter state
- owner-submitted counter awaiting response
- actual returned counter exists
Confirm the correct notice, score behavior, and action/button visibility in each state.

QA REQUIRED
For each of the 4 target routes:
- confirm actual counter payload loaded correctly
- confirm title is correct
- confirm same score block shell
- confirm same diff badge treatment
- confirm same section order
- confirm same awaiting-response treatment when applicable
- confirm no false blank-state
- confirm back button still works

PASS CONDITIONS
A pass is complete only if:
1. all four pages share the same visual shell
2. all four pages use the same title system
3. all four pages use the same notice placement and style
4. all four pages use the same diff treatment for changed/added/missing
5. all four pages preserve the correct role/property-specific fields
6. no schema, route, or negotiation logic changes were introduced

If any page is still using a lighter legacy blade or partial, replace or align it with the richer standardized implementation instead of patching the old presentation in place.

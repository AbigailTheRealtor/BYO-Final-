Replit Instructions — Combine “Accepted Bid Summary” + “E-Sign Acknowledgement” Into One Page (Hire a Tenant’s Agent)
Applies to

BidYourAgent → Hire a Tenant’s Agent

When a Tenant clicks “Accept” on an Agent’s bid

Accepted Bid Summary + E-Sign must be on the SAME page

SAFEGUARDS (MUST FOLLOW — DO NOT BREAK OTHER FLOWS)

DO NOT:

Delete or rename any database tables/columns, migrations, or models.

Change existing route names/URLs that other pages depend on (you may add routes).

Remove or rewrite the PDF generation system (reuse/extend what exists).

Change UI layout, styling, icons, colors, font families, or spacing outside of the specific pages/components listed below.

Modify Buyer/Seller/Landlord flows.

Remove the existing standalone E-Sign page (if it exists). Leave it in place but stop routing users to it for this flow.

Change field names/keys used for signatures, bid status, accepted status, or summary storage.

ALLOWED:

Add a new combined view and/or reuse the summary view and embed the e-sign panel at the bottom.

Add backend logic for: signature capture, status updates, PDF creation, and admin copy.

Add a “Download PDF” button that is only enabled when BOTH parties have signed.

GOAL / EXPECTED BEHAVIOR
1) On Tenant “Accept”

Immediately redirect Tenant to: Accepted Bid Summary (combined page).

The page shows:

Summary content

E-Sign section at bottom (Tenant signature box)

Agent signature status box (Pending/Signed)

2) E-Sign on the same page

No separate “Tenant E-Sign Acknowledgement” page.

Tenant signs on the summary page:

Checkbox + signature input (typed name)

No visible fields for IP/time/timezone — those are captured automatically.

3) Metadata capture (automatic)

IP address captured server-side at signing time.

Date/time captured server-side at signing time.

Timezone captured from user device at signing time (browser/device setting), stored in DB.

Do not display IP/time/timezone fields in UI.

4) After both parties sign

Summary status updates to fully executed (Acknowledgement complete).

Generate a PDF copy and:

Tenant can download PDF

Agent can download PDF

Admin can access/download a copy under Admin account/dashboard

5) One combined checkbox statement

Replace the old “notice + separate checkbox text” with ONE checkbox label:

✅ Checkbox text (exact):
“Important: By signing below, you acknowledge receipt and review of this Accepted Bid Summary. This is an acknowledgement only, not a contract execution. I confirm that I have reviewed the Accepted Bid Summary and acknowledge its contents.”

IMPLEMENTATION STEPS
Step A — Identify the current files to edit (search terms)

In Replit, use global search for these phrases and note the files:

“Accepted Bid Summary”

“Tenant: E-Sign Acknowledgement”

“Agent: E-Sign Acknowledgement”

“accept bid” / “Accept” button handler

“Pending Acknowledgement”

PDF generation keyword: dompdf, snappy, wkhtmltopdf, generatePdf, pdfPath, storage/pdf

You will likely find:

A bid accept controller method (ex: acceptBid()).

A summary controller/view (ex: accepted_bid_summary.blade.php).

A separate e-sign view (ex: tenant_esign.blade.php, agent_esign.blade.php).

Step B — Change Tenant “Accept” redirect to go to the combined summary page

Where: the backend method that processes Tenant clicking Accept (controller handling accept).

Requirement:

When accept occurs:

Create (or fetch) the Accepted Bid Summary record for that bid.

Redirect to the Accepted Bid Summary page (combined page), not a separate e-sign page.

Pseudo logic (do not rename existing variables/tables):

acceptedSummary = AcceptedBidSummary::firstOrCreate([bid_id => $bid->id], [...])

return redirect()->route('accepted_bid_summary.show', $acceptedSummary->id)

Important: Do not remove existing routes. Only update the redirect target for this Hire-a-Tenant accept action.

Step C — Embed E-Sign section into the Accepted Bid Summary page (ONE page)

Where: the Accepted Bid Summary view that shows the summary content.

C1) Keep the summary exactly as-is (layout/styling unchanged)

Do not change summary formatting.

Do not change fonts/colors.

C2) Add an “E-Sign Acknowledgement” section at the bottom of the summary page

Add below the summary content:

1) Document Status module stays

Show:

Tenant: Pending/Signed

Agent: Pending/Signed

2) Tenant signing UI

Show Tenant signing box ONLY if Tenant has not signed yet.

Inputs:

Checkbox with the combined text (exact text above)

“Type Your Full Legal Name” input

“Sign” button

3) Agent signing UI

If logged-in user is Agent and Agent not signed:

show the same signing UI

If logged-in user is Tenant:

show Agent status only (Pending/Signed)

4) Do NOT show

IP

Timestamp

Timezone fields

Step D — Signature capture endpoint (Tenant + Agent use the same signature handler)

Create (or reuse) a POST endpoint for signing the summary:

D1) Add/Reuse route (do not break existing)

Example:

POST /accepted-bid-summary/{id}/sign

name: accepted_bid_summary.sign

D2) Backend sign handler must:

Validate:

checkbox checked

name typed (non-empty)

Determine signer role from authenticated user:

if Tenant user → mark tenant_signed_at, tenant_signed_name

if Agent user → mark agent_signed_at, agent_signed_name

Capture metadata automatically:

ip_address from request on server side

signed_at from server time

timezone passed from browser/device (see Step E)

Update status:

If only one party signed → status remains Pending (with correct side signed)

If both signed → mark summary “Fully Signed / Complete”

Do not change DB column names.
If separate columns already exist (tenant_ip, agent_ip, etc.) use them.

Step E — Capture device timezone automatically (browser/device)
E1) On the summary page, add a hidden input:

timezone

value set by JS:

Use:

Intl.DateTimeFormat().resolvedOptions().timeZone

Also include:

timezone offset minutes (optional, helpful):

new Date().getTimezoneOffset()

E2) On submit, include timezone in POST

Backend stores it in the signer record/fields (tenant_timezone/agent_timezone or generic field if that’s what exists).

Important: This timezone must reflect the user’s device, not the server timezone.

Step F — PDF generation after both signatures are complete
F1) Trigger point

When the second party signs and the summary becomes “Complete”:

Generate PDF immediately (same request) OR queue it if your system already queues jobs.

Store PDF file path in the existing field (do not create new schema unless absolutely necessary).

F2) PDF content

Use the existing Accepted Bid Summary HTML layout (the one currently displayed).

Include signature acknowledgements section at the bottom of the PDF:

Tenant signed name + signed date/time (server time is okay)

Agent signed name + signed date/time

IP address may appear in PDF only if it already exists in the current template; otherwise optional.

If your current PDF template already prints “IP Address / Date/Time”, keep it. If not, don’t add new visible UI fields.

Step G — Download PDF button behavior
G1) On summary page:

Show a “Download PDF” button.

Button rules:

Disabled or hidden until tenant_signed_at AND agent_signed_at are present AND pdf_path exists.

Once ready, clicking downloads the stored PDF.

G2) Access rules:

Tenant can download if they are the Tenant on that listing.

Agent can download if they are the accepted Agent on that listing.

Admin can download always (admin panel link, Step H).

Step H — Admin copy required (Admin gets a copy in their account)
H1) Store the PDF in a central location

Example: storage/app/accepted-bid-summaries/{summary_id}.pdf

Do not change storage configuration.

H2) Admin dashboard listing

Add a simple admin-only list entry:

Listing ID

Tenant name/email

Agent name/email

Signed status

“Download PDF” link

Important safeguard: Do not change any existing admin dashboard layout. Add this as a new section/page or new row item only.

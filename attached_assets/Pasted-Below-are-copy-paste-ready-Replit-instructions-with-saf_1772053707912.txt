# 3) Fix: Edit Listing “Acceptable Property Conditions” flashes all options + causes tab to scroll/jump

This is almost always Livewire re-render / Alpine init timing.

### Required fix (high impact, low risk)

1. Put the options list behind `x-cloak` so it can’t “flash” before Alpine finishes.
2. Use **wire:model.defer** (not live) on the multi-select so changing state doesn’t trigger rapid rerenders.
3. Make sure the container has a stable `wire:key` so Livewire doesn’t rebuild it.

#### In the edit blade for that tab section:

```html
<div wire:key="acceptable-property-conditions-section">
  <div x-data x-cloak>
    <!-- Use wire:model.defer to prevent constant re-renders -->
    <div class="your-options-wrapper">
      <!-- Example checkbox list -->
      @foreach($acceptablePropertyConditionsOptions as $opt)
        <label class="block">
          <input
            type="checkbox"
            wire:model.defer="acceptable_property_conditions"
            value="{{ $opt }}"
          >
          <span>{{ $opt }}</span>
        </label>
      @endforeach
    </div>
  </div>
</div>

<style>[x-cloak]{ display:none !important; }</style>
```

### If you’re using Alpine show/hide logic

Make sure you use `x-cloak` on the block that toggles, and avoid initial `x-show="true"` defaults that then flip.

### Optional extra stabilization

If that list is being rebuilt due to computed props, stop recomputing it every render. In the Livewire component:

* build the options once in `mount()` and keep it in a public property like `$acceptablePropertyConditionsOptions`.

# 5) Fix: Next button on edits requires multiple clicks

This is typically caused by one of these:

* button is inside a `<form>` but missing `type="button"` so it submits + rerenders
* button has both `wire:click` and JS navigation fighting each other
* an overlay/loading state captures the first click
* scroll jump is re-focusing something and cancelling click

### Safe standard fix (recommended)

Make the Next button:

* `type="button"`
* `wire:click.prevent="goNextStep"`
* disable during loading
* add a small loading indicator if you want

```html
<button
  type="button"
  wire:click.prevent="goNextStep"
  wire:loading.attr="disabled"
  wire:target="goNextStep"
  class="btn btn-primary"
>
  Next
</button>

<span wire:loading wire:target="goNextStep">Loading…</span>
```

### In Livewire component

Make sure `goNextStep()`:

* increments step once
* does not reset tab state
* does not dispatch multiple events that re-render twice

```php
public function goNextStep(): void
{
    // validate only what’s needed for this step if you do step-validation
    // $this->validate([...]);

    $this->step = min($this->step + 1, $this->maxStep);
}
```

### Also do this if you have “scroll to top” logic

If you’re calling JS to scroll to top on step change, remove it OR only run it once. That’s often what makes the UI feel like it “ignored” the first click.

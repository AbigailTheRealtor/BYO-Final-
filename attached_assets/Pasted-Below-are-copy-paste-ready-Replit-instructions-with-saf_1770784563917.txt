# DIAGNOSIS (Why this is happening)

Because Commercial edit shows the fields, the base edit screen works. Residential missing them indicates one of these code-level issues:

1. Residential edit uses a different Blade branch/partial that **doesn’t include** the location section
2. Residential edit includes it, but has a conditional like `@if($isCommercial)` around the location block
3. Residential edit binds inputs to different Livewire properties, so they render empty or not at all
4. Residential edit loads a different component/view than commercial (route conditional)

Your fix must address all 4 possibilities by enforcing shared rendering + shared keys.

## FIX 2 — Ensure Residential Edit is hydrated with the same keys Commercial uses

Even if the fields render, they can appear blank if Residential uses different property names.

✅ Requirement
In the Landlord Edit component `mount()` / `loadDraft()`:

* set the same four Livewire properties regardless of property type:

* `$this->property_city`

* `$this->property_state`

* `$this->property_county`

* `$this->property_zip`

✅ **Safeguard**
Do not do this:

* `if (commercial) load property_city...`
* `else load city...`

Residential must load the same canonical property_* fields.

## FIX 3 — Ensure Residential Edit inputs bind to the canonical Livewire properties

In the shared partial, inputs must bind to:

* `wire:model="property_city"`
* `wire:model="property_state"`
* `wire:model="property_county"`
* `wire:model="property_zip"`

🚫 Do not use a different wire:model in residential.

✅ **Safeguard**
Do not set a manual `value="..."` attribute (avoid printing false/empty mismatches). Livewire binding only.

## FIX 4 — Ensure Save/Update persists those same keys for Residential

If commercial persists but residential doesn’t, residential update might be saving different keys or not saving them at all.

✅ Requirement
In the edit save method:

* always save `property_city/state/county/zip` for both property types

✅ **Safeguard**

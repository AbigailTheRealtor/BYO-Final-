# 1) SAFEGUARD — Confirm you’re saving to the same dataset the search uses

### Problem pattern

The “submit” action may be saving into one table/model (or one listing type), while the search bar reads from another table/model/query.

### Required dev action

1. Identify the exact query used by the **Hire a Seller search bar**.
2. Identify the exact model/table written by **Seller submit** and **Seller draft save**.
3. Confirm they match.

✅ **Safeguard:** Add a 1-time debug log on submit:

* listing ID
* table/model name (or class)
* listing type
* status
* published flag
* visibility flag
* user_id/tenant_id
  Then compare to what the search query expects.

---

# 2) FIX — On Submit, ensure listing is marked “Published / Active / Searchable”

### Common cause

Listings save successfully but remain:

* `status = draft`
* `is_published = 0`
* `is_active = 0`
* `is_visible = 0`
* missing `submitted_at` / `published_at`
* missing required listing_type/category value used by search filters

### Required changes (Submit action)

When user clicks **Submit Listing** for Hire a Seller, set all required “searchable” flags explicitly:

**Required fields to set on submit (use your actual column names):**

* `status = 'active'` (or `'submitted'` if your search includes it)
* `is_draft = 0`
* `is_published = 1`
* `is_active = 1`
* `published_at = now()` (or `submitted_at = now()`)
* `listing_type = 'seller_agent'` (or whatever your search filter expects)

✅ **Safeguard:** Do not rely on defaults. Set them explicitly in the submit method.

---

# 3) FIX — Search query must include the statuses you’re using

### Common cause

Search query filters only include one status, like:

* `where status = 'active'`
  …but your submitted listing is set to `'submitted'`, or `'pending'`.

### Required dev action

Update the search query to include every status that should show publicly in that search.

Examples (conceptual):

* include: `active`, `submitted`, `pending_review`
* exclude: `draft`, `archived`, `deleted`

✅ **Safeguard:** Make the status filter a constant array used across the app:

* `SEARCHABLE_SELLER_STATUSES = ['active','submitted','pending_review']`

---

# 4) FIX — Remove tenant-only scoping if the search is meant to be sitewide

### Common cause

Search is unintentionally scoped like:

* `where user_id = auth()->id()`
  or
* `where tenant_id = currentTenant`

So you “submitted” but it won’t show in the global search bar.

### Required dev action

Decide the intended behavior:

## A) If search bar is PUBLIC/sitewide:

* remove user-specific scoping
* include all published/active listings

## B) If search bar is “My Listings” only:

* keep user scoping (but then you must have a separate global search)

✅ **Safeguard:** Add a label in UI:

* “Search My Listings” vs “Search All Listings”
  So behavior matches expectation.

---

# 5) FIX — Drafts not showing: drafts list must query `status = draft` AND correct listing type

### Common cause

Drafts are saved correctly but drafts list/search is missing filters:

* not filtering `listing_type = seller_agent`
* or filtering by wrong `role` or wrong `auction_type`

### Required dev action

Drafts list query must include:

* `listing_type = 'seller_agent'` (or your equivalent)
* `status = 'draft'` (or `is_draft = 1`)
* correct owner: `user_id = auth()->id()`

✅ **Safeguard:** Draft save must always set:

* `status = 'draft'`
* `is_draft = 1`
* `is_published = 0`

---

# 6) FIX — Search indexing (if you use a separate search index / cached list)

### Common cause

The “search bar” is not querying DB directly; it’s hitting:

* cached results
* a search index table
* a JSON file
* an in-memory list

So new listings won’t appear until a refresh job runs.

### Required dev action

1. Identify whether search uses:

* direct DB query, OR
* cached/indexed dataset

2. If indexed:

* on submit, **trigger re-index** for that listing (sync)
* or enqueue re-index job and ensure queue is running

✅ **Safeguard:** Until indexing is stable, fallback:

* If listing is published but not found in index, the search endpoint should query DB directly for that listing ID.

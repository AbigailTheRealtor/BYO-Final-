ISSUE:
The Buyer Offer Listing Edit flow has a blocking bug.

In `BuyerOfferListingEdit.php`, `saveDraft()` always creates a new `HireBuyerAgentAuction` row, assigns `$this->auctionId` to that new row, and then saves metadata to the new row.

That behavior is correct for CREATE mode, but wrong for EDIT mode.

When a user opens:
`/offer-listing/buyer/edit/{auctionId}`

Then clicks “Save Draft”:

1. A new draft row is created.
2. `$this->auctionId` is overwritten with the new draft ID.
3. The original listing ID is lost from component state.
4. If the user then clicks “Update Listing,” the submit flow updates the new draft row instead of the original listing.
5. The original listing is not updated.

REQUIRED FIX:
In `BuyerOfferListingEdit.php` only:

- Update `saveDraft()` so EDIT mode does NOT create a new auction row.
- In edit mode, `saveDraft()` must save against the current existing `$this->auctionId`.
- Preserve the original listing identity during the entire edit session.
- Do not overwrite `$this->auctionId` with a newly-created draft ID.
- Do not delete old drafts.
- Do not change the CREATE component’s draft versioning behavior.

IMPLEMENTATION GUIDANCE:
- Add a guard inside `BuyerOfferListingEdit.php::saveDraft()`.
- If `$this->auctionId` exists, find that existing `HireBuyerAgentAuction`.
- Update that existing row with the current form fields.
- Keep/save metadata using `saveAllMetadata($auction)` against the existing auction.
- Do NOT call `new HireBuyerAgentAuction()` inside edit-mode saveDraft unless there is no valid existing auction.
- Do NOT set `$this->auctionId` to any newly-created draft ID in edit mode.
- If needed, preserve a separate variable like `$originalAuctionId` before saving and confirm it remains unchanged.

PART A — FIX “LISTING DATE” (Editable + Draft-safe)
Problem

Listing Date is auto-set (good), but when saving a draft and returning the next day, the listing may not appear in “Actual Listings” because it’s being filtered/excluded based on an older Listing Date.

Expected Behavior

Listing Date defaults to today when a listing is first created.

When user saves draft and returns later, Listing Date:

remains whatever it was last saved as

is editable via a date picker

Actual Listings page must NOT exclude listings solely because Listing Date is not today.

Expiration logic (if tied to Listing Date) must not “auto-expire” drafts incorrectly.

A1) UI Change (All 4 listing types)

In Tab 1: Listing Details, for each flow (Tenant/Buyer/Seller/Landlord):

Ensure Listing Date field is a date input / date picker.

It should:

auto-fill with today’s date on first create

on edit/resume draft, it should load the saved value

allow user to change it manually

Important: Do not remove any existing fields. Just ensure Listing Date is editable.

A2) Backend Save Logic

Wherever “Save Draft” and “Submit/Publish” happen:

REQUIRED:

Do not override Listing Date on every save.

Only set default Listing Date if it is currently null/empty.

If a Listing Date already exists, keep it as-is unless user changes it.

Pseudo-behavior:

if listing_date is empty → set to today

else → keep existing listing_date unless user submitted a new value

A3) Actual Listings Query Fix (This is what’s hiding drafts)

Find the query used to display “Actual Listings” / “My Listings” under the top navigation tabs.

REQUIRED:

Remove any filter like:

listing_date = today

listing_date >= today

whereDate(listing_date, today)

or any logic that excludes older listing_date rows by default

Replace with:

Listings should be shown based on ownership + status, NOT the listing_date being “today.”

Example logic (conceptually):

show listings where:

created_by / user_id matches current user (or correct account context)

status in: Active, Pending, Hired Agent, Expired (whatever your system uses)

optionally not deleted/archived

If you need sorting:

sort by updated_at DESC (best for drafts) or created_at DESC

Listing Date should be display-only metadata, not the gatekeeper for visibility

A4) If Expiration Is Calculated From Listing Date

If your app calculates expiration as listing_date + X days:

REQUIRED:

Drafts should not become hidden/expired simply because listing_date is old.

If you have a status = Draft, Drafts must always appear in “Drafts” or “My Listings” regardless of listing_date.

If you don’t have a draft status:

Use updated_at for draft visibility.

Do not auto-expire unless the listing is actually “Active” (published).

(Do not change status names. Just ensure expiration/visibility doesn’t depend on listing_date for drafts.)

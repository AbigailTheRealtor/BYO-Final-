Do not touch listing creation, accepted-hire flows, dashboard UI, admin UI, or model relationships yet.

- Use the existing session keys from Phase 4 exactly:
  - referral.agent_id
  - referral.code
  - referral.captured_at
  - referral.visit_id
- If this codebase has multiple registration paths, inspect first and patch the real successful-account-creation path(s) only.

PHASE 5 — SIGNUP PERSISTENCE ONLY

2. After a brand-new user is successfully created:
- Check whether session contains referral.agent_id
- If no session referral exists, do nothing
- If session referral exists:
  - only proceed if the new user’s referred_by_agent_id is currently null/empty
  - update the user record with:
    - referred_by_agent_id = session('referral.agent_id')
    - referral_source_code = session('referral.code')
    - referral_captured_at = session('referral.captured_at')

3. Update referral_visits if session('referral.visit_id') exists:
- Find that visit row
- Set:
  - visitor_user_id = newly created user id
  - converted_to_signup = true

4. Increment signup_count on agent_referral_links only when:
- referral.agent_id exists in session
- and a brand-new referral was actually attached to the user
- and the visit/user linkage succeeded or is clearly valid
Use atomic increment.

5. Do NOT clear the referral session yet.
Keep the session intact for later listing attribution in Phase 6.

6. Safety rules:
- If user already has referred_by_agent_id, do not overwrite it
- If referral.visit_id is missing, still attach the referral to the user if session agent/code exists
- If the visit row is missing, fail gracefully and do not break signup
- Never crash the registration flow because referral data is missing or stale
- No UI changes in this phase

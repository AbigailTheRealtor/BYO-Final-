# STEP 3 — Add a Tenant-only “safe label helper” (prevents recurrence)

If those labels are assembled dynamically or pulled from a config/array that may contain escaped quotes, add a Tenant-only sanitize layer.

## Option A (Best): a small helper function (Tenant-only usage)

Create:
`app/Support/TextSanitizer.php`

```php
<?php

namespace App\Support;

class TextSanitizer
{
    public static function unescapeApostrophes(string $text): string
    {
        // Convert escaped apostrophes into real apostrophes
        return str_replace("\\'", "'", $text);
    }
}
```

Then in Tenant templates where these labels print:

```php
use App\Support\TextSanitizer;

{{ TextSanitizer::unescapeApostrophes($label) }}
```

✅ Safeguard: apply only to the labels you know are affected (not global).

# STEP 4 — Ensure the issue is NOT caused by double-escaping output

Sometimes the value is already safe, but someone is printing using the wrong echo style.

### If you see:

* `{{ e($label) }}` or `{{ htmlentities($label) }}` combined with already-escaped strings, you can end up with visible backslashes.

✅ In Tenant-only templates, ensure you’re using normal Blade escaping:

```php
{{ $label }}
```

and that `$label` is not manually escaped as `Tenant\'s`.

🚫 Do not switch to `{!! !!}` unless absolutely necessary (can create XSS risk). This is a label, so keep `{{ }}`.

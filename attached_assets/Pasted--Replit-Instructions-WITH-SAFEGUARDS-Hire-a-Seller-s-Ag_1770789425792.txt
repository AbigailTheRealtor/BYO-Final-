# REQUIRED IMPLEMENTATION

## Step 1 — Confirm canonical storage key

Use ONE canonical DB key (column or JSON metadata key), e.g.:

* `personal_photo_path`

✅ Must be used everywhere:

* draft save
* draft load/hydration
* edit save
* edit load/hydration

🚫 No different key on create vs edit (ex: `personalPhoto` vs `personal_photo`).

---

## Step 2 — Draft Save: store the file and save the path

In Seller Create/Draft component save method:

### Requirements

1. When a user uploads a photo and clicks Save Draft:

   * move the upload to permanent storage
   * write the saved path to the draft record (`personal_photo_path`)

2. If no new upload happened during this save:

   * DO NOT overwrite the existing saved path

✅ **Safeguard: only update if present**

* only set `personal_photo_path` if a new file was uploaded
* otherwise keep current value

---

## Step 3 — Draft Reopen: auto-populate the photo preview

In `loadDraft()` / `mount($draftId)` for the draft/edit screen:

### Requirements

* `$this->personal_photo_path = $draft->personal_photo_path ?? null;`
* Ensure it’s never set to `false`

### Blade/UI requirement

If `personal_photo_path` exists:

* show image preview (same look as create)
* show buttons:

  * **Remove**
  * **Replace**

---

## Step 4 — Remove + Replace behaviors (must work in drafts)

### Remove Photo

When user clicks Remove:

1. clear `personal_photo_path` in Livewire state
2. persist null to DB on save (or immediately)
3. delete file from storage (recommended), or flag for cleanup

✅ **Safeguard:** removing must be explicit user action.

### Replace Photo

When user uploads a new file:

1. store new file
2. update DB path
3. delete old file (recommended)

---

## Step 5 — Ensure preview works in both Draft and Edit

Photo preview must show in:

* Draft reopen screen
* Edit listing screen (after listing is published/created)

✅ Same rendering logic, same partial if possible.

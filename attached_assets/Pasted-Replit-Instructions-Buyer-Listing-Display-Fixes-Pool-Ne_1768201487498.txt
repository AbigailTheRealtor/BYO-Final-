Problem

…but Buyer is not consistently showing the selected sub-options inline and/or format differs.

If Pool Needed = Yes:

•	If Pool Needed = No:

(no parentheses)

•	Treat Pool sub-options as an array and render:
Yes (Option1, Option2, ...)

2) Remove “Other” label from multi-select display values (show answer only)
Fields
•	View Preference Needed
•	Non-Negotiable Amenities

Problem
When a user selects Other and enters custom text, the listing display shows the word “Other” as a visible item in the answers.
Expected Behavior
On the listing display:
•	Do NOT show the literal word “Other” as a selectable chip/value.
•	Instead:
o	show the user’s custom “Other” text (if provided)
o	plus any other selected options
Example
If selected: Water View, Other + custom text: “Sunset-facing balcony”
✅ Display should show:
View Preference Needed: Water View, Sunset-facing balcony
❌ Not: Water View, Other, Sunset-facing balcony
Implementation Notes (safe + consistent)
For each affected field:
1.	Start with the selected options array.
2.	Filter out the literal string "Other" from the displayed list.
3.	If a custom Other text field exists and is not empty, append that text to the display list.

3) Fix currency formatting: “$” must appear BEFORE the amount
Problem
In Buyer Purchasing Terms, some currency displays show the dollar sign after the number:
•	Assignment Fee to Broker: 5,000$
Expected Behavior
All currency displays must be formatted like:
•	Assignment Fee to Broker: $5,000.00
(or if you don’t show cents in this view: $5,000 — but be consistent with the rest of the platform)
Implementation Notes
•	Do not concatenate strings like amount + "$".
•	Use a shared helper/formatter for money fields:
o	$ prefix
o	thousands separators
o	optional 2 decimals (match Tenant/Seller conventions)
Apply this fix to:
•	Assignment Fee to Broker (and any other Buyer fields that are currency-based)

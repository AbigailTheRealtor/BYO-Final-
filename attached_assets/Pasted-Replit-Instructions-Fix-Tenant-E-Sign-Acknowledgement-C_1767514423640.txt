1) Fix Timezone — Must Use Signer Device Timezone (Not Server Timezone)
Problem

The “Timestamp” shown is wrong (server time / ET), and the signed time shown later doesn’t match the user’s actual local time.

Expected Behavior

Capture the signer’s timezone from their browser/device (IANA timezone, e.g., America/New_York)

Store the signature timestamp in UTC in the database (recommended best practice)

Display it back using the signer’s captured timezone (or the viewer’s local timezone if that’s the chosen UX)

Implementation Steps
A) Frontend — capture timezone + client timestamp

On the Tenant: E-Sign Acknowledgement form submit, include:

signer_timezone (IANA string)

client_signed_at (ISO timestamp)

Add hidden fields to the form:

signer_timezone

client_signed_at

Populate them via JS:

Intl.DateTimeFormat().resolvedOptions().timeZone

new Date().toISOString()

Also update the “Your Timezone” display field to show that captured value.

Replit: find the Tenant e-sign acknowledgement view and locate the form that submits the signature. Add hidden inputs and a small script that sets them on page load.

B) Backend — store in UTC + store signer timezone

In the controller action that processes Tenant’s acknowledgement:

Save:

tenant_signed_name (already being captured)

tenant_signed_at_utc = now()->utc() (or use the server time as canonical)

tenant_signer_timezone = request('signer_timezone')

Optionally store tenant_client_signed_at if you already store “client time” (nice to have, but not required)

Important: Always store the canonical signed time in UTC, then convert for display using the stored timezone.

C) Display — show correct signed date/time

On the Accepted Bid Summary page (Document Status + Signature Acknowledgement blocks):

If tenant_signed_at_utc exists:

Display tenant_signed_at_utc converted to tenant_signer_timezone

If timezone missing, fall back to app default timezone

2) Fix IP Address — Must Capture Real Client IP (Not 10.x.x.x)
Problem

You’re capturing a private/internal IP (10.82.2.159) which usually means the app is behind a proxy/load balancer and Laravel isn’t trusting forwarded headers.

Expected Behavior

Capture the real external client IP from forwarded headers (when behind a proxy)

Store it at signing

Display it everywhere signature info appears

Implementation Steps
A) Backend — ensure the framework trusts proxy headers

If this is Laravel, update TrustProxies so X-Forwarded-For is honored.

Locate: app/Http/Middleware/TrustProxies.php

Ensure it is configured to trust the proxy and forwarded headers, commonly:

$proxies = '*'; (or the specific proxy IPs if you have them)

$headers = Request::HEADER_X_FORWARDED_ALL;

Replit: do NOT guess randomly—open the TrustProxies file and apply the standard Laravel proxy trust configuration appropriate for the hosting environment.

B) Signing controller — store IP at signing

In the “Tenant signed acknowledgement” controller action:

$ip = request()->ip();

Save to the accepted bid summary record:

tenant_signed_ip = $ip

(If you already store IP somewhere else, use that existing field—do not create duplicates unless necessary.)

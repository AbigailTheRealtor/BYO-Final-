Task: Fix Acknowledgement Summary document-upload access logic using safe ownership + listing type split

We need to correct the access logic for the new Optional Document Sharing (Recommended) section on the Acknowledgement Summary page.

Important

Do not use a role string alone (example: tenant) to decide whether the upload section should be shown.

That approach is too fragile and can break Buyer, Seller, and Landlord flows.

At the same time, do not do a broad rewrite that risks breaking the overall accepted bid / acknowledgement flow.

This must be a surgical fix.

Goal

Use a two-part rule:

1. Ownership / permission check

Controls whether the user is allowed to see and submit the document-sharing section

2. listing_type

Controls which fields are shown inside that section

Correct logic model
A. Access control = ownership / correct party

Only the correct listing-side user tied to that accepted bid summary should be allowed to:

see the upload form
submit documents
edit/replace documents

Do not gate this with a hardcoded role string like:

$userRole === 'tenant'

That is not permission logic.

B. Field selection = listing type

Continue using listing_type to determine which document fields to show:

buyer → Government-Issued ID + Proof of Funds + Pre-Approval Letter
tenant → Government-Issued ID + Proof of Income
seller → Government-Issued ID + Property Record Link
landlord → Government-Issued ID + Property Record Link

That part is correct and should remain role-based.

Required implementation approach
Step 1 — Identify the correct ownership field on the accepted bid summary

Determine which field on the summary or related record represents the user who created the listing / is the listing-side party in the acknowledgement flow.

Possible examples:

user_id
listing_user_id
owner_id
related listing record owner id

Do not assume the correct field without checking the actual summary creation flow for:

Buyer
Tenant
Seller
Landlord
Step 2 — Create one explicit boolean for upload permission

In the controller or view-model passed to the sign page, create a variable like:

$canUploadAcknowledgementDocuments

This should evaluate to true only if the authenticated user is the correct listing-side party for that summary.

Preferred pattern

Compute this in the controller, not inline in Blade if possible.

Then pass it into the view.

Step 3 — Use that boolean in the Blade

Replace any role-based visibility checks like:

$userRole === 'tenant'

with:

@if($canUploadAcknowledgementDocuments)

This should control whether the document-sharing section is shown at all.

Step 4 — Keep listing_type for field rendering only

Inside the section, continue using listing_type to decide which fields to display.

Example:

buyer fields
tenant fields
seller/landlord fields

Do not mix access control and field selection into one condition.

Safeguards
Safeguard 1 — Do not break existing acknowledgement flow

Do not change:

sign flow
signature logic
accepted bid summary creation logic
existing role labels
existing listing_type values

This task is only about fixing document-upload access visibility and submission permission.

Safeguard 2 — Do not rely on role string for access control

Do not use:

$userRole === 'tenant'
$userRole === 'buyer'
any direct role string alone

for deciding whether the user can upload documents.

Role strings should only be used to determine which document fields appear.

Safeguard 3 — Keep authorization mirrored server-side

Even after the Blade is fixed, the storeDocuments() controller method must continue to enforce the same permission rule server-side.

That means:

the same user who can see the form is the same user who can post to it
a direct POST from another user must still be rejected

Do not rely on frontend visibility alone.

Safeguard 4 — Do not rewrite unrelated files

Do not refactor unrelated accepted bid summary services unless absolutely required.

If possible:

only inspect them to confirm the correct ownership field
do not rewrite them broadly
Safeguard 5 — Keep listing_type values unchanged

Do not rename:

buyer
tenant
seller
landlord

Keep those values exactly as-is if they are already being written correctly.

Additional wording fix

Inside the Seller / Landlord document-sharing fields, change the field label from anything like:

Property MLS/Record Link

to:

Property Record Link

Use helper text like:

Paste a public county property appraiser or property record link related to this property.

Do not reference MLS for user-facing client uploads.

Private storage safeguard

Confirm that uploaded files remain:

private
not publicly accessible by direct URL
available only to:
the uploading user
the selected/accepted agent if that access is intentionally allowed later
any authorized server-side viewer

Do not move files to public storage.

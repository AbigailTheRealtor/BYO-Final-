# NON-NEGOTIABLE SAFEGUARDS (Must Implement)

### SAFEGUARD A — No Partial Tab Saves / No Data Loss

1. **Draft save must persist the entire listing payload**, not only the active tab.
2. Saving a later tab **must not overwrite earlier tabs** with null/empty defaults.
3. Implement a **merge-save** strategy:

   * When saving, merge the current Livewire/form state into the existing draft record (or listing record), field-by-field.
   * Never replace the entire JSON/object with only the current tab’s subset unless you are sure all fields are included.

**Acceptance test:**

* Fill Tab 1 fields → Save Draft
* Fill Tab 2 fields → Save Draft
* Reopen draft: Tab 1 + Tab 2 values must still be present (no resets)

---

### SAFEGUARD B — Single Source of Truth for Field Keys

1. Create/maintain **one canonical map** of field keys used by:

   * create form state
   * draft save payload
   * edit form state
   * edit save payload
   * show/actual listing view

2. Keys must match **exactly** across:

   * database columns (or JSON keys)
   * Livewire public properties (or state array)
   * Blade input `wire:model`
   * edit mount/hydrate logic

**Do not** use slightly different names between create vs edit (example: `county` vs `property_county`).

---

### SAFEGUARD C — “Other” Field Logic Must Persist + Re-render Correctly

For any field that has an “Other” option:

1. Store BOTH:

   * selected option value (e.g., `occupancy_requirement_type = 'Other'`)
   * the “Other” text input value (e.g., `occupancy_requirement_other = 'Must occupy within 90 days of closing'`)
2. On edit mount/reopen draft:

   * if selected option is “Other”, automatically set the UI boolean to show the text input
   * repopulate the “Other” text input value

---

### SAFEGUARD D — Photos must persist (Draft + Edit)

1. Personal Photo must be stored in a durable way:

   * recommended: store file path in DB + store file in persistent disk (not temp)
2. Draft reopen/edit must display:

   * existing photo preview
   * ability to remove/replace photo
3. Delete photo should:

   * clear DB path
   * remove file from storage (or mark orphan cleanup job)

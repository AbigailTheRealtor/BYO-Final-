1) Create Persistent “Recent Notices” (Tenant + Agent)
1.1 Data layer (notifications must persist)

Implement a persistent notification system (DB-backed).

If you already have a notifications table/model: use it.
If not: add a new table (addition-only migration), e.g. user_notifications.

Recommended fields (add only if missing):

id

user_id (who receives the notice)

role_context (tenant/agent/admin) or infer by user role

listing_id

bid_id (critical)

type (enum/string):

bid_received (Tenant)

bid_countered (Tenant + Agent)

bid_accepted (Tenant + Agent)

bid_rejected (Tenant + Agent)

message (display text)

link_url OR link_route_name + params (preferred)

is_read boolean default false

timestamps

1.2 When to create notifications (events)

Create notifications at these moments:

Tenant receives:

When any Agent submits a bid on Tenant’s listing → bid_received

When an Agent counters Tenant → bid_countered

When Tenant accepts a bid (create a Tenant notice too if helpful) → bid_accepted

When Tenant rejects a bid → bid_rejected

Agent receives:

When Tenant counters Agent → bid_countered

When Tenant accepts Agent bid → bid_accepted

When Tenant rejects Agent bid → bid_rejected

Admin receives (records)

When a bid is accepted → Admin gets notice with link to Accepted Bid Summary

IMPORTANT: notifications must be created using bid_id so they link to the correct bid.

4) Bid Status logic (Active vs Pending vs Countered etc.)

We decided today:

Timed listings

Active = bid submitted, timer still running, no counter/accept/reject yet

Bid is editable by Agent until timer ends

Traditional (non-timed)

Pending = bid submitted, awaiting Tenant action

Bid may be editable until counter/accept/reject (depending on your existing rules)

Shared statuses (both listing types)

Countered = a counter was issued (never label this as “Inactive”)

Accepted

Rejected

Fix required: If your code currently sets bids to Inactive when countering, replace that with Countered and ensure both sides can see/respond.

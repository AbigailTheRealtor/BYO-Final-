# REPLIT INSTRUCTIONS WITH SAFEGUARDS — ENFORCE REQUIRED FIELDS ON EDIT FOR SELLER / BUYER / LANDLORD / TENANT, WITH SAME STEP-BY-STEP REQUIRED FIELD UX AS CREATE

## GOAL

Fix **Edit Listing** validation for all 4 listing types:

* Seller
* Buyer
* Landlord
* Tenant

Right now, when a user clicks **Edit** on an existing listing, they can delete a required field and still save.

That is wrong.

Any field that is required during **Create Listing** must also be required during **Edit Listing**.

This must be enforced for:

* **Save Draft** on Edit
* **Submit / Save Edit** on Edit

This is **not** about the separate brand-new draft button behavior for a brand-new listing.

---

# REQUIRED UX

The Edit flow should behave the **same way Create Listing already behaves** for required fields.

That means when the user clicks **Save Draft** or **Submit / Save Edit** on an existing listing:

* required fields must be validated
* the system must identify each missing required field
* it should take the user to the required field
* it should move to the next missing required field the same way create does
* required field messaging/behavior should match the existing create flow as closely as possible

Do **not** invent a new edit-only validation UX.
Reuse the existing create required-field UX pattern.

---

# ISSUE

Current Edit Listing problem:

* user opens an existing listing
* deletes a required field
* clicks **Save Draft** or **Submit / Save Edit**
* save still goes through

That should not happen.

Also, the required-field UX on edit should match create, meaning it should:

* name the missing required fields
* take the user to the missing field/tab/section
* behave like the existing create validation flow

---

## E) Edit mode is excluded from the create-style field navigation UX

Example:

* missing fields may exist

Implement fix from audit: separate Create Seller Listing view/edit routing from Hire Seller’s Agent

Use the completed audit as the source of truth.

Problem:
Create Seller Listing at /offer-listing/seller currently redirects to the Hire Seller’s Agent view:

WRONG:
/seller/agent/auction/view/{id}

And the Edit button then routes to:

WRONG:
/hire/agent/auction/edit/{id}/seller

Correct behavior:
Create Seller Listing should route to its own Offer Listing Seller view and edit path:

CORRECT VIEW:
/offer-listing/seller/view/{id}

CORRECT EDIT:
/offer-listing/seller/edit/{id}

STRICT SCOPE:
- Full Service Seller Offer Listing only.
- Do NOT modify Hire Seller’s Agent create/edit/view behavior.
- Do NOT touch Buyer, Tenant, Landlord flows.
- Do NOT touch Limited Service legacy flows.
- Do NOT change database schema.
- Do NOT rename/delete existing meta keys.
- Preserve all existing Seller Offer Listing fields, tabs, tooltips, photos, documents, metadata, draft behavior, and save logic.

Files expected to touch:
- routes/web.php
- app/Http/Livewire/OfferListing/Seller/SellerOfferListing.php
- A new dedicated Seller Offer Listing view controller method OR lightweight controller
- A new dedicated Seller Offer Listing view blade

Step 1 — Add workflow separation meta

In:
app/Http/Livewire/OfferListing/Seller/SellerOfferListing.php

Inside saveAllMetadata(), add:

$auction->saveMeta('workflow_type', 'offer_listing');

Also confirm existing metadata still saves:
- user_type
- service_type
- all Seller Offer Listing fields

Do not remove or rename anything.

Step 2 — Add dedicated Seller Offer Listing view route

In routes/web.php, add a dedicated route:

GET /offer-listing/seller/view/{id}

Name it:

offer.listing.seller.view

Example:

Route::get('/offer-listing/seller/view/{id}', [SellerOfferListingController::class, 'view'])
    ->name('offer.listing.seller.view');

Use existing controller patterns if the project already has an Offer Listing controller namespace. Do not place this in the Hire Seller Agent controller unless absolutely necessary.

Step 3 — Create dedicated Seller Offer Listing view method

Create or extend an Offer Listing Seller controller.

The method should:

- Load the record from SellerAgentAuction model by id
- Load all metas
- Verify this is a Seller Offer Listing, preferably by:
  - workflow_type = offer_listing
  - OR fallback service_type/user_type values if needed for old records
- Render a dedicated Offer Listing Seller blade
- Pass the auction and metadata to the blade

Do NOT render:

hire_seller_agent.view

Step 4 — Create dedicated Seller Offer Listing view blade

Create a new blade such as:

resources/views/offer-listing/seller/view.blade.php

or follow the project’s existing naming convention.

This blade must be dedicated to Create Seller Listing display.

It should show populated Seller Offer Listing fields from these tabs/sections:

- Listing Details
- Property Details
- Sale Terms
- Description
- Tax, Legal & HOA
- Documents & Disclosures
- Photos & Tours
- Broker Compensation & Agency Agreement Terms
- Agent Credentials & Contact Info

Important:
- AI Questions should remain hidden from public listing view unless already intentionally displayed elsewhere.
- Do not show Hire Seller’s Agent-only tabs/questions.
- Do not reuse hire_seller_agent/view.blade.php.

Edit button must point to:

route('offer.listing.seller.edit', ['auctionId' => $auction->id])

or the exact existing named route for:

/offer-listing/seller/edit/{id}

NOT:

route('hire.agent.auction.edit', ...)

Step 5 — Fix submit redirect

In:
app/Http/Livewire/OfferListing/Seller/SellerOfferListing.php

Find the incorrect redirect around the audited lines:

route('seller.agent.auction.detail', ['id' => $auction->id])

Replace it with:

route('offer.listing.seller.view', ['id' => $auction->id])

Also update the browser-event fallback redirect to the same route:

$this->dispatchBrowserEvent('force-redirect', ['url' => $url]);

The resulting URL after Seller Offer Listing submit should be:

/offer-listing/seller/view/{id}

Step 6 — Backward compatibility for old Seller Offer Listing records

Because older Seller Offer Listings may not have workflow_type saved, make the new view route able to load old records if they were created from /offer-listing/seller.

Use safe fallback detection only for the new Offer Listing Seller view.

Do not change Hire Seller view routing.

Potential fallback signals:
- user_type = seller
- service_type = full_service
- presence of Offer Listing-only metadata fields such as tax_legal_hoa fields, listing_documents, AI question fields, or other known Seller Offer Listing-only meta keys

Do not rely on one weak field if it may overlap with Hire Seller’s Agent.

Step 7 — Add safety guard

If someone tries to open a Hire Seller’s Agent record through:

/offer-listing/seller/view/{id}

and it is clearly not an Offer Listing record, return 404 or redirect safely.

Do not accidentally display Hire Seller’s Agent records inside the Offer Listing Seller view.

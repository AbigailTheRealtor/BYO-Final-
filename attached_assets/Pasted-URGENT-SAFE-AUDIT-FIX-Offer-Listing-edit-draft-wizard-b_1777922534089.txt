URGENT SAFE AUDIT + FIX — Offer Listing edit draft wizard broken across all roles

Problem:
Saved draft edit wizard may be broken for all Offer Listing roles, not just Seller.

Affected edit routes:
- /offer-listing/seller/edit/{id}
- /offer-listing/buyer/edit/{id}
- /offer-listing/landlord/edit/{id}
- /offer-listing/tenant/edit/{id}

Issues to audit/fix:
1. Saved fields are not all reloading.
2. Typing into a field jumps the user back to the first tab/page.
3. Next button is not working.
4. Submit button is appearing incorrectly on the first page.
5. Save Draft / Submit buttons should not break Back/Next navigation.

SCOPE:
Offer Listing create/edit draft wizards only:
- Seller
- Buyer
- Landlord
- Tenant

DO NOT TOUCH:
- Agent preset flows
- Counter bid flows
- Hire Agent flows
- Accepted bid summaries
- Public listing views
- Referral logic
- Database schema unless a real missing field requires it
- Do not run migrate:fresh or reset database

STEP 1 — READ-ONLY AUDIT FIRST:
For each role, identify:
- Livewire component used by the edit route
- Main wizard blade file
- Current tab/page variable
- Next/Back method names
- Save Draft / Submit button placement
- Field hydration method
- Any mismatched save/load meta keys

LIKELY CAUSES TO CHECK:
- activeTab/currentStep resets inside render()
- activeTab/currentStep resets during hydration
- Next button missing type="button"
- Buttons inside form accidentally submit
- Submit button conditional uses wrong tab index
- Tab count/index mismatch after adding/removing tabs
- Edit mode does not hydrate all saved meta fields
- Save keys and hydrate keys do not match

FIX REQUIREMENTS:
1. Stable tab/page state
- Typing in any field must not reset the wizard to page 1.
- Only initialize current tab once in mount().
- Do not reset current tab in render().
- Preserve current tab across Livewire updates.

2. Working Next/Back buttons
- Next and Back buttons must be type="button".
- Next should call existing Livewire next-step method.
- Back should call existing Livewire previous-step method.
- If validation blocks navigation, show errors clearly.

3. Correct Submit button behavior
Preferred behavior:
- Back/Next navigation appears on every applicable tab.
- Save Draft remains available.
- Final Submit/Finish button appears only on the final tab unless the existing design intentionally supports submit from every tab.
- Submit button must not appear on page 1 in a way that replaces or disables Next.

4. Saved draft hydration
- All saved fields must reload correctly on edit.
- Compare save meta keys against hydrate/load meta keys.
- Fix mismatched keys with backward-compatible fallbacks.
- Do not delete or rename existing meta keys.
- Preserve existing saved data.

5. Role-specific safety
- Seller changes must not break Buyer/Tenant/Landlord.
- Buyer changes must not break Seller/Tenant/Landlord.
- Landlord changes must not break Seller/Buyer/Tenant.
- Tenant changes must not break Seller/Buyer/Landlord.
- If a shared component is used, apply the smallest safe shared fix only after confirming all affected roles.

VERIFY:
For each role:
1. Open an existing saved draft edit page.
2. Confirm saved values reload on every tab.
3. Go to tab 2 or later.
4. Type into a field.
Expected:
- Stays on same tab.
- Field value remains.
- Does not jump to first page.

5. Click Next on each tab.
Expected:
- Advances correctly.
- Does not submit form accidentally.

6. Click Back.
Expected:
- Goes back correctly.

7. Save Draft from a later tab.
Expected:
- Saves data.
- Does not force tab 1 during typing.
- Reopened draft shows saved values.

8. Confirm Submit/Finish button visibility is correct.
9. Confirm Agent Preset, Counter Bid, Hire Agent, Accepted Summary, and Public Listing views were not changed.

STOP RULE:
If the root issue is in a shared wizard/layout/component used outside Offer Listing flows, stop and report:
- exact file/component,
- all flows that use it,
- safest minimal fix before editing.

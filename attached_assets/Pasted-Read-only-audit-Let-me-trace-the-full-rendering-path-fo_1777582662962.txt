D. Root Causes
Root Cause 1 — view_preference and appliances Select2 are not re-initialized after Livewire updates (PRIMARY)

initializeFullService() runs once at DOMContentLoaded. It initializes Select2 on #view_preference (line 1661) and #appliances (line 1951). But neither is registered in the Livewire.hook('message.processed', …) callback.

Every time the user changes any Livewire-bound field (property type, property sub-type, any text input), Livewire re-renders the component. morphdom patches the DOM. The outer form-group divs for view_preference (no wire:key) and appliances (wire:key only, no outer wire:ignore) can be morphed or re-keyed. When that happens, the Select2 widget is destroyed. It is never re-initialized because message.processed only calls:

attachAuctionDropdownListener
toggleGarageOptions
attachBathroomsDropdownListener
initMlsMultiSelects (for the Residential-only MLS fields)
but not the view_preference or appliances init blocks.

Root Cause 2 — view_preference outer div has no wire:key (amplifies Root Cause 1)

Without wire:key, morphdom cannot reliably match the view_preference form-group across re-renders when surrounding elements change (e.g., the Residential @if block adding or removing elements above it changes the DOM node ordering). This makes Select2 destruction more likely.

Root Cause 3 — Duplicate IDs for Commercial Property block fields (secondary)

Fields water (lines 902, 1177), sewer (lines 920, 1195), and property_utilities (lines 945, 1152) appear twice in the landlord property-preferences blade — once in the Residential block (@if property_type === 'Residential Property') and again in a Commercial block below. Since both @if branches render conditionally, at most one set is ever in the DOM at a time. However if the Blade compiler or Livewire's diff algorithm sees both blocks in the compiled output, and the @if guard fails silently, both could briefly co-exist with duplicate IDs, causing $('#water') to find the wrong element. This is lower risk but worth noting.

Root Cause 4 — Wrong placeholder for non_negotiable_amenities (content bug)

offer-landlord-listing.blade.php line 1495: placeholder: "Select credit score rating(s)" — pasted from a buyer/tenant flow. Should be something like "Select amenities and property features".

Root Cause 5 — The screenshot timing

The screenshot was taken after Task #258 was merged (the code IS in the files) but likely while testing showed that Select2 still reverts to native listboxes on the first Livewire interaction (property type selection). That first interaction triggers message.processed, which re-runs initMlsMultiSelects() — but the view_preference and appliances inits (not in the hook) get destroyed on that same render cycle, making them regress to native listboxes. The MLS fields (heading_fuel etc.) similarly may lose Select2 if initMlsMultiSelects() runs before morphdom finishes patching (the 150ms delay sometimes isn't enough).

## 1. Purpose

This document defines the specification for tracking token usage and model costs generated by the Ask AI feature on the Bid Your Offer platform. It is a **planning and reference document only** — no code, migrations, or schema changes are included here.

The goals of cost tracking are:

- **Operational visibility** — provide the platform team with a clear, real-time picture of how much the Ask AI feature costs to run, broken down by listing, user, and time period.
- **Abuse detection** — identify users or listings generating abnormally high token volume so that rate limits and safeguards can be tuned appropriately.
- **Budget control** — enable the platform to set and monitor cost ceilings before they translate into unexpected bills from the AI provider.
- **Audit support** — maintain a metadata-only record of AI interactions sufficient to verify that responses were generated within approved boundaries, without storing the substantive content of those interactions.

This spec does **not** authorize storing full user prompts, raw AI responses, or any personally identifiable information in the cost tracking layer. See [Section 10 — Privacy Limits](#10-privacy-limits) for the authoritative constraints.

---

## 2. Scope and Phase Gating

Cost tracking is a **Phase 3 or later concern**. OpenAI is not called until Phase 3 is confirmed complete (see `ASK_AI_ROADMAP_AND_GUARDRAILS.md`, Section 2). No cost tracking infrastructure — database tables, logging middleware, or aggregation jobs — may be created before Phase 3 begins.

| Phase | Cost Tracking Status |
|---|---|
| Phase 1 — Context Assembly | Not applicable. No OpenAI calls. |
| Phase 2 — Prompt Engineering | Not applicable. No OpenAI calls. |
| Phase 3 — Backend Integration | Cost tracking **must** be implemented alongside the first OpenAI call. |
| Phase 4 and beyond | Cost tracking continues. Aggregations and the admin dashboard are added as Phase 4 and later tasks are completed. |

The cost tracking layer must go live at the same time as the first live OpenAI call in Phase 3. It must not be retrofitted after the fact.

## 3. Data to Track Per Request

Every successful Ask AI request to the OpenAI API must produce exactly one cost tracking record. Each record captures **metadata only** — no prompt content, no response text. The following fields are required unless noted as optional:

| Field | Type | Description |
|---|---|---|
| `id` | integer (PK, auto-increment) | Unique record identifier. |
| `model` | string | The model identifier as returned by the OpenAI API (e.g., `gpt-4o`, `gpt-4-turbo`). Must be recorded verbatim from the API response — never inferred or assumed. |
| `prompt_tokens` | integer | Number of tokens in the prompt as reported by the API response `usage.prompt_tokens`. |
| `completion_tokens` | integer | Number of tokens in the completion as reported by the API response `usage.completion_tokens`. |
| `total_tokens` | integer | Total tokens for the request as reported by `usage.total_tokens`. Must equal `prompt_tokens + completion_tokens`. If the API value differs, store the API value and flag the discrepancy in the application log. |
| `estimated_cost_usd` | decimal(10, 6) | Estimated cost in US dollars, calculated at request time using the rate table for the model used. See [Section 4](#4-cost-estimation-method). |
| `question_type` | string | The classifier output type assigned to this request (e.g., `property_standout`, `educational`). Sourced from the classifier output. See `ASK_AI_QUESTION_CLASSIFICATION_SPEC.md`. |
| `listing_id` | integer (nullable) | The ID of the listing associated with the request, if any. Null for non-listing-scoped questions. |
| `listing_type` | string (nullable) | The listing role type (`seller`, `buyer`, `landlord`, `tenant`), if a listing is associated. Null otherwise. |
| `user_id` | integer **(nullable)** | The platform user who submitted the request. Null for unauthenticated (guest) requests from public listing pages. Authenticated users always have a value. See note below. |
| `user_role` | string (nullable) | The user's role at the time of the request (`seller`, `buyer`, `landlord`, `tenant`, `agent`, `admin`). Null for unauthenticated requests; use `guest` as the stored value for guest requests if a role label is required by the implementation. |
| `ip_address` | string (nullable) **(optional)** | The client IP address, stored for rate limiting, abuse detection, and security review only. See [Section 10 — Privacy Limits](#10-privacy-limits) for permitted uses and hard prohibitions. |
| `session_id` | string (nullable) **(optional — future field)** | An opaque, non-identifying session identifier for deduplication and request grouping. Not required in the initial Phase 3 implementation; the existing fields (`user_id`, `listing_id`, `created_at`, `question_type`) are sufficient for cost analysis. If added later, must not be the user's HTTP session ID or any value derivable from PII. |
| `api_request_id` | string | The request ID returned by the OpenAI API in the response headers (`x-request-id`). Used for provider-side reconciliation only. |
| `response_status` | string | The outcome of the API call: `success`, `rate_limited`, `error`, or `refused`. Refused applies when the platform classifier returned a refusal before calling the API. |
| `latency_ms` | integer | Round-trip time from platform request to platform receipt of complete API response, in milliseconds. |
| `created_at` | timestamp | UTC timestamp of the request. |

**Authenticated vs. guest requests:**

| Scenario | `user_id` | `user_role` | Notes |
|---|---|---|---|
| Authenticated platform user | Platform user ID (integer) | User's assigned role | Normal case for dashboard and listing-owner flows. |
| Unauthenticated visitor on a public listing page | `null` | `null` (or `guest`) | Ask AI is accessible from public listing pages. Cost is tracked without a user identity. Rate limiting for guests is enforced via `ip_address`. |

Fields `prompt_tokens`, `completion_tokens`, `total_tokens`, and `estimated_cost_usd` must be set to `0` for records where `response_status` is `refused` (since no API call was made). All other required fields remain required regardless of authentication state.

## 4. Cost Estimation Method

Estimated cost is computed at request time using a **platform-managed rate table**. The rate table maps model identifiers to per-token rates for prompt tokens and completion tokens separately.

### Rate Table Format

```json
{
  "gpt-4o": {
    "prompt_cost_per_1k_tokens": 0.005,
    "completion_cost_per_1k_tokens": 0.015
  },
  "gpt-4-turbo": {
    "prompt_cost_per_1k_tokens": 0.010,
    "completion_cost_per_1k_tokens": 0.030
  },
  "gpt-3.5-turbo": {
    "prompt_cost_per_1k_tokens": 0.0005,
    "completion_cost_per_1k_tokens": 0.0015
  }
}
```

### Calculation

```
estimated_cost_usd =
  (prompt_tokens / 1000 * prompt_cost_per_1k_tokens)
  + (completion_tokens / 1000 * completion_cost_per_1k_tokens)
```

### Rate Table Management Rules

- The rate table must be stored in application configuration, not hardcoded in business logic.
- The rate table must be updated whenever OpenAI publishes a price change for a model the platform uses.
- If a model identifier returned by the API is not present in the rate table, the estimated cost must be stored as `null` and an alert must be logged. The cost tracking record must still be written — a missing rate must never block request completion.
- The rate table represents an **estimate**, not an invoice. Actual billing is governed by the OpenAI account statement. The estimated cost field exists for operational budgeting and anomaly detection, not for billing users.

## 5. Storage Schema (Future)

This section describes the intended database table structure for the cost tracking log. No migration is authorized by this document — this is a planning reference for the Phase 3 implementation task.

### Planned Table: `ask_ai_usage_logs`

```sql
CREATE TABLE ask_ai_usage_logs (
    id                   BIGSERIAL PRIMARY KEY,
    model                VARCHAR(100)      NOT NULL,
    prompt_tokens        INTEGER           NOT NULL DEFAULT 0,
    completion_tokens    INTEGER           NOT NULL DEFAULT 0,
    total_tokens         INTEGER           NOT NULL DEFAULT 0,
    estimated_cost_usd   DECIMAL(10, 6)    NULL,
    question_type        VARCHAR(100)      NOT NULL,
    listing_id           BIGINT            NULL,
    listing_type         VARCHAR(50)       NULL,
    user_id              BIGINT            NULL,         -- null for unauthenticated / guest requests
    user_role            VARCHAR(50)       NULL,         -- null for unauthenticated; 'guest' if role label is needed
    ip_address           VARCHAR(45)       NULL,         -- optional; stored for rate limiting and abuse detection only
    session_id           VARCHAR(255)      NULL,         -- optional future field; omit in initial Phase 3 implementation
    api_request_id       VARCHAR(255)      NULL,
    response_status      VARCHAR(50)       NOT NULL,
    latency_ms           INTEGER           NULL,
    created_at           TIMESTAMP         NOT NULL DEFAULT NOW()
);
```

`VARCHAR(45)` for `ip_address` accommodates both IPv4 (max 15 chars) and IPv6 (max 45 chars) addresses.

### Planned Indexes

| Index | Columns | Purpose |
|---|---|---|
| `idx_ask_ai_usage_user` | `user_id, created_at` | Aggregations by user over time. Partial index (WHERE user_id IS NOT NULL) recommended. |
| `idx_ask_ai_usage_listing` | `listing_id, created_at` | Aggregations by listing over time. |
| `idx_ask_ai_usage_date` | `created_at` | Aggregations by day/week/month. |
| `idx_ask_ai_usage_model` | `model` | Cost breakdown by model. |
| `idx_ask_ai_usage_status` | `response_status` | Filtering errors and refusals. |
| `idx_ask_ai_usage_ip` | `ip_address, created_at` | Rate limiting and abuse detection for guest requests. Omit if `ip_address` column is not implemented in Phase 3. |

The `listing_id` foreign key constraint should be deferred or omitted at the table level because usage logs reference listings across four separate listing tables (`seller_agent_auctions`, `buyer_agent_auctions`, `landlord_agent_auctions`, `tenant_agent_auctions`). The `listing_type` column provides the table discriminator. Application-level validation of `listing_id` + `listing_type` consistency is required.

## 6. Aggregations — By Listing

Cost aggregations at the listing level provide the platform team with visibility into which listings are driving the most AI usage and cost. These aggregations are intended for the admin dashboard (see [Section 9](#9-admin-dashboard-plan-future)) and for the listing-owner view in future phases.

### Planned Aggregation Fields

| Metric | Description |
|---|---|
| `total_requests` | Total number of Ask AI requests associated with this listing, across all users. |
| `total_prompt_tokens` | Sum of `prompt_tokens` for all requests on this listing. |
| `total_completion_tokens` | Sum of `completion_tokens` for all requests on this listing. |
| `total_tokens` | Sum of `total_tokens` for all requests on this listing. |
| `estimated_total_cost_usd` | Sum of `estimated_cost_usd` for all requests on this listing. |
| `unique_users` | Count of distinct `user_id` values who have submitted Ask AI requests for this listing. |
| `requests_by_question_type` | Breakdown of request count by `question_type`. |
| `last_request_at` | Timestamp of the most recent Ask AI request on this listing. |

### Grouping Dimensions

Listing-level aggregations can be filtered or segmented by:
- `listing_type` (seller / buyer / landlord / tenant)
- Date range (`created_at` between)
- `response_status` (to exclude or isolate errors and refusals)

## 7. Aggregations — By User

User-level aggregations support abuse detection and per-user usage reporting. These aggregations must be treated as internal operational data — they are not surfaced to the user being measured without explicit product design review.

### Planned Aggregation Fields

| Metric | Description |
|---|---|
| `total_requests` | Total number of Ask AI requests submitted by this user. |
| `total_tokens` | Sum of `total_tokens` across all requests by this user. |
| `estimated_total_cost_usd` | Sum of `estimated_cost_usd` across all requests by this user. |
| `requests_by_role` | Count of requests segmented by `user_role` (in case a user changes role over time). |
| `requests_by_question_type` | Count of requests segmented by `question_type`. |
| `refusal_count` | Count of requests where `response_status = 'refused'`. |
| `error_count` | Count of requests where `response_status = 'error'`. |
| `last_request_at` | Timestamp of the user's most recent Ask AI request. |

### Abuse Detection Thresholds

Implementation of alert thresholds is deferred to the Phase 3 task. This spec establishes that thresholds should be configurable per environment and must not be hardcoded. Suggested starting points (subject to revision during Phase 3):

| Window | Soft Threshold (log warning) | Hard Threshold (block + alert) |
|---|---|---|
| Per hour, per user | 50 requests | 200 requests |
| Per day, per user | 200 requests | 500 requests |
| Per listing, per day | 500 requests | 1,000 requests |

Exceeding a hard threshold must result in a temporary block for that user or listing, plus an alert to the platform admin. The block must not be silent — the user must receive a rate-limit message via the standard refusal template.

## 8. Aggregations — By Day

Daily aggregations provide the platform team with a time-series view of overall Ask AI activity and cost. These are the primary inputs for budget monitoring and trend analysis.

### Planned Aggregation Fields

| Metric | Description |
|---|---|
| `date` | Calendar date (UTC) for the aggregation window. |
| `total_requests` | Total Ask AI requests received platform-wide on this date. |
| `successful_requests` | Count where `response_status = 'success'`. |
| `refused_requests` | Count where `response_status = 'refused'`. |
| `error_requests` | Count where `response_status = 'error'`. |
| `total_prompt_tokens` | Sum of `prompt_tokens` for all requests on this date. |
| `total_completion_tokens` | Sum of `completion_tokens` for all requests on this date. |
| `total_tokens` | Sum of `total_tokens` for all requests on this date. |
| `estimated_total_cost_usd` | Sum of `estimated_cost_usd` for all requests on this date. |
| `unique_users` | Count of distinct users who submitted at least one request on this date. |
| `unique_listings` | Count of distinct listings referenced in requests on this date. |
| `avg_latency_ms` | Average `latency_ms` across all successful requests on this date. |
| `cost_by_model` | JSON breakdown of estimated cost by model identifier. |

### Aggregation Approach

Daily aggregations may be computed either:

1. **On-the-fly** from the raw `ask_ai_usage_logs` table using indexed queries — acceptable when the table is small (early platform, low volume).
2. **Via a nightly materialized summary table** (`ask_ai_daily_summaries`) — preferred once daily request volume exceeds a threshold that makes real-time queries slow (implementation team to define based on observed query performance in Phase 3).

The choice of approach is deferred to the Phase 3 implementation task. This spec requires that the approach be documented when implemented, and that any summary table be rebuilt from raw logs (not from prior summaries) to preserve recalculation accuracy.

## 9. Admin Dashboard Plan (Future)

An admin-facing cost dashboard is planned for a future phase after Phase 4. This section defines the intended scope so that the data model in Phase 3 is designed to support it without backfilling.

### Planned Dashboard Views

| View | Description |
|---|---|
| **Daily Cost Overview** | A time-series chart of estimated cost and total token volume by day, with a rolling 30-day default and configurable date range. |
| **Cost by Model** | A breakdown of token volume and cost segmented by model identifier, showing which models are consuming the most budget. |
| **Top Listings by Usage** | A ranked table of listings by total token consumption, filterable by listing type and date range. |
| **Top Users by Usage** | A ranked table of users by total request count and token consumption. Intended for abuse detection and rate limit tuning. |
| **Request Type Breakdown** | A distribution of requests by `question_type`, showing which question categories are most frequently used. |
| **Error and Refusal Rates** | A time-series view of refusal and error rates, used to monitor classifier health and API reliability. |
| **Budget Alert Status** | A summary panel showing current daily spend against configured thresholds, with alert history. |

### Access Control

The cost dashboard is an admin-only feature. No cost data — individual records or aggregates — is surfaced to non-admin users without an explicit product decision and separate implementation review.

### Integration with Existing Admin Views

The cost dashboard must be consistent with the platform's existing admin UI conventions. It must not introduce new third-party charting libraries or admin frameworks without review, to avoid adding unreviewed dependencies to the production build.

## 10. Privacy Limits

This section is authoritative. It governs all cost tracking implementations across all phases. No implementation task may override or weaken these constraints.

### What Must NOT Be Stored

| Prohibited Data | Reason |
|---|---|
| Full prompt text sent to the API | Contains the user's natural-language question, which may include PII, sensitive transaction details, or proprietary information. |
| Full response text returned by the API | Contains AI-generated content that is subject to change, hallucination risk, and potential legal exposure if stored and later surfaced out of context. |
| Any paraphrase, summary, or excerpt of the prompt | Even partial prompt storage risks reconstructing the user's question and reintroduces the PII exposure risk. |
| Any paraphrase, summary, or excerpt of the response | Same rationale as full response storage. |
| User name, email, or any directly identifying attribute | The `user_id` foreign key is the only user identifier permitted in the cost log. |
| Device fingerprint | Browser fingerprinting data, canvas fingerprints, hardware identifiers, or any value constructed by combining multiple device/browser attributes to uniquely identify a client. |
| Geolocation enrichment | Derived city, region, or coordinates inferred from the IP address. The raw IP is permitted (see below); enriched location data is not. |
| Behavioral profiling data | Click paths, scroll depth, dwell time, or any behavioral signal beyond the metadata fields defined in Section 3. |
| Geographic location beyond what is already on the listing record | The listing's location data is captured via `listing_id`; no additional location data is recorded in the usage log. |

### IP Address — Permitted for Operational Use Only

`ip_address` **may** be stored in the cost tracking record for the following operational purposes:

- **Rate limiting** — enforcing per-IP request caps for unauthenticated (guest) users who have no `user_id` to rate-limit against.
- **Abuse detection** — identifying IP addresses generating abnormally high request volumes, in conjunction with the thresholds defined in Section 7.
- **Security review** — supporting platform security investigations when an IP address is implicated in a policy violation or terms-of-service breach.

`ip_address` **must not** be used for:

- Geolocation enrichment (deriving city, region, country, or coordinates from the IP address).
- Behavioral profiling or building any persistent profile of a user's activity across sessions.
- Any purpose not listed in the permitted uses above.

The `ip_address` field is optional in the initial Phase 3 implementation. If omitted, rate limiting for guest users must be enforced at the application or infrastructure level (e.g., via a reverse proxy or middleware) without storing the IP in the database. If stored, it is subject to the retention policy in Section 11 and must be purged on the same schedule as the rest of the cost tracking record.

### What IS Permitted

The cost tracking record is **metadata only**. Permitted fields are defined in [Section 3](#3-data-to-track-per-request). Every permitted field is either:
- A numeric metric (token counts, cost, latency)
- An enumerated categorical value (model name, question type, user role, response status)
- An opaque identifier (`user_id`, `listing_id`, `api_request_id`)
- An operational signal (`ip_address`, subject to the constraints above)

No field in the cost tracking record encodes the substance of what the user asked or what the AI responded.

### Session ID — Optional Future Field

`session_id` is not required in the initial Phase 3 implementation. The core fields (`user_id`, `listing_id`, `created_at`, `ip_address`, `question_type`, `model`) are sufficient for cost analysis, abuse detection, and budget monitoring at launch.

If `session_id` is added in a later implementation, it must be an opaque, randomly generated token. It must not be the user's HTTP session ID, a value derived from PII, or a value that can cross-reference user identity outside the platform.

### Consent and Disclosure

The platform's privacy policy must be updated before Phase 3 goes live to disclose that Ask AI usage metadata (token counts, question type, model used) is collected for operational purposes. This is a product and legal team obligation, not a development task.

## 11. Retention Policy

Cost tracking records are operational data, not user content. Retention rules balance budget auditability against storage efficiency.

| Data Layer | Retention Period | Rationale |
|---|---|---|
| Raw `ask_ai_usage_logs` records | 12 months from `created_at` | Sufficient for a full annual budget cycle; supports OpenAI invoice reconciliation for up to one year. |
| Daily summary aggregations (`ask_ai_daily_summaries`, if implemented) | 36 months | Longer retention for trend analysis without the storage cost of raw logs. |
| Per-user abuse alert history | 12 months | Sufficient to identify repeat abuse patterns within a billing year. |

Records may be deleted earlier than the retention period only if required by a valid legal hold, user deletion request, or platform data governance policy change. No cost tracking record may be deleted solely because it shows high cost or potential abuse — deletion of records for that reason would undermine the audit trail.

Automated purge jobs must be implemented alongside the initial schema migration in Phase 3 to ensure retention policy is enforced from the first day of operation, not retroactively.
